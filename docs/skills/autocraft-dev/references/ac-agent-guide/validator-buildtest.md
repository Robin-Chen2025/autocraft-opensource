# BUILD-TEST 验证规范

**适用**: 验证子代理验证BUILD-TEST任务时使用
**核心原则**: 验证标准是设计文档和测试方案，不是执行子代理的产出物描述；审阅代码质量，不运行测试

---

## 验证流程

```
1. 读取任务信息（input_data中的所有字段）
2. 读取设计文档和测试方案（input_files指定的文件）—— 这是验证标准
3. 前置检查：验证条件是否充分
4. 读取测试代码文件
5. 审阅测试代码质量（不运行测试！）
6. 按下述6维度逐项检查
7. 写验证结果JSON
```

---

## 前置检查（验证条件校验）

在开始验证之前，**必须先检查验证条件是否充分**：

| 检查项 | 判定标准 |
|--------|----------|
| 设计文档是否可读取 | 文件不存在或无法读取 → FAIL |
| 测试方案是否可读取 | 文件不存在 → 记录warning |
| acceptance_criteria是否具体可验证 | 模糊描述 → FAIL |
| input_files是否指定了设计文档 | 缺少 → FAIL |
| **acceptance_criteria是否属于该组件** | 对照组件源码和设计文档，验收标准不属于该组件职责 → 标记design_issue，该AC不计入完整性判定 |

**原则**：验证条件不充分时，宁可FAIL也不放过。严谨利大于弊。

---

## 验证标准来源

⚠️ **验证标准 = 设计文档 + 测试方案 + 组件规范**

**不得以执行子代理的产出物描述或执行日志作为验证标准。**

---

## 验证维度（6维度）

### 1. 完整性

| 检查项 | 方法 | 判定 |
|--------|------|------|
| 每个acceptance_criteria至少1个测试用例 | 对照acceptance_criteria逐项找对应it() | 缺一项 → FAIL |
| 测试场景覆盖设计文档要求 | 对照测试方案/组件规范检查关键场景 | 缺关键场景 → FAIL |

### 2. 正确性（对照设计文档）

| 检查项 | 方法 | 判定 |
|--------|------|------|
| 断言验证内容与测试层级匹配 | L1: vm属性/composable返回值合理; L2: 必须DOM断言 | L2用vm断言 → FAIL |
| 断言符合设计文档 | 抽查3-5个断言对照API设计/组件规范 | 与设计不符 → FAIL |
| Mock数据结构符合API设计文档 | 对照API文档的响应结构 | 字段/类型不一致 → FAIL |
| Props/Events符合组件规范 | 对照组件规范的接口定义 | 不一致 → FAIL |

**L1 vs L2 断言标准**：

| 测试层级 | 断言方式 | 合理性 |
|---------|---------|--------|
| **L1 单元测试** | vm属性、composable返回值 | ✅ 合理——L1隔离测逻辑，不测渲染 |
| **L2 集成测试** | DOM文本/样式/交互 | ✅ 正确——L2验证组件真实渲染 |
| **L2 用vm断言** | wrapper.vm.xxx | ❌ 降级——L2的价值就是验证渲染 |

**L2断言降级判定**：

| 设计文档要求 | ✅ L2正确断言 | ❌ L2降级断言 |
|-------------|-----------|-----------|
| 显示统计数据 | `expect(text).toContain('1')` | `expect(vm.subjectCount).toBe(1)` |
| 显示成功图标 | `expect(find('.success-icon').exists())` | `expect(vm.status).toBe('success')` |
| 点击触发事件 | `await trigger('click'); expect(emitted)` | `expect(vm.showRetry).toBe(true)` |
| 列表显示数据 | `expect(findAll('.item').length).toBe(3)` | `expect(vm.dataList.length).toBe(3)` |

### 3. 可运行性（代码审阅，不实际运行）

| 检查项 | 方法 | 判定 |
|--------|------|------|
| import路径正确 | 检查import语句与项目结构 | 路径错误 → FAIL |
| 依赖完整 | 检查是否有未定义的引用 | 缺依赖 → FAIL |
| 语法正确 | 检查是否有明显语法错误 | 语法错误 → FAIL |
| 测试结构完整 | 每个it()有断言 | 无断言 → FAIL |
| **L2前端测试导入真实组件** | 检查是否 `import ... from '@/views/'` 或 `import ... from '@/components/'` | 使用内联假组件 → FAIL |

### 4. 一致性

| 检查项 | 方法 | 判定 |
|--------|------|------|
| 产出物路径与任务单一致 | 任务单要求 vs 实际文件位置 | 不一致 → FAIL |
| 用例数与声称一致 | execution_log说N个 vs 实际it()数 | 不一致 → FAIL |
| 验收标准覆盖声称与实际一致 | 声称覆盖AC1-AC3 vs 实际测试 | 不一致 → FAIL |

### 5. 无降级

| 检查项 | 方法 | 判定 |
|--------|------|------|
| 无skip/todo/注释测试 | 搜索`it.skip`、`it.todo`、`xit`、`// it(` | 发现 → FAIL |
| L2无断言弱化 | L2测试检查是否用DOM断言（L1用vm属性合理） | L2用vm断言 → FAIL |
| 无删减测试用例 | 对照设计文档要求的场景 | 缺少场景 → FAIL |

### 6. 代码质量

| 检查项 | 方法 | 判定 |
|--------|------|------|
| describe/it命名清晰 | 描述能看出测什么 | 模糊命名 → warning |
| 测试独立性 | 每个it()不依赖其他it()的副作用 | 强依赖 → warning |
| Mock隔离 | 测试间mock状态不互相影响 | 共享可变状态 → warning |

---

## 判定规则

- 任一维度 FAIL → verification_success: false
- 前置检查 FAIL → 直接 verification_success: false
- 全部 PASS → verification_success: true

```json
{
  "dimension_results": {
    "完整性": "PASS",
    "正确性": "PASS",
    "可运行性": "PASS",
    "一致性": "PASS",
    "无降级": "PASS",
    "代码质量": "PASS"
  }
}
```

---

## 归因原则

当发现测试覆盖不足或失败时：

1. **先质疑验收标准是否合理** — 对照组件实际职责和设计文档，验收标准不属于该组件 → design_issue，该AC不计入完整性
2. **再假设是程序代码问题**（code_issue）——测试正确地暴露了bug
3. 只有确认程序代码**完全符合设计文档**后，才归因为test_issue
4. **不得默认归因为测试环境/工具限制**
