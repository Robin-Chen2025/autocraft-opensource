# PRD 审核 Checklist

**版本：** v2.9  
**日期：** 2026-04-10  
**用途：** AI审核节点检查指导  
**整理人：** AutoCraft 团队

---

## 一、核心原则（审核底线）

### 🚫 三大禁止行为

| 编号 | 禁止行为 | 验证方法 | 违规处理 |
|------|----------|----------|----------|
| **B01** | 禁止凭空添加功能 | 每个功能点必须标注需求来源 | ❌ 严重问题，必须修复 |
| **B02** | 禁止在PRD中添加技术实现细节 | 技术实现应在技术方案文档中定义，PRD仅描述业务需求 | ❌ 严重问题，必须修复 |
| **B03** | 禁止根据"常识"发挥 | 禁止将概念自动转化为具体实现（如"遗忘曲线"→艾宾浩斯算法） | ❌ 严重问题，必须修复 |

### ✅ 三大必须行为

| 编号 | 必须行为 | 验证方法 |
|------|----------|----------|
| **M01** | 功能必须有出处 | 每个功能标注需求来源文档章节 |
| **M02** | 技术术语需有来源依据 | PRD中出现的技术术语需在需求来源文档中有依据 |
| **M03** | 子代理独立审核 | 生成后由子代理按Checklist独立审核 |

## 二、审核前准备检查

### 2.1 预处理清单审核（可选）

> **说明**：需求清单为可选，仅在需求来源复杂时使用。推荐主代理交互式生成，直接引用需求来源文档章节。

若使用需求清单，需通过以下检查：

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **R01** | 需求清单文件存在 | 文件路径存在且非空 | 终止审核，提示生成需求清单 |
| **R02** | 需求项列表存在 | 存在表格格式的需求列表（包含ID列） | 终止审核，提示补充需求列表 |
| **R03** | REQ-ID格式正确 | 匹配正则`REQ-\d{3}`，无重复 | 终止审核，提示修正ID格式 |
| **R04** | 优先级已标注 | 每个需求有P0/P1/P2标记 | 终止审核，提示补充优先级 |
| **R05** | P0数量限制 | P0 ≤ 5个 | 终止审核，提示调整优先级 |
| **R06** | 来源可追溯 | 每个需求有来源文档标注 | 终止审核，提示补充来源 |

### 2.2 输入材料完整性

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **P01** | 需求来源文件存在 | 审核输入目录包含需求描述文件 | 终止审核，提示补充 |
| **P02** | 需求来源文件非空 | 需求来源文件大小 > 1KB | 终止审核，提示补充内容 |
| **P03** | 需求来源可解析 | 文件为 Markdown/YAML/JSON 格式 | 终止审核，提示转换格式 |
| **P04** | PRD文件存在 | 待审核PRD文件路径存在 | 终止审核 |
| **P05** | PRD文件非空 | PRD文件大小 > 10KB | 终止审核，可能生成截断 |

### 2.3 准备合格标准

```
✅ 输入材料完整（P01-P05） → 开始审核
⚠️ 使用需求清单时：需求清单审核通过（R01-R06） + 输入材料完整 → 开始审核
❌ 任一项不通过 → 终止审核，提示补充
```

## 三、阻断项检查 [Error] - 必须通过

以下检查项**必须全部通过**，否则无论评分多少都需修复：

### 3.1 需求追溯类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E01** | FR有需求来源 | 每个FR在"需求来源"列有对应来源文档章节 | FR-003无来源标注 |
| **E02** | 功能需求完整 | 所有功能需求内容完整描述（不分P0/P1/P2） | P1功能仅有框架描述 |
| **E03** | 验收标准完整 | 每个功能需求有对应验收标准 | FR-002无AC引用 |

### 3.2 核心架构完整性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E13** | 核心架构体现 | PRD功能需求是否体现需求来源文档中的核心架构设计 | 需求来源定义了核心流程，PRD未体现 |
| **E14** | 关键设计点覆盖 | 对照需求来源文档的关键设计点，逐一确认是否在PRD中体现 | 需求来源有关键机制，PRD无描述 |

> ⚠️ **E13/E14 特殊规则**：核心架构和关键设计点是系统骨架，**缺失即为严重问题**，不适用"部分通过"。E13或E14不通过时，无论其他项得分如何，必须修复后重新审核。

### 3.3 技术术语类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E04** | 技术术语有来源 | PRD中出现的专业技术术语需在需求来源中有依据 | PRD写"使用Redis缓存"，需求来源无此描述 |
| **E05** | 无凭空添加技术约束 | PRD中的技术约束需有业务背景支撑 | PRD写"必须支持千万级并发"，需求来源无此规模描述 |

### 3.4 ID规范性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E06** | 功能需求ID格式 | 匹配正则`FR-\d{3}`，无重复 | FR-1、FR-001出现两次 |
| **E07** | 验收标准ID格式 | 匹配正则`AC-\d{3}`，无重复 | AC-1、无AC编号 |
| **E08** | ID连续性 | 编号连续（001,002,003...），无跳跃 | FR-001后直接FR-005 |

### 3.4 术语一致性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E09** | 术语一致性 | 对照《术语表》或需求来源中的术语定义 | 需求来源用"题目"，PRD用"试题" |
| **E10** | 文档内部一致性 | PRD内同一概念在不同位置（功能需求、验收标准、术语表、数据格式示例等）的描述必须保持一致，不得出现矛盾或冲突。检查维度：(1)数据流一致：上游声明的输出/能力，下游必须有对应处理；(2)业务规则一致：同一业务规则在不同章节的描述不能矛盾；(3)术语一致：术语表的定义与正文中的使用保持一致；(4)验收标准一致：验收标准与功能描述、业务规则不矛盾 | 功能需求声明"支持.md/.json上传"，验收标准只验证Markdown格式（数据流断裂）；功能需求"文件≤10MB"，验收标准"文件≤5MB"（规则矛盾） |
| **E10** | 用户角色定义 | PRD中定义用户角色，数量≥3或按需求来源定义 | 无用户角色章节，或角色定义缺失 |

### 3.5 文档规范类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E11** | 版本格式正确 | 版本号匹配正则`v\d+\.\d+` | 版本写"V1"而非"v1.0" |
| **E12** | 修改记录完整 | 修改记录包含日期、作者、变更说明三列 | 修改记录只有日期列 |

### 3.6 阻断项合格标准

```
✅ E01-E12 全部通过 → 继续评分判断
❌ 任一项不通过 → 标记严重问题，必须修复
```

## 四、警告项检查 [Warning] - 建议通过

以下检查项不通过时标记为中等/轻微问题：

### 4.1 结构完整性类

| ID | 检查项 | 检测规则 | 问题级别 |
|----|--------|---------|---------|
| **W01** | 文档标题存在 | 第一行包含"产品需求文档"或"PRD" | 中等 |
| **W02** | 版本信息存在 | 存在"版本：vX.X"字段 | 中等 |
| **W03** | 修改记录存在 | 存在"修改记录"或"版本历史"章节 | 中等 |
| **W04** | 必备章节完整 | 存在：执行摘要、功能需求、验收标准、非功能需求、风险依赖 | 中等 |

### 4.2 内容质量类

| ID | 检查项 | 检测规则 | 问题级别 |
|----|--------|---------|---------|
| **W05** | P0数量限制 | P0 ≤ 5个（MoSCoW标准） | 中等 |
| **W06** | P0有验收标准 | 每个P0功能有AC-XXX引用 | 中等 |
| **W07** | AC关联FR | 每个验收标准在"功能需求ID"列有对应FR-XXX | 中等 |
| **W08** | 优先级格式一致 | 全文档使用同一体系（P0/P1/P2或Must/Should/Could） | 轻微 |
| **W09** | 无歧义表达 | 不存在"可能、大概、尽量"等模糊词 | 轻微 |

### 4.3 格式规范类

| ID | 检查项 | 检测规则 | 问题级别 |
|----|--------|---------|---------|
| **W10** | 标题层级正确 | 不跳级（H1→H2→H3） | 轻微 |
| **W11** | 表格有表头 | 所有表格第一行是列标题 | 轻微 |
| **W12** | 代码块标注语言 | 代码块注明语言（如 ```json） | 轻微 |

### 4.4 非功能需求类

| ID | 检查项 | 检测规则 | 问题级别 |
|----|--------|---------|---------|
| **W13** | 性能要求存在 | 存在响应时间/并发量/数据量指标 | 中等 |
| **W14** | 安全要求存在 | 存在认证/授权/加密描述 | 中等 |
| **W15** | 指标可量化 | 性能/可用性指标包含具体数值（如"< 2秒"） | 中等 |

### 4.5 验收标准格式类

| ID | 检查项 | 检测规则 | 问题级别 |
|----|--------|---------|---------|
| **W16** | BDD格式 | 包含"前置条件/操作步骤/预期结果"结构 | 中等 |
| **W17** | 步骤可执行 | 包含编号步骤（1. 2. 3.） | 轻微 |
| **W18** | 结果可验证 | 包含预期结果描述 | 轻微 |

### 4.6 业务目标质量类

| ID | 检查项 | 检测规则 | 问题级别 |
|----|--------|---------|---------|
| **W19** | 业务目标可衡量 | 业务目标包含具体指标（数字或百分比，如"提升效率30%"而非"提升效率"） | 中等 |
| **W20** | 验收标准可验证 | 预期结果可测试验证，不使用主观描述（如"用户满意"应改为"用户评分≥4分"） | 中等 |

## 五、审核结果判断

### 5.1 阻断项优先判断

```
第一步：检查阻断项 E01-E09
    ├─ 有不通过项 → 无论评分多少，必须修复
    └─ 全部通过 → 进入第二步评分判断
```

### 5.2 评分判断规则

**评分计算规则**：

| 问题级别 | 扣分 |
|---------|------|
| 阻断项不通过 | 直接不通过，不计分 |
| 警告项中等 | -5 分 |
| 警告项轻微 | -2 分 |

> **说明**：评分从 100 分起扣，扣完为止。审核报告应列出扣分明细，让评分有据可查。

**判断规则**：

| 评分 | 警告项状态 | 处理 |
|------|-----------|------|
| ≥90分 | 全部通过 | ✅ 定稿 |
| ≥90分 | 有中等项 | ⚠️ 可定稿，建议修复 |
| 80-89分 | 全部通过 | ⚠️ 小修后定稿 |
| 80-89分 | 有中等项 | ⚠️ 修复后定稿 |
| 70-79分 | - | ⚠️ 修复后重新审核 |
| <70分 | - | ❌ 重新生成 |

### 5.3 问题分级与修复优先级

| 问题级别 | 来源 | 修复优先级 |
|---------|------|-----------|
| **严重** | 阻断项不通过（E01-E14） | 必须修复 |
| **中等** | 警告项W01-W07/W13-W16/W19-W20不通过 | 建议修复 |
| **轻微** | 警告项W08-W12/W17-W18不通过 | 可选修复 |

## 附录A：功能点追踪矩阵

### A.1 追踪矩阵模板

| 需求来源ID | PRD功能需求ID | 验收标准ID | 系统功能ID | API接口 | 数据库表 | 状态 |
|-----------|---------------|-----------|-----------|---------|---------|------|
| REQ-001 | FR-001 | AC-001 | F-001 | /api/xxx | table_xxx | ✅ |
| REQ-002 | FR-002 | AC-002, AC-003 | F-002 | - | - | ⚠️ 待设计 |
| REQ-003 | - | - | - | - | - | ❌ 遗漏 |

### A.2 追踪规则

| 规则 | 说明 | 对应检查项 |
|------|------|-----------|
| REQ→FR | 每个需求来源ID至少对应一个FR-XXX | E01, E02 |
| FR→AC | 每个FR-XXX至少对应一个AC-XXX | W06, W07 |
| FR→F | 每个功能需求有对应系统功能ID（设计阶段补充） | E02（下游覆盖） |
| 状态标记 | ✅=完整链路，⚠️=部分缺失，❌=缺失源头 | - |

### A.3 完整追溯链

```
需求来源 → 需求清单 → PRD → 验收标准 → 系统功能

REQ-001 ─→ FR-001 ─→ AC-001 ─→ F-001 ─→ /api/xxx
                    └→ AC-002 ─→ F-002
```

**追溯要求**：
- REQ → FR：每个FR必须有REQ来源（E01）
- FR → AC：每个P0/P1的FR必须有AC（W06, W07）
- AC → FR：每个AC必须关联FR（W07）

### A.4 追踪矩阵检查流程

```
1. 提取需求来源中的所有REQ-XXX
2. 提取PRD中的所有FR-XXX
3. 逐项对照：REQ → FR
4. 标记状态：有对应→✅，无对应→❌
5. 计算覆盖率：已覆盖数 / 总需求数
```

## 附录B：检查项检测方法汇总

### B.1 ID格式检查（可正则匹配）

| 检查项 | 正则表达式 |
|--------|-----------|
| 功能需求ID格式 | `FR-\d{3}` |
| 验收标准ID格式 | `AC-\d{3}` |
| 需求来源ID格式 | `REQ-\d{3}` |
| 文档编号 | `PRD-\d{4}-\d{3}` |
| 版本信息 | `版本[：:]\s*v\d+\.\d+` |
| P0标记 | `P0|Must|必须` |
| 模糊词 | `可能|大概|尽量|或许|大约` |

### B.2 需对照外部文件项

| 检查项 | 对照文件 | 对照方法 |
|--------|---------|---------|
| FR需求来源 | 需求清单（REQ列表） | FR的"需求来源ID"列对照REQ |
| P0/P1需求覆盖 | 需求清单 | REQ列表 vs FR列表 |
| 技术术语来源 | 需求来源文档 | 语义理解，判断术语是否有依据 |

### B.3 需语义理解项

| 检查项 | 检测方法 | 说明 |
|--------|---------|------|
| 技术术语有来源 | 语义分析+需求来源匹配 | PRD中的技术术语需在需求来源有依据 |
| 指标可量化 | 语义分析 | 检查是否包含具体数值 |
| 步骤可执行 | 语义分析 | 检查是否包含编号步骤 |
