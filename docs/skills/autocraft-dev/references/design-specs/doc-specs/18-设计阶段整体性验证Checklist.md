# 设计阶段整体性验证 Checklist

**版本：** v1.0  
**日期：** 2026-04-13  
**用途：** 设计阶段全部文档定稿后执行，验证跨文档一致性  
**执行时机：** 8个文档全部定稿后，开发实施前  
**整理人：** AutoCraft 团队

---

## 一、核心原则

### 🎯 验证目标

**不是看文档"有没有"，而是验证"能不能用"**

| 编号 | 验证原则 | 说明 |
|------|----------|------|
| **V01** | 组件能否接收API数据 | Props字段与API响应字段一致 |
| **V02** | 流程能否实际执行 | API接口串联、数据流转完整 |
| **V03** | 功能追溯链完整 | PRD→系统功能→API→数据库 |
| **V04** | 字段定义跨文档一致 | 同一字段在各文档定义一致 |

### ✅ 三大必须行为

| 编号 | 必须行为 | 验证方法 |
|------|----------|----------|
| **M01** | 必须执行全部6个维度验证 | 功能追溯、API映射、UI绑定、流程、数据、字段 |
| **M02** | 必须模拟核心业务流程 | 推演流程步骤，检查API→数据库→UI链路 |
| **M03** | 必须生成验证报告 | 输出42项检查结果、问题汇总、修复建议 |

## 二、验证前准备

### 2.1 输入材料完整性

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **P01** | 8个设计文档全部定稿 | 所有文档状态为"已定稿"或"已批准" | 终止验证，补充未定稿文档 |
| **P02** | 单文档审核报告存在 | 每个文档有对应审核报告 | ⚠️ 续验但标记缺失审核 |
| **P03** | 文档版本一致 | 所有文档引用PRD版本一致 | ⚠️ 续验但标记版本不一致 |

### 2.2 准备合格标准

```
✅ P01 通过 → 开始验证
❌ P01不通过 → 终止验证，先完成单文档定稿
⚠️ P02/P03不通过 → 续验但标记，验证报告中备注
```

## 三、阻断项检查 [Error] - 必须通过

### 3.1 功能追溯链验证（G01-G06）

**验证方法**：对照PRD功能需求ID → 系统功能ID → API接口ID → 数据库表名

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **G01** | FR有对应系统功能 | 每个PRD功能需求ID（FR-001）在系统功能设计中有对应模块ID（M-01） | FR-007无对应M-XX |
| **G02** | 系统功能有对应API | 每个系统功能ID（F-001）在API设计中有对应接口ID（API-001） | M-01 F-006无API |
| **G03** | API有对应数据库表 | 每个API接口操作的数据表在数据库设计中有定义 | API-053无对应表 |
| **G04** | 数据库表有功能追溯 | 每个数据库表有"支持功能ID"标注 | 某表无功能ID |
| **G05** | FR覆盖率100% | 所有FR都能追溯到数据库表 | FR-005追溯到暂停模块，标记⏸️ |
| **G06** | 无虚增表/API | 所有表/API都能追溯到FR | 数据库有audit_logs表但无FR |

**验证步骤**：
```
步骤1: 读取PRD → 提取FR-001~FR-N列表
步骤2: 读取系统功能设计 → 建立FR→M→F映射表
步骤3: 读取API设计 → 建立F→API映射表
步骤4: 读取数据库设计 → 建立API→Table映射表
步骤5: 输出追溯链完整性报告 → 标记断裂点
```

---

### 3.2 API→数据库字段映射验证（G07-G26）

**验证方法**：逐字段对照API响应字段与数据库表字段

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **G07-G26** | API字段与数据库字段一致 | 核心API响应字段在数据库表中有对应字段，类型一致 | API返回execution_retry_count:number，数据库无此字段 |

**优先验证API**（按核心流程排序）：

| 优先级 | API ID | 验证字段数 | 说明 |
|--------|--------|-----------|------|
| P0 | API-024 | 10 | 任务单详情（TaskNodeProgress组件绑定） |
| P0 | API-034 | 8 | 问题单详情（恢复流程依赖） |
| P0 | API-030 | 5 | 核心执行状态 |
| P0 | API-047 | 8 | 配置链查询 |
| P1 | API-018 | 6 | 工作计划详情 |
| P1 | API-003 | 5 | 项目档案详情 |

**验证步骤**：
```
步骤1: 读取API响应字段列表 → 提取字段名、类型
步骤2: 读取数据库表字段定义 → 对照字段名、类型
步骤3: 检查类型一致性 → API:number对应DB:INTEGER/INT
步骤4: 检查必填一致性 → API:required对应DB:NOT NULL
步骤5: 输出字段映射报告 → 标记不一致字段
```

---

### 3.3 UI组件→API数据绑定验证（G27-G30）

**验证方法**：对照核心组件Props与API响应字段

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **G27** | 组件Props覆盖API响应 | 核心业务组件Props字段在API响应中存在 | TaskNodeProgress缺少current_node Props |
| **G28** | Props类型与API一致 | Props类型与API响应字段类型匹配 | Props:string vs API:number |
| **G29** | 组件Events定义 | 组件有数据更新Events（如@update-status） | 组件无Events，无法绑定API数据 |
| **G30** | 无虚增组件 | 组件规范中所有组件在UI设计中有使用场景 | 组件规范有ExportButton但UI设计无 |

**优先验证组件**：

| 组件 | 验证Props数 | 依赖API | 说明 |
|------|------------|---------|------|
| TaskNodeProgress | 5 | API-024/API-030 | 5节点进度核心组件 |
| PlanTree | 4 | API-018/API-22 | 折叠树核心组件 |
| IssueCard | 3 | API-034 | 问题单展示组件 |
| ProfileCard | 3 | API-003 | 项目档案组件 |

### 3.4 业务流程可执行性验证（G31-G32）

**验证方法**：模拟完整业务流程步骤，检查API→数据库→UI链路

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **G31** | 核心流程可执行 | 模拟流程步骤，每步有API+数据库操作 | 核心恢复流程缺少关键API |
| **G32** | 异常流程可执行 | 模拟异常场景，有问题单创建+恢复机制 | 问题单表缺少关键关联字段 |

**必验流程**：

| 流程ID | 流程名称 | 步骤数 | 关键检查点 |
|--------|---------|--------|-----------|
| F-001 | 核心创建流程（拆解任务） | 4 | API→数据库写入完整 |
| F-002 | 核心执行流程（含异常恢复） | 5 | 执行→异常→恢复链路完整 |
| F-003 | 配置查询→任务执行 | 3 | 配置链→执行链路完整 |

**验证步骤**：
```
步骤1: 选择核心业务流程 → 提取流程步骤
步骤2: 模拟每步操作 → 确认API存在、数据库操作可行
步骤3: 检查异常分支 → 确认问题单创建、恢复机制完整
步骤4: 输出流程可执行性报告 → 标记断裂点
```

---

### 3.5 数据模型完整性验证（G33-G36）

**验证方法**：检查ER图中外键链完整性

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **G33** | 核心数据链完整 | profiles→plans→tasks外键链存在 | tasks表缺少plan_id FK |
| **G34** | 外键级联规则正确 | CASCADE用于强关联，SET NULL用于弱关联 | profiles→plans用RESTRICT（删除失败） |
| **G35** | 恢复流程字段完整 | issues表有task_id、plan_id、recovery字段 | issues表缺少recovery字段 |
| **G36** | 配置查询字段完整 | task_type_config表有复合索引支持配置链查询 | 缺少idx_composite索引 |

**验证步骤**：
```
步骤1: 读取ER图或外键列表 → 提取外键关系
步骤2: 检查核心数据链 → profiles→plans→tasks→issues
步骤3: 检查级联规则 → CASCADE/SET NULL是否合理
步骤4: 检查恢复流程依赖字段 → issues.task_id/plan_id/recovery
步骤5: 输出数据链完整性报告 → 标记缺失外键
```

### 3.6 新增字段一致性验证（G37-G41）

**验证方法**：跨文档搜索同一字段，对照定义

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **G37** | 跨文档字段定义一致 | 同一字段在API/数据库/组件规范中定义一致 | execution_retry_count在API为string，DB为INTEGER |
| **G38** | 字段枚举值一致 | plan_type/status等枚举字段在各文档定义一致 | API定义6种plan_type，数据库定义5种 |
| **G39** | 字段长度一致 | VARCHAR字段长度在API maxLength与数据库长度一致 | API maxLength=100，数据库VARCHAR(50) |
| **G40** | 字段默认值一致 | API默认值与数据库DEFAULT一致 | API默认pending，数据库默认active |
| **G41** | JSON字段格式一致 | JSON字段在各文档有格式示例且一致 | recovery字段在API/数据库格式不一致 |

**必验字段**：

| 字段名 | 验证文档 | 说明 |
|--------|---------|------|
| execution_retry_count | API+数据库+组件规范 | 执行返工计数 |
| verification_retry_count | API+数据库+组件规范 | 验证返工计数 |
| plan_type | API+数据库+组件规范 | 计划类型枚举 |
| recovery | API+数据库 | 恢复信息JSON |
| current_node | API+组件规范 | 当前节点编号 |

## 五、验证结果判断

### 5.1 阻断项优先判断

```
第一步：检查阻断项 G01-G41
    ├─ 有不通过项 → 无论评分多少，必须修复后重新验证
    └─ 全部通过 → 进入第二步评分判断
```

### 5.2 评分判断规则

| 评分 | 警告项状态 | 处理 |
|------|-----------|------|
| ≥90分 | 全部通过 | ✅ 整体性验证通过，进入开发实施 |
| ≥90分 | 有中等项 | ⚠️ 可进入开发，建议修复 |
| 80-89分 | 全部通过 | ⚠️ 修复后重新验证 |
| <80分 | - | ❌ 重新验证 |

### 5.3 问题分级与修复优先级

| 问题级别 | 来源 | 修复优先级 |
|---------|------|-----------|
| **严重** | 阻断项不通过（G01-G41） | 必须修复后重新验证 |
| **中等** | 警告项W01-W06不通过 | 建议修复，不影响开发 |
| **轻微** | 文档格式问题 | 可选修复 |

## 七、整体性验证方法论

### 7.1 验证方法总结

| 方法 | 工具/步骤 | 发现问题类型 |
|------|---------|-------------|
| 文档对照阅读 | 逐章节读取、建立映射表 | 结构缺失、追溯断裂 |
| 关键字段搜索 | grep跨文档搜索字段名 | 定义不一致、字段缺失 |
| 流程模拟推演 | 手动推演业务流程步骤 | 流程断裂、接口缺失 |
| ER图分析 | 检查外键链和级联规则 | 外键缺失、级联错误 |
| 组件Props对照 | 对照组件Props与API响应 | Props无法绑定API数据 |

### 7.2 核心原则

**不是看文档"有没有"，而是验证"能不能用"**

| 验证维度 | 核心问题 |
|---------|---------|
| UI→API绑定 | 组件能否接收API返回的数据？ |
| 业务流程 | 流程能否实际执行下去？ |
| 数据流转 | 数据能否从起点流转到终点？ |
