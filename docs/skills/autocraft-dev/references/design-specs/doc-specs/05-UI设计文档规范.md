## 一、文档概述

UI设计文档定义页面结构、组件定义和交互流程，是前端开发的核心依据。

**前置依赖**：
- 系统功能设计文档（必须）：提供功能清单、功能ID（F-001...）
- 业务流程文档（强烈建议）：提供用户操作步骤，复杂交互必须有业务流程支撑
- 技术方案文档（必须）：提供UI框架选择
- **Vue骨架原型（必须）**：提供已确认的页面结构、路由布局（用户已确认）
- 原型/设计稿（可选）：提供视觉设计参考（如有可视化设计稿）

**核心作用**：

| 场景 | 是否需要 |
|------|---------|
| 纯后端项目 | ❌ 不需要 |
| 有前端页面 | ✅ 需要 |
| 复杂交互流程 | ✅ 需要 |

## 二、分阶段设计策略

### 2.1 UI设计分阶段原则

UI设计采用**页面清单整体 + 详情分阶段**策略：

| 原则 | 说明 |
|------|------|
| **页面清单整体** | 必须完整列出所有P0/P1/P2页面（确保路由架构正确） |
| **详情分阶段** | P0详细设计，P1/P2占位符 |

### 2.2 各优先级设计深度

| 优先级 | 页面清单 | 页面详情 |
|--------|---------|---------|
| **P0** | ✅ 必须列出 | ✅ 详细设计（完整布局、交互、组件引用） |
| **P1** | ✅ 必须列出 | ❌ 占位符（仅页面名称 + 功能说明） |
| **P2** | ✅ 必须列出 | ❌ 占位符（仅页面名称 + 功能说明） |

### 2.3 为什么页面清单必须完整？

| 原因 | 说明 |
|------|------|
| **路由架构正确** | 页面清单完整才能确保前端路由架构设计正确 |
| **组件规范对齐** | 页面清单完整才能确保组件规范覆盖所有页面 |
| **追溯矩阵完整** | 系统功能 → UI页面，不能有遗漏 |

### 2.4 一个文档 vs 多个文档

| 方案 | 推荐 | 说明 |
|------|------|------|
| **一个文档** | ✅ 推荐 | 页面清单完整、追溯矩阵完整、维护成本低 |
| **多个文档** | ❌ 不推荐 | 每个迭代单独文档会导致清单不完整 |

### 2.5 设计指导

参见总纲文档《设计阶段文档规范（总纲）》第10.9节"分阶段设计策略汇总"。

## 五、验收 Checklist

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **设计规范完整** | 颜色、字体、间距规范明确（有具体数值） | ☐ |
| **页面结构清晰** | 主要页面有ASCII结构图，标注组件名称 | ☐ |
| **组件列表完整** | 每个页面列出使用的组件（含来源） | ☐ |
| **交互说明明确** | 关键交互有触发条件和效果说明 | ☐ |
| **状态覆盖** | 空状态、加载状态、错误状态都有定义 | ☐ |
| **响应式设计** | 断点有具体宽度值，布局调整规则明确 | ☐ |

## 六、常见问题

### Q1: UI设计文档和原型有什么区别？

**A**: 
- **原型**：可视化展示，用于评审和确认
- **UI设计文档**：文字化规范，用于指导开发

原型是UI设计文档的输入。

### Q2: 是否需要设计图？

**A**: 如果有Figma或原型工具的设计图，可以在文档中插入截图或链接。文档重点在于结构、交互和状态说明。
