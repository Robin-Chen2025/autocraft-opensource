# API设计文档审核 Checklist

**版本：** v1.3  
**日期：** 2026-04-10  
**用途：** AI审核节点检查指导  
**整理人：** AutoCraft 团队

## 一、核心原则（审核底线）

### 🚫 三大禁止行为

| 编号 | 禁止行为 | 验证方法 | 违规处理 |
|------|----------|----------|----------|
| **B01** | 禁止违反RESTful规范 | 资源命名使用名词复数、小写、连字符 | ❌ 严重问题，必须修复 |
| **B02** | 禁止接口无系统功能ID | 每个接口必须关联F-XXX | ❌ 严重问题，必须修复 |
| **B03** | 禁止响应格式不统一 | code字段统一使用字符串类型 | ❌ 严重问题，必须修复 |

### ✅ 三大必须行为

| 编号 | 必须行为 | 验证方法 |
|------|----------|----------|
| **M01** | 必须有请求和响应示例 | 每个接口详情包含示例 |
| **M02** | 必须有参数说明 | 参数表格包含类型、必填、说明 |
| **M03** | 必须有错误响应示例 | 每个接口至少一个错误响应示例 |

## 二、审核前准备检查

### 2.1 输入材料完整性

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **P01** | 系统功能设计文档存在 | 待审核系统功能设计文件路径存在 | 终止审核 |
| **P02** | 技术方案文档存在 | 待审核技术方案文件路径存在 | ⚠️ 续审但标记缺失（需API框架信息） |
| **P03** | API设计文档存在 | 待审核API设计文件路径存在 | 续审但标记缺失 |
| **P04** | 数据库设计文档存在 | 待审核数据库设计文件路径存在 | ⚠️ 续审但标记缺失（可交叉验证字段） |

### 2.2 准备合格标准

```
✅ P01, P03 全部通过 → 开始审核
❌ P01不通过 → 终止审核，提示补充系统功能设计
⚠️ P02不通过 → 续审但标记"无技术方案"（无法确认API框架）
⚠️ P04不通过 → 续审但标记"无数据库设计"（无法交叉验证）
```

> 📋 **说明**：数据库设计是API设计的下游文档，但如有可交叉验证字段定义。

## 三、阻断项检查 [Error] - 必须通过

### 3.1 文档结构类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E01** | 文档信息完整 | 1节表格包含：版本、基础URL、认证方式、PRD版本 | 缺少认证方式字段 |
| **E02** | 通用规范章节存在 | 第2节存在通用规范内容 | 无通用规范章节 |
| **E03** | 接口列表章节存在 | 第3节存在接口列表内容 | 无接口列表章节 |

### 3.2 通用规范类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E04** | 响应格式定义存在 | 2.2节有响应格式定义 | 无响应格式定义 |
| **E05** | 响应code为字符串类型 | code字段使用字符串（如`SUCCESS`而非`200`） | code使用数字类型 |
| **E06** | 成功响应格式完整 | 成功响应包含code、message、data三字段 | 缺少message字段 |
| **E07** | 错误响应格式完整 | 错误响应包含code、message字段 | 缺少code字段 |
| **E07a** | Pydantic校验错误响应格式 | 设计文档明确规定Pydantic/FastAPI自动校验错误（422）的响应格式：要么统一为业务错误格式（需自定义异常处理器），要么明确声明422使用FastAPI默认格式并记录在文档中 | 未提及422响应格式，或Pydantic校验错误与业务错误使用不同的响应结构且未声明 |
| **E07b** | 错误响应路径一致性 | 错误响应中code/message字段的JSON路径必须一致（如统一为顶层`{"code":...,"message":...}`或统一为`{"detail":{"code":...,"message":...}}`），不得部分嵌套部分不嵌套 | 业务错误返回`{"code":"ERROR"}`，Pydantic错误返回`{"detail":[{"msg":...}]}`（路径不一致） |

### 3.3 接口列表类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E08** | 接口列表表格存在 | 3节存在接口列表表格 | 无接口列表表格 |
| **E09** | 接口列表有系统功能ID | 接口列表表格有"系统功能ID"列 | 无系统功能ID列 |
| **E10** | 接口数量≥1 | 接口列表有至少1个接口定义 | 无接口定义 |
| **E11** | HTTP方法正确 | 方法为GET/POST/PUT/PATCH/DELETE之一 | 使用了GETS等非标准方法 |

### 3.4 RESTful规范类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E12** | 路径使用名词复数 | 路径如`/users`、`/questions`（非`/user`、`/getUsers`） | 路径使用动词`/getUsers` |
| **E13** | 路径使用小写 | 路径全小写（可含连字符） | 路径使用大写`/Users` |

### 3.5 接口详情类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E14** | 接口详情存在 | 第4节有至少1个接口详情描述 | 无接口详情 |
| **E15** | 请求参数表格存在 | 接口详情有请求参数表格 | 无请求参数表格 |
| **E16** | 参数表格有三列 | 请求参数表格有"类型"、"必填"、"说明"三列 | 参数表格缺少必填列 |
| **E17** | 请求示例存在 | 接口详情有请求示例（JSON代码块） | 无请求示例 |
| **E18** | 成功响应示例存在 | 接口详情有成功响应示例 | 无成功响应示例 |
| **E19** | 错误响应示例存在 | 接口详情有至少一个错误响应示例 | 无错误响应示例 |

### 3.6 业务码规范类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E20** | 业务码规范章节存在 | 第5节存在业务码规范内容 | 无业务码规范章节 |
| **E21** | 业务码表格完整 | 业务码表格有业务码、HTTP状态码、说明三列 | 缺少HTTP状态码列 |
| **E22** | 业务码使用字符串 | 业务码如`SUCCESS`、`VALIDATION_ERROR`（非数字） | 业务码使用数字 |

### 3.7 认证授权类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E23** | 认证方式说明存在 | 2.3节有认证方式说明 | 无认证方式说明 |

### 3.8 分页规范类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E24** | 分页规范章节存在 | 第6节存在分页规范内容 | 无分页规范章节 |

### 3.9 内容一致性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E25** | 接口字段与系统功能一致 | API字段定义与系统功能设计的业务规则一致，无理解偏差或简化 | 系统功能定义"题目内容最长2000字"，API字段maxLength=500（简化） |
| **E26** | 文档内部一致性 | API设计文档内同一概念在不同位置（接口列表、接口详情、数据模型、错误码等）的描述必须保持一致，不得出现矛盾或冲突。检查维度：(1)字段一致：接口列表与接口详情中的字段定义一致；(2)状态码一致：错误码表与接口详情中的状态码引用一致；(3)数据模型一致：数据模型定义与接口字段的类型/约束一致 | 接口列表标注"content: string"，接口详情写"content: object"（类型矛盾）；错误码表有"4001"，接口详情引用"4010"（错误码矛盾） |
| **E26** | 无虚增接口 | 接口列表中的所有接口都能追溯到系统功能ID，无凭空添加 | 接口列表有`/api/export`，但无对应F-XXX（虚增） |

### 3.10 业务流程可执行性检查

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E27** | 核心流程可执行性 | 模拟文档中标识的核心业务流程（如主要BPM流程），验证接口串联逻辑正确、数据流转完整 | 核心流程步骤返回缺少关键字段，导致后续无法调用 |
| **E28** | 异常流程可执行性 | 模拟文档中标识的异常/恢复流程，验证关联字段完整、恢复逻辑可执行 | 异常记录缺少关键字段，无法恢复执行 |

**E27模拟验证步骤**：
```
步骤1: 识别文档中的核心业务流程（优先级P0的BPM）
步骤2: 提取核心流程涉及的API接口列表
步骤3: 按流程顺序模拟调用，验证每步返回包含下一步所需的关键字段
步骤4: 验证数据流转完整性 → 前一步的输出字段能作为后一步的输入参数
步骤5: 输出可执行性报告 → 标记断裂的接口调用链
```

**E28模拟验证步骤**：
```
步骤1: 识别文档中的异常/恢复流程
步骤2: 检查异常记录包含恢复所需的关键关联字段
步骤3: 验证恢复接口的参数完整性
步骤4: 输出异常流程可执行性报告 → 标记缺失的关联字段
```

### 3.11 阻断项合格标准

```
✅ E01-E28 全部通过 → 继续评分判断
❌ 任一项不通过 → 标记严重问题，必须修复
```

## 五、审核结果判断

### 5.1 阻断项优先判断

```
第一步：检查阻断项 E01-E25
    ├─ 有不通过项 → 无论评分多少，必须修复
    └─ 全部通过 → 进入第二步评分判断
```

### 5.2 评分判断规则

| 评分 | 警告项状态 | 处理 |
|------|-----------|------|
| ≥90分 | 全部通过 | ✅ 定稿 |
| ≥90分 | 有中等项 | ⚠️ 可定稿，建议修复 |
| 80-89分 | 全部通过 | ⚠️ 小修后定稿 |
| 80-89分 | 有中等项 | ⚠️ 修复后定稿 |
| 70-79分 | - | ⚠️ 修复后重新审核 |
| <70分 | - | ❌ 重新生成 |

### 5.3 问题分级与修复优先级

| 问题级别 | 来源 | 修复优先级 |
|---------|------|-----------|
| **严重** | 阻断项不通过（E01-E24） | 必须修复 |
| **中等** | 警告项W03/W05-W07/W11-W15/W17-W18/W20-W24不通过 | 建议修复 |
| **轻微** | 警告项W01/W02/W04/W08-W10/W16/W19/W25-W26不通过 | 可选修复 |

## 八、与其他文档的分工约定

### 8.1 与系统功能设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **系统功能设计** | 业务功能 + 业务规则 | F-001创建题目（题目内容不能为空，最长2000字） |
| **API设计** | 接口路径 + 参数类型 | POST /api/questions, name: string, required, maxLength=100 |

### 8.2 与数据库设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **API设计** | 接口层面的字段定义 | name: string, required, maxLength=100 |
| **数据库设计** | 存储层面的字段定义 | name VARCHAR(100) NOT NULL |

### 8.3 与UI设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **UI设计** | 交互行为 + 组件状态 | 点击"保存"按钮 → 显示加载状态 |
| **API设计** | 接口调用 + 数据格式 | POST /api/questions, 返回创建结果 |

### 8.4 审核时的注意事项

- API设计文档定义**接口规范**
- 不应包含业务规则细节（由系统功能设计定义）
- 不应包含数据库表结构（由数据库设计定义）
- 响应code必须使用字符串类型（如`SUCCESS`而非`200`）
- 每个接口必须关联系统功能ID

## 九、来源追溯（业界标准对照）

### 9.1 业界标准对照

| 我们的检查项 | 业界标准 | 核心要求 | 来源 |
|-------------|---------|---------|------|
| RESTful规范 | **REST架构风格** | 资源命名、HTTP方法语义、状态码使用 | Roy Fielding博士论文 |
| 路径命名规范 | **URI设计规范** | 名词复数、小写、连字符 | RFC 3986、Google API设计指南 |
| 响应格式统一 | **JSON API规范** | 统一的响应结构、错误格式 | JSON API、Google API设计指南 |
| 业务码设计 | **错误码设计最佳实践** | 业务错误码与HTTP状态码分离 | Google API设计指南、Microsoft REST API指南 |
| 分页规范 | **分页设计模式** | page/pageSize参数、pagination响应 | Google API设计指南 |

### 9.2 RESTful核心原则

**定义**（Roy Fielding博士论文）：
> REST（Representational State Transfer）是一种软件架构风格，强调资源的统一接口和无状态通信。

**核心原则**：

| 原则 | 说明 | 对应检查项 |
|------|------|-----------|
| **统一接口** | 资源使用统一的方法和格式 | E11, E12, E13 |
| **无状态** | 每个请求包含所有必要信息 | W03（请求头） |
| **资源标识** | 每个资源有唯一URI | E12, E13 |
| **分层系统** | 客户端无需知道后端架构 | E04（响应格式） |

### 9.3 HTTP方法语义

| 方法 | 语义 | 幂等性 | 安全性 | 对应场景 |
|------|------|--------|--------|----------|
| GET | 获取资源 | 是 | 是 | 查询 |
| POST | 创建资源 | 否 | 否 | 新增 |
| PUT | 全量更新 | 是 | 否 | 修改 |
| PATCH | 部分更新 | 否 | 否 | 部分修改 |
| DELETE | 删除资源 | 是 | 否 | 删除 |

### 9.4 我们的定义与业界标准的差异

| 维度 | 业界标准 | 我们的定义 | 差异说明 |
|------|---------|-----------|---------|
| 业务码格式 | 数字或字符串均可 | **强制字符串**（如`SUCCESS`） | 便于AI生成和前端统一处理 |
| 版本管理 | URL/Header/Query | URL版本（`/api/v1`） | 简单直观 |
| 分页方式 | Offset/Cursor | Page/PageSize（偏移分页） | 适合中小规模数据 |
| HATEOAS | 建议包含链接 | 未强制要求 | 简化实现 |
