## 一、文档概述

API设计文档定义系统的接口规范，是前后端开发的桥梁。

**核心作用**：
- 前后端并行开发依据
- 自动生成API文档
- 测试代码生成依据

**前置依赖**：
- 系统功能设计（必须）：提供功能清单和业务规则
- 技术方案（必须）：提供API框架选择
- 数据库设计（下游）：可与API设计交叉验证

---

## 二、分阶段设计策略

### 2.1 API设计分阶段原则

API设计采用**接口清单整体 + 详情分阶段**策略：

| 原则 | 说明 |
|------|------|
| **接口清单整体** | 必须完整列出所有P0/P1/P2接口（确保模块划分正确） |
| **详情分阶段** | P0详细设计，P1/P2占位符 |

### 2.2 各优先级设计深度

| 优先级 | 接口清单 | 接口详情 |
|--------|---------|---------|
| **P0** | ✅ 必须列出 | ✅ 详细设计（完整参数、响应、错误码） |
| **P1** | ✅ 必须列出 | ❌ 占位符（仅接口名称 + 路径） |
| **P2** | ✅ 必须列出 | ❌ 占位符（仅接口名称 + 路径） |

### 2.3 为什么接口清单必须完整？

| 原因 | 说明 |
|------|------|
| **模块划分正确** | 接口清单完整才能确保技术方案的模块划分正确 |
| **数据库设计对齐** | 接口清单完整才能确保数据库表设计覆盖所有接口 |
| **追溯矩阵完整** | 系统功能 → API接口，不能有遗漏 |

### 2.4 一个文档 vs 多个文档

| 方案 | 推荐 | 说明 |
|------|------|------|
| **一个文档** | ✅ 推荐 | 接口清单完整、追溯矩阵完整、维护成本低 |
| **多个文档** | ❌ 不推荐 | 每个迭代单独文档会导致清单不完整 |

### 2.5 设计指导

参见总纲文档《设计阶段文档规范（总纲）》第10.9节"分阶段设计策略汇总"。

## 三、详细模板

```markdown
# API 设计文档 - [模块名称]

## 1. 文档信息

| 属性 | 值 |
|------|-----|
| 版本 | v1 |
| 基础URL | `/api/v1` |
| 认证方式 | Bearer Token / DEV_MODE |
| 数据格式 | JSON |
| PRD版本 | v1.0 |

## 2. 通用规范

### 2.1 请求头

```
Content-Type: application/json
Authorization: Bearer {token}
X-User-Id: {user_id}  # 开发模式使用
```

### 2.2 响应格式

**成功响应：**
```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {
    // 业务数据
  }
}
```

**错误响应：**
```json
{
  "code": "VALIDATION_ERROR",
  "message": "参数验证失败",
  "details": [
    {"field": "name", "message": "不能为空"}
  ]
}
```

> ⚠️ **规范约定**：`code` 字段统一使用字符串类型，成功返回 `"SUCCESS"`，错误返回错误码字符串（如 `"VALIDATION_ERROR"`）。便于AI生成和前端统一处理。

### 2.3 认证授权

| 模式 | 说明 | 使用场景 |
|------|------|----------|
| DEV_MODE | 使用X-User-Id请求头模拟用户 | 开发环境 |
| JWT | 使用Bearer Token认证 | 生产环境 |

## 3. 接口列表

### 3.1 [资源名称]

| 方法 | 路径 | 描述 | 系统功能ID | 权限 |
|------|------|------|-----------|------|
| GET | `/resources` | 获取列表 | F-001 | user |
| GET | `/resources/{id}` | 获取详情 | F-002 | user |
| POST | `/resources` | 创建 | F-003 | user |
| PUT | `/resources/{id}` | 全量更新 | F-004 | user |
| PATCH | `/resources/{id}` | 部分更新 | F-004 | user |
| DELETE | `/resources/{id}` | 删除 | F-005 | user |

> ⚠️ **追溯约定**：每个API接口必须关联系统功能ID（F-XXX），确保从接口可追溯到业务需求。

## 4. 接口详情

### 4.1 POST /resources

**描述**：创建资源

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| name | string | 是 | 名称，最长100字符 |
| description | string | 否 | 描述，最长500字符 |
| status | integer | 否 | 状态，默认1 |

**请求示例**：
```json
{
  "name": "示例名称",
  "description": "示例描述",
  "status": 1
}
```

**响应参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| id | integer | 资源ID |
| name | string | 名称 |
| description | string | 描述 |
| status | integer | 状态 |
| created_at | string | 创建时间 (ISO 8601) |

**成功响应 (201 Created)**：
```json
{
  "code": "CREATED",
  "message": "创建成功",
  "data": {
    "id": 1,
    "name": "示例名称",
    "description": "示例描述",
    "status": 1,
    "created_at": "2026-04-07T00:00:00Z"
  }
}
```

**错误响应 (400 Bad Request)**：
```json
{
  "code": "VALIDATION_ERROR",
  "message": "参数验证失败",
  "details": [
    {"field": "name", "message": "名称不能为空"}
  ]
}
```

### 4.2 GET /resources

**描述**：获取资源列表（支持分页）

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | integer | 否 | 页码，默认1 |
| pageSize | integer | 否 | 每页数量，默认20，最大100 |
| status | integer | 否 | 按状态筛选 |

**成功响应 (200 OK)**：
```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": [
    {
      "id": 1,
      "name": "示例1",
      "status": 1
    },
    {
      "id": 2,
      "name": "示例2",
      "status": 1
    }
  ],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 100,
    "totalPages": 5
  }
}
```

### 4.3 GET /resources/{id}

**描述**：获取资源详情

**路径参数**：

| 参数名 | 类型 | 说明 |
|--------|------|------|
| id | integer | 资源ID |

**成功响应 (200 OK)**：
```json
{
  "code": "SUCCESS",
  "message": "操作成功",
  "data": {
    "id": 1,
    "name": "示例名称",
    "description": "示例描述",
    "status": 1,
    "created_at": "2026-04-07T00:00:00Z",
    "updated_at": "2026-04-07T00:00:00Z"
  }
}
```

**错误响应 (404 Not Found)**：
```json
{
  "code": "NOT_FOUND",
  "message": "资源不存在"
}
```

## 5. 业务码规范

> ⚠️ **说明**：本章节包含成功码和错误码，统一作为"业务码"管理，便于前端统一处理。

| 业务码 | HTTP状态码 | 说明 | 处理建议 |
|--------|-----------|------|----------|
| SUCCESS | 200 | 操作成功 | 正常处理 |
| CREATED | 201 | 创建成功 | 正常处理 |
| VALIDATION_ERROR | 400 | 参数验证失败 | 检查参数格式 |
| UNAUTHORIZED | 401 | 未授权 | 检查Token |
| FORBIDDEN | 403 | 禁止访问 | 检查权限 |
| NOT_FOUND | 404 | 资源不存在 | 检查资源ID |
| CONFLICT | 409 | 资源冲突 | 检查唯一性约束 |
| INTERNAL_ERROR | 500 | 服务器错误 | 联系技术支持 |

## 6. 分页规范

**请求参数**：
- `page`：页码，从1开始
- `pageSize`：每页数量，默认20，最大100

**响应格式**：
```json
{
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 100,
    "totalPages": 5
  }
}
```

## 四、RESTful 规范

### 4.1 资源命名

| 规则 | 示例 |
|------|------|
| 使用名词复数 | `/users`, `/questions` |
| 不使用动词 | ❌ `/getUsers` ✅ `GET /users` |
| 使用小写 | ❌ `/Users` ✅ `/users` |
| 使用连字符 | ❌ `/userProfiles` ✅ `/user-profiles` |

### 4.2 HTTP方法语义

| 方法 | 语义 | 幂等性 |
|------|------|--------|
| GET | 获取资源 | 是 |
| POST | 创建资源 | 否 |
| PUT | 全量更新 | 是 |
| PATCH | 部分更新 | 否 |
| DELETE | 删除资源 | 是 |

### 4.3 状态码使用

| 状态码 | 场景 |
|--------|------|
| 200 | 成功 |
| 201 | 创建成功 |
| 400 | 参数错误 |
| 401 | 未认证 |
| 403 | 无权限 |
| 404 | 不存在 |
| 409 | 冲突 |
| 500 | 服务器错误 |

## 六、常见问题

### Q1: PUT vs PATCH？

**A**: 
- **PUT**：全量更新，需要传所有字段
- **PATCH**：部分更新，只传需要修改的字段

### Q2: 错误响应用code还是HTTP状态码？

**A**: 两者都用。
- HTTP状态码：协议层面，客户端可直接判断
- 业务错误码：业务层面，提供更详细的错误信息

### Q3: 分页参数放在哪里？

**A**: 放在Query参数中：`GET /resources?page=1&pageSize=20`
