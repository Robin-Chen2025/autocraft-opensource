## 一、文档概述

技术方案文档定义项目的技术选型、系统架构、目录结构和部署方案。

**前置依赖**：
- 系统功能设计文档（必须）：提供功能模块划分，技术选型需支撑功能需求
- PRD（辅助）：提供业务需求和可选的技术栈要求

**核心作用**：
- 统一技术栈选择
- 指导项目结构搭建
- 作为INFRA计划的输入

## 二、文档边界定义

| 文档 | 包含内容 | 不包含内容 |
|------|---------|-----------|
| **技术方案文档** | 技术选型、版本号、选型理由、架构设计、目录结构、部署方案、安全策略 | 业务逻辑细节、API接口定义、数据库表结构 |

**与其他文档分工**：

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **PRD** | 业务需求 + **技术栈要求**（可选） | "系统需支持高并发，建议使用FastAPI" |
| **技术方案** | 技术选型 + 版本 + 架构 + 目录 + 部署 | "后端：Python 3.10+ + FastAPI 0.100+" |
| **API设计** | 接口路径、请求参数、响应格式 | `/api/tasks`, `{title: string}` |
| **数据库设计** | 表结构、字段定义、索引 | `tasks`表，`id`, `title`字段 |

**技术栈约定**：

| 场景 | PRD定义 | 技术方案处理 |
|------|---------|-------------|
| PRD明确技术栈 | "使用Vue 3 + FastAPI" | 必须一致（审核Checklist W03检查） |
| PRD未定义技术栈 | 无技术要求 | 技术方案自由选型，需有完整选型理由 |
| PRD定义技术原则 | "前端需响应式" | 技术方案选择符合原则的技术 |

## 三、分阶段设计策略

### 3.1 技术方案采用一次性设计

技术方案文档**不分阶段**，采用一次性设计策略。

| 原则 | 说明 |
|------|------|
| **架构一次性规划** | 避免重构，前瞻性设计 |
| **技术选型整体视角** | 需要考虑所有功能模块的技术需求 |
| **架构分层一次性** | 模块划分、目录结构、部署方案需整体规划 |

### 3.2 为什么技术方案不分阶段？

| 原因 | 说明 |
|------|------|
| **技术选型影响全局** | 框架、数据库、中间件的选择影响所有功能 |
| **架构分层不可分割** | 分层架构（前端/后端/数据库）是整体设计 |
| **目录结构统一** | 项目骨架一次性搭建，后续迭代只增加文件 |
| **避免重构** | 后续迭代改技术选型代价极高 |

### 3.3 技术方案与分阶段设计的关系

虽然技术方案不分阶段，但它需要**支撑分阶段开发**：

| 设计要素 | 一次性设计 | 分阶段支撑 |
|---------|-----------|-----------|
| 技术选型 | ✅ 一次性 | 所有迭代使用同一技术栈 |
| 系统架构 | ✅ 一次性 | 为P1/P2功能预留扩展点 |
| 目录结构 | ✅ 一次性 | 为后续迭代预留目录空间 |
| 部署方案 | ✅ 一次性 | 支持增量部署 |

### 3.4 设计指导

参见总纲文档《设计阶段文档规范（总纲）》第10.9节"分阶段设计策略汇总"。

## 5. 部署方案

### 5.0 系统边界与端口规划

> ⚠️ **必须项**：当项目与同服务器其他系统共存时，必须明确定义端口分配和系统边界，避免端口冲突和地址混用。

| 系统 | 后端端口 | 前端端口 | 说明 |
|------|---------|---------|------|
| **本项目** | `[后端端口]` | `[前端端口]` | 独立部署，开发/生产统一 |
| `[其他系统A]` | `[端口]` | `[端口]` | `[关系说明]` |

**铁律**：
- 代码中**严禁硬编码**其他系统的端口
- API地址必须通过环境变量注入，默认值应为本项目的后端端口

### 5.1 开发环境

| 组件 | 配置 |
|------|------|
| 后端 | `uvicorn main:app --reload --port [后端端口]` |
| 前端 | `npm run dev` (Vite开发服务器) |
| 数据库 | SQLite本地文件 |
| 认证 | DEV_MODE=true (无需登录) |

#### 5.1.1 前后端通信架构（开发环境）

> ⚠️ **必须项**：必须明确说明开发环境中前端如何连接后端API。

**方式1：Vite Proxy（推荐）**

```typescript
// vite.config.ts
export default defineConfig({
  server: {
    proxy: {
      '/api': {
        target: 'http://localhost:[后端端口]',
        changeOrigin: true,
      },
    },
  },
})
```

- 前端使用相对路径`/api/*`的请求会被Vite代理到后端
- 仅在`npm run dev`时生效，生产环境需Nginx配置

**方式2：Axios baseURL（环境变量）**

```typescript
// config/index.ts
VITE_API_BASE_URL: import.meta.env.VITE_API_BASE_URL || 'http://localhost:[后端端口]'
```

- 启动前端时设置：`VITE_API_BASE_URL=http://localhost:[后端端口] npm run dev`
- 生产环境在`.env.production`中配置

**两种方式的共存关系**（如同时使用）：
- FileUpload等组件使用`upload-url="/api/..."`（相对路径）→ 走Vite proxy
- API模块使用`baseURL + '/api/...'`（绝对路径）→ 直连后端
- 两者最终都到达后端，但路径不同

#### 5.1.2 环境变量规范

| 变量名 | 开发环境默认值 | 生产环境 | 说明 |
|--------|--------------|---------|------|
| `VITE_API_BASE_URL` | `http://localhost:[后端端口]` | Nginx代理地址 | Axios请求的baseURL |
| `VITE_TIMEOUT` | `30000` | `30000` | API请求超时(ms) |

**`.env.development`示例**：
```
VITE_API_BASE_URL=http://localhost:[后端端口]
VITE_TIMEOUT=30000
```

**`.env.production`示例**：
```
VITE_API_BASE_URL=/api
VITE_TIMEOUT=30000
```

### 5.2 生产环境

| 组件 | 配置 |
|------|------|
| 后端 | Gunicorn + Uvicorn Workers |
| 前端 | Nginx静态文件服务 + API代理 |
| 数据库 | SQLite / MySQL |
| 认证 | JWT Token认证 |

#### 5.2.1 Nginx配置要点

> ⚠️ **必须项**：生产环境需配置Nginx同时服务前端静态文件和API代理。

```nginx
server {
    listen [前端端口];
    
    # 前端静态文件
    location / {
        root /path/to/frontend/dist;
        try_files $uri $uri/ /index.html;
    }
    
    # API代理到后端
    location /api/ {
        proxy_pass http://127.0.0.1:[后端端口];
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        client_max_body_size 10M;  # 文件上传限制
    }
}
```

## 6. 安全策略

### 6.1 认证方式
- 开发环境：DEV_MODE=true，使用X-User-Id请求头模拟用户
- 生产环境：JWT Token认证

### 6.2 数据保护
- 密码存储：bcrypt哈希
- 敏感数据：环境变量存储
- SQL注入：使用ORM参数化查询

### 6.3 权限控制
- 基于用户ID的数据隔离
- 所有API验证用户身份

## 六、常见问题

### Q1: 技术方案文档和PRD的关系？

**A**: PRD定义"做什么"，技术方案文档定义"用什么做"。技术方案是PRD的技术实现指导。

### Q2: 目录结构为什么要标准化？

**A**: 标准化目录结构可以：
- 降低团队沟通成本
- 方便自动化工具处理
- 支持子代理自动生成代码
