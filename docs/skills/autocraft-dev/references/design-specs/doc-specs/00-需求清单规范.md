## 一、概述

### 1.1 文档定位

需求清单是 PRD 生成的前置输入材料，用于将非结构化的需求源文件转换为结构化的需求项列表。

| 维度 | 说明 |
|------|------|
| **输入** | 需求源文件（业务方案、技术方案、沟通记录等） |
| **输出** | 结构化需求清单（Markdown 表格格式） |
| **作用** | 确保 PRD 生成时需求完整、可追溯 |

### 1.2 与 PRD 的关系

```
需求源文件 ──→ 需求清单 ──→ PRD
    │              │          │
    │              │          │
    ▼              ▼          ▼
 非结构化      结构化      优先级分级
```

- **需求清单**：完整提取需求，不分级优先级
- **PRD**：根据业务价值分级（P0/P1/P2）

## 二、需求清单格式

### 2.1 文档结构

```markdown
# 需求清单 - [产品名称]

## 文档信息
- 来源文档：[需求源文件1.md, 需求源文件2.md, ...]
- 提取时间：YYYY-MM-DD HH:MM
- 提取模型：[模型名称]

## 需求统计
- 总需求数：X 个

---

## 一、[功能模块名称]

**概要描述**：[该模块的业务目标和核心能力描述，2-3句话]

| ID | 需求描述 | 来源文档 | 来源位置 |
|----|----------|----------|----------|
| REQ-001 | [用户视角需求描述] | 需求源文件.md | 章节/行号 |
| REQ-002 | [用户视角需求描述] | 需求源文件.md | 章节/行号 |

---

## 二、[功能模块名称]

**概要描述**：[该模块的业务目标和核心能力描述]

| ID | 需求描述 | 来源文档 | 来源位置 |
|----|----------|----------|----------|
| REQ-003 | ... | ... | ... |
```

### 2.2 章节组织原则

| 原则 | 说明 |
|------|------|
| **按功能模块分组** | 不按源文档分组，按业务功能分组 |
| **概要描述** | 每个章节开头用2-3句话描述该模块的业务目标和核心能力 |
| **需求编号连续** | 全文档编号连续（REQ-001, REQ-002...），不按章节重新编号 |

### 2.2 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| **ID** | ✅ | 唯一标识，格式 `REQ-\d{3}` |
| **需求描述** | ✅ | 用户视角的需求描述 |
| **来源文档** | ✅ | 需求来源的**原始源文件**名称（不是需求清单自己！） |
| **来源位置** | ✅ | 需求在源文档中的位置（章节号/行号） |

### 2.3 来源标注规则（重要！）

| 规则 | 说明 |
|------|------|
| **来源文档必须是原始源文件** | 不能把"需求清单.md"作为来源文档 |
| **来源位置必须是源文档中的位置** | 格式：章节号（如"4.4.3节"）或行号（如"L15-20"） |

**正确示例**：

| ID | 需求描述 | 来源文档 | 来源位置 |
|----|----------|----------|----------|
| REQ-001 | 主代理启动工作计划 | 完整业务流程方案.md | 4.4.2节 |
| REQ-002 | 执行子代理pytest闭环修复 | 任务单执行规范.md | 4.4.3节 |

**错误示例**（绝对禁止）：

| ID | 需求描述 | 来源文档 | 来源位置 |
|----|----------|----------|----------|
| REQ-001 | 主代理启动工作计划 | **需求清单.md** | ❌ 错误！ |

### 2.4 不要包含的字段

| 字段 | 原因 |
|------|------|
| **优先级** | 优先级在 PRD 阶段确定，需求清单不分级 |
| **状态** | PRD 生成后再跟踪状态 |

## 三、需求清单核心原则

### 3.1 需求清单做什么

| 做什么 | 说明 |
|--------|------|
| **结构化** | 非结构化 → 表格格式 |
| **去重合并** | 相似需求合并 |
| **用户视角** | 技术描述 → 用户场景 |
| **颗粒度调整** | 统一颗粒度 |
| **来源追溯** | 标注来源 |

### 3.2 需求清单不做

| 不做什么 | 原因 |
|---------|------|
| 详细展开 | 这是 PRD 的事 |
| 闭环流程描述 | 这是系统功能设计的事 |
| 优先级分级 | 这是 PRD 阶段的事 |

### 3.3 颗粒度标准

**从用户视角描述一个完整的用户场景**

| ✅ 正确示例 | ❌ 错误示例 | 问题 |
|------------|------------|------|
| 主代理启动工作计划并监控执行进度 | FlowTicketManager 单例管理器 | 太技术化 |
| 执行子代理生成代码并自动验证 | APScheduler 每 30 秒检测任务状态 | 太细，是技术任务 |
| 验证子代理检查代码逻辑正确性 | pytest 执行并生成 JSON 报告 | 太技术化 |

### 3.4 颗粒度检验问题

| 问题 | 是 | 否 |
|------|----|----|
| 用户能理解这个需求吗？ | ✅ 继续 | ❌ 太技术化，重写 |
| 这是一个完整的用户场景吗？ | ✅ 继续 | ❌ 太细，合并 |
| 能定义明确的验收标准吗？ | ✅ 继续 | ❌ 太模糊，细化 |

## 四、用户角色定义

### 4.1 用户角色识别

需求清单必须明确用户角色。用户角色是从**使用系统的人或系统**的角度定义的。

**通用用户角色**：

| 用户角色 | 描述 | 典型场景 |
|---------|------|---------|
| **主代理** | Agent 系统的核心协调者 | 启动工作计划、处理异常、监控进度 |
| **拆解子代理** | 负责分析工作计划并拆解为任务 | 读取工作计划、生成任务单 |
| **执行子代理** | 负责执行具体任务生成代码 | 生成代码、执行测试、修复问题 |
| **验证子代理** | 负责验证任务产出物正确性 | 检查代码逻辑、验证测试覆盖 |

### 4.2 用户角色 vs 系统内部组件

| 类型 | 示例 | 是否写入需求清单 |
|------|------|-----------------|
| **用户角色** | 主代理、拆解子代理、执行子代理 | ✅ 是 |
| **系统内部组件** | FlowTicket、Guardian、pytest | ❌ 否（是系统功能，不是需求来源） |

## 五、需求提取流程

### 5.1 流程步骤

```
1. 读取需求源文件
      │
      ▼
2. 识别用户角色（谁在使用系统？）
      │
      ▼
3. 提取用户场景（用户要做什么？）
      │
      ▼
4. 验证颗粒度（是否完整？是否可验收？）
      │
      ▼
5. 标注来源（哪个文档？哪个位置？）
      │
      ▼
6. 去重合并（相似需求合并）
      │
      ▼
7. 生成需求清单
```

### 5.2 AI 提示词模板

```
从需求源文件提取用户需求清单。

【重要】来源标注规则：
- 来源文档：必须是原始需求源文件名称（如：完整业务流程方案.md），绝对不能是"需求清单.md"自己！
- 来源位置：必须是源文档中的章节号（如：4.4.3节）或行号（如：L15-20）

需求描述要求：
1. 用户视角：从用户（主代理/拆解子代理/执行子代理/验证子代理）视角描述需求
2. 颗粒度适中：一个需求对应一个完整的用户场景
3. 不分级优先级：需求清单只记录需求，不分 P0/P1/P2

输出格式：
| ID | 需求描述 | 来源文档 | 来源位置 |
|----|----------|----------|----------|
| REQ-001 | [用户场景描述] | [原始源文件名] | [章节号] |

输出到：[需求清单文件路径]
```

## 六、Checklist

### 6.1 需求清单完整性检查

| ID | 检查项 | 标准 | 检查方式 |
|----|--------|------|---------|
| **C01** | 文档结构完整 | 包含文档信息、需求项列表、需求统计 | 人工/AI 检查 |
| **C02** | REQ-ID 格式正确 | 匹配 `REQ-\d{3}` 格式 | 正则匹配 |
| **C03** | 需求描述非空 | 每个需求描述 > 10 个字符 | 长度检查 |
| **C04** | 用户视角描述 | 使用用户角色作为主语 | 语义检查 |
| **C05** | 来源文档正确 | **来源文档必须是原始需求源文件**，不能是"需求清单.md"自己 | 文件名检查 |
| **C06** | 来源位置正确 | 来源位置是章节号或行号（如"4.4.3节"） | 格式检查 |
| **C07** | 无重复需求 | 相似需求已合并 | 去重检查 |
| **C08** | 颗粒度适中 | 不太粗、不太细 | 人工审核 |

### 6.2 需求覆盖率检查

| ID | 检查项 | 标准 | 检查方式 |
|----|--------|------|---------|
| **C11** | 所有用户角色覆盖 | 每个用户角色至少有 1 个需求 | 统计检查 |
| **C12** | 所有源文档覆盖 | 每个源文档至少提取 1 个需求 | 统计检查 |
| **C13** | 核心流程覆盖 | 启动、执行、验证、异常处理全覆盖 | 流程检查 |

### 6.3 检查执行方式

| 方式 | 适用场景 |
|------|---------|
| **自动检查** | C02, C03, C05, C06, C07, C11, C12 |
| **人工审核** | C01, C04, C08, C13 |
