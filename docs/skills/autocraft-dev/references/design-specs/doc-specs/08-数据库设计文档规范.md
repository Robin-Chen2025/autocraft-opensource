## 一、文档概述

数据库设计文档定义数据存储结构，是后端开发的核心依据。

**核心作用**：
- 定义数据模型
- 指导ORM模型创建
- 数据迁移依据

**前置依赖**：
- 系统功能设计（必须）：提供功能清单
- API设计（上游）：提供字段定义参考
- 技术方案（必须）：提供数据库类型选择

## 二、分阶段设计策略

### 2.1 数据库设计分阶段原则

数据库设计采用**表清单整体 + 详情分阶段**策略：

| 原则 | 说明 |
|------|------|
| **表清单整体** | 必须完整列出所有P0/P1/P2表（确保数据库架构正确） |
| **详情分阶段** | P0详细设计，P1/P2占位符 |

### 2.2 各优先级设计深度

| 优先级 | 表清单 | 表详情 |
|--------|---------|---------|
| **P0** | ✅ 必须列出 | ✅ 详细设计（完整字段、索引、约束） |
| **P1** | ✅ 必须列出 | ❌ 占位符（仅表名 + 用途说明） |
| **P2** | ✅ 必须列出 | ❌ 占位符（仅表名 + 用途说明） |

### 2.3 为什么表清单必须完整？

| 原因 | 说明 |
|------|------|
| **数据库架构正确** | 表清单完整才能确保表关联关系（ER图）设计正确 |
| **API设计对齐** | 表清单完整才能确保API接口有对应的数据存储 |
| **追溯矩阵完整** | 系统功能 → 数据库表，不能有遗漏 |

### 2.4 一个文档 vs 多个文档

| 方案 | 推荐 | 说明 |
|------|------|------|
| **一个文档** | ✅ 推荐 | 表清单完整、ER图完整、维护成本低 |
| **多个文档** | ❌ 不推荐 | 每个迭代单独文档会导致ER图不完整 |

### 2.5 设计指导

参见总纲文档《设计阶段文档规范（总纲）》第10.9节"分阶段设计策略汇总"。

## 4. 表结构详情

### 4.1 users（用户表）

**描述**：存储用户基本信息

**支持功能ID**：T-001（用户认证）

| 字段名 | 类型 | 长度 | 可空 | 默认值 | 说明 |
|--------|------|------|------|--------|------|
| id | INTEGER | - | NO | AUTO_INCREMENT | 主键 |
| username | VARCHAR | 50 | NO | - | 用户名，唯一 |
| email | VARCHAR | 100 | NO | - | 邮箱，唯一 |
| password_hash | VARCHAR | 255 | NO | - | 密码哈希 |
| status | TINYINT | - | NO | 1 | 状态：1=正常, 0=禁用 |
| created_at | DATETIME | - | NO | CURRENT_TIMESTAMP | 创建时间 |
| updated_at | DATETIME | - | NO | CURRENT_TIMESTAMP | 更新时间 |
| deleted_at | DATETIME | - | YES | NULL | 删除时间（软删除） |

**索引**：

| 索引名 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| PRIMARY | id | PRIMARY | 主键索引 |
| uk_username | username | UNIQUE | 用户名唯一 |
| uk_email | email | UNIQUE | 邮箱唯一 |
| idx_status | status | NORMAL | 状态查询 |

**外键**：无

## 四、命名规范

### 4.1 表命名

| 规则 | 示例 |
|------|------|
| 使用小写 | `users`, `questions` |
| 使用复数 | `users` ✅ `user` ❌ |
| 使用snake_case | `question_knowledge_points` |
| 关联表命名 | `{table1}_{table2}` |

### 4.2 字段命名

| 规则 | 示例 |
|------|------|
| 使用小写 | `username`, `created_at` |
| 使用snake_case | `password_hash` |
| 外键命名 | `{table}_id` |
| 布尔字段 | `is_{state}` |

### 4.3 索引命名

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 主键 | PRIMARY | - |
| 唯一索引 | `uk_{field}` | `uk_username` |
| 普通索引 | `idx_{field}` | `idx_status` |
| 组合索引 | `idx_{field1}_{field2}` | `idx_user_status` |

## 六、常见问题

### Q1: 什么时候用软删除？

**A**: 当数据有恢复需求或需要保留历史记录时使用软删除。用`deleted_at`字段，NULL表示未删除。

### Q2: 外键级联规则如何选择？

**A**: 
- **CASCADE**：主表删除时，关联数据也删除（如用户删除，题目也删除）
- **SET NULL**：主表删除时，外键设为NULL（如分类删除，文章的分类设为NULL）
- **RESTRICT**：有关联数据时禁止删除（如有关联题目，禁止删除知识点）

### Q3: JSON字段什么时候用？

**A**: 当数据结构不固定或不需要单独查询时使用，如图片列表、配置信息等。
