# 技术方案文档审核 Checklist

**版本：** v1.5  
**日期：** 2026-04-13  
**用途：** AI审核节点检查指导  
**整理人：** AutoCraft 团队

---

## 一、核心原则（审核底线）

### 🚫 三大禁止行为

| 编号 | 禁止行为 | 验证方法 | 违规处理 |
|------|----------|----------|----------|
| **B01** | 禁止技术选型无理由 | 每项技术必须说明"为什么选择" | ❌ 严重问题，必须修复 |
| **B02** | 禁止架构图缺失 | 系统架构必须有可视化架构图 | ❌ 严重问题，必须修复 |
| **B03** | 禁止与PRD技术栈冲突 | 技术选型必须与PRD定义的技术栈一致 | ❌ 严重问题，必须修复 |

### ✅ 三大必须行为

| 编号 | 必须行为 | 验证方法 |
|------|----------|----------|
| **M01** | 技术版本必须标注 | 所有关键技术标注版本号（如 Python 3.10+） |
| **M02** | 模块划分必须完整 | 每个模块有职责和技术栈说明 |
| **M03** | 安全策略必须覆盖三项 | 认证方式、数据保护、权限控制三项都有说明 |

## 二、审核前准备检查

### 2.1 输入材料完整性

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **P01** | PRD文件存在 | 待审核PRD文件路径存在 | 终止审核 |
| **P02** | 技术方案文件存在 | 待审核技术方案文件路径存在 | 续审但标记缺失 |
| **P03** | PRD技术栈提取成功 | 提取PRD中定义的技术栈（如有） | ⚠️ 续审但标记无PRD技术栈定义 |

### 2.2 准备合格标准

```
✅ P01, P02 全部通过 → 开始审核
❌ P01不通过 → 终止审核，提示补充PRD
⚠️ P03不通过 → 续审但标记"PRD未定义技术栈"（非强制要求）
```

> 📋 **说明**：PRD可能不定义具体技术栈，此时B03不适用。

| **E02** | 后端技术栈表格存在 | 2.1节存在且为表格形式 | 无后端技术栈表格 |
| **E03** | 前端技术栈表格存在 | 2.2节存在且为表格形式 | 无前端技术栈表格 |

### 3.2 技术选型质量类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E04** | 技术选型理由完整 | 技术栈表格必须有"选型理由"列，且每项技术有说明 | Python无选型理由 |
| **E05** | 技术版本标注完整 | 关键技术（语言、框架、数据库）均标注版本号，格式如`Python 3.10+` | 语言/框架/数据库任一缺少版本号 |
| **E06** | 技术栈覆盖完整 | 技术栈覆盖：语言、框架、ORM、数据库（≥4项） | 缺少ORM或数据库 |

### 3.3 系统架构类

| ID | 检查项 | 检测规则 | 不通过示例 |

| **E08** | 模块划分表格存在 | 3.2节存在且为表格形式 | 无模块划分表格 |
| **E09** | 模块职责说明完整 | 每个模块有"职责"列且非空 | 用户认证模块职责为空 |

### 3.4 目录结构类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E10** | 目录结构树存在 | 第4节包含目录结构树 | 无目录结构 |

### 3.5 部署方案类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E11** | 开发环境配置完整 | 5.1节配置项≥4项 | 开发环境仅2项配置 |
| **E12** | 生产环境配置完整 | 5.2节配置项≥4项 | 生产环境仅1项配置 |

### 3.6 安全策略类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E13** | 认证方式说明存在 | 6.1节有认证方式说明 | 无认证方式 |
| **E14** | 数据保护说明存在 | 6.2节有数据保护说明 | 无数据保护 |
| **E15** | 权限控制说明存在 | 6.3节有权限控制说明 | 无权限控制 |

### 3.7 内容一致性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E16** | 技术选型与PRD一致 | 技术栈选型与PRD定义的技术约束一致（如PRD要求高并发，技术方案应体现） | PRD要求"高并发"，技术方案未说明并发处理方案（理解偏差） |
| **E17** | 文档内部一致性 | 技术方案内同一概念在不同位置（架构设计、技术选型、性能方案、部署方案等）的描述必须保持一致，不得出现矛盾或冲突。检查维度：(1)架构一致：架构图与文字描述不能矛盾；(2)选型一致：技术选型表与架构描述中使用的框架一致；(3)性能一致：性能指标与超时/并发配置不矛盾 | 架构图标注"Redis缓存"，技术选型表无Redis（选型矛盾）；超时配置5分钟，性能指标要求"3秒响应"（性能矛盾） |
| **E17** | 无虚增技术栈 | 技术栈表格中的技术都有PRD或业务需求支撑，无凭空添加 | 技术方案添加Redis缓存，但PRD无性能要求（虚增） |

### 3.8 架构可行性验证类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E28** | 技术选型支持关键业务场景 | 核心技术选型能支撑关键业务需求：asyncio支持长时超时、AI调用机制有技术支撑、SQLite支持并发读写 | 技术方案未说明asyncio超时配置，无法支撑长时间任务执行 |

**E28模拟验证步骤**：
```
步骤1: 检查asyncio配置 → 确认有wait_for超时参数说明（符合业务需求的最长超时）
步骤2: 检查AI调用机制 → 确认有AI服务调用方式说明（如OpenClaw Agent、API调用等）
步骤3: 检查SQLite并发方案 → 确认有WAL模式或连接池配置说明
步骤4: 输出技术可行性报告 → 标记无法支撑的业务场景
```

### 3.9 部署架构完整性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E29** | 前后端通信架构定义 | 开发环境必须明确定义前端如何连接后端API（Vite proxy / 环境变量 / 直连），不能只有启动命令 | 部署方案只写了启动命令，未说明前端API请求如何到达后端 |
| **E30** | 端口规划与系统边界 | 当项目与同服务器其他系统共存时，必须明确定义各系统的端口分配，且API地址默认值必须属于本项目 | 技术方案使用8402作为前端端口，但8402是另一个系统的端口 |

**E29验证步骤**：
```
步骤1: 检查部署方案5.1节 → 确认有"前后端通信架构"说明
步骤2: 检查通信方式 → 确认至少定义了一种方式（Vite proxy 或 环境变量baseURL）
步骤3: 检查vite.config.ts配置 → 确认proxy配置存在（如使用proxy方式）
步骤4: 检查环境变量 → 确认.env文件规范存在（如使用环境变量方式）
```

**E30验证步骤**：
```
步骤1: 检查部署方案 → 确认有端口分配表或端口说明
步骤2: 检查同服务器其他系统 → 如有，确认各系统端口不冲突
步骤3: 检查代码默认值 → config/index.ts等文件的默认端口必须属于本项目
步骤4: 输出端口规划报告 → 标记冲突或混用问题
```

### 3.9 阻断项合格标准

```
✅ E01-E30 全部通过 → 继续评分判断
❌ 任一项不通过 → 标记严重问题，必须修复
```

## 五、审核结果判断

### 5.1 阻断项优先判断

```
第一步：检查阻断项 E01-E15
    ├─ 有不通过项 → 无论评分多少，必须修复
    └─ 全部通过 → 进入第二步评分判断
```

### 5.2 评分判断规则

| 评分 | 警告项状态 | 处理 |
|------|-----------|------|
| ≥90分 | 全部通过 | ✅ 定稿 |
| ≥90分 | 有中等项 | ⚠️ 可定稿，建议修复 |
| 80-89分 | 全部通过 | ⚠️ 小修后定稿 |
| 80-89分 | 有中等项 | ⚠️ 修复后定稿 |
| 70-79分 | - | ⚠️ 修复后重新审核 |
| <70分 | - | ❌ 重新生成 |

### 5.3 问题分级与修复优先级

| 问题级别 | 来源 | 修复优先级 |
|---------|------|-----------|
| **严重** | 阻断项不通过（E01-E15） | 必须修复 |
| **中等** | 警告项W03-W06/W08/W12/W18-W26/W28不通过 | 建议修复 |
| **轻微** | 警告项W01/W02/W07/W09/W10/W11/W13-W17/W27不通过 | 可选修复 |

## 八、与PRD的分工约定

### 8.1 职责边界

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **PRD** | 业务需求 + **技术栈要求**（可选） | "系统需支持高并发，建议使用FastAPI" |
| **技术方案** | 技术选型 + 版本 + 架构 + 目录 + 部署 | "后端：Python 3.10+ + FastAPI 0.100+" |

### 8.2 技术栈约定

| 场景 | PRD定义 | 技术方案处理 |
|------|---------|-------------|
| PRD明确技术栈 | "使用Vue 3 + FastAPI" | 必须一致（B03/W03检查） |
| PRD未定义技术栈 | 无技术要求 | 技术方案自由选型 |
| PRD定义技术原则 | "前端需响应式" | 技术方案选择符合原则的技术 |

### 8.3 审核时的注意事项

- 技术方案文档定义**技术选型和架构设计**
- 不应包含业务逻辑细节（由系统功能设计定义）
- 如PRD定义技术栈，技术方案选型必须一致
- 如PRD未定义技术栈，技术方案需有完整选型理由

## 九、来源追溯（业界标准对照）

### 9.1 业界标准对照

| 我们的检查项 | 业界标准 | 核心要求 | 来源 |
|-------------|---------|---------|------|
| 技术选型表格 | **技术选型对比表** | 技术、版本、选型理由 | 腾讯云技术方案 |
