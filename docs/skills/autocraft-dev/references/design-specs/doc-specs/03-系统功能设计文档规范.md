# 系统功能设计文档规范

**版本：** v2.0  
**日期：** 2026-04-10

---

## 一、文档概述

系统功能设计文档在PRD的基础上，进行详细的功能设计和拆分，识别公共功能和技术功能。

**前置依赖**：
- PRD（必须）：提供功能需求ID（FR-001...）、验收标准
- 业务流程文档（强烈建议）：提供用户操作步骤，P0功能必须有业务流程支撑

**核心作用**：
- 将PRD业务需求转化为系统功能
- 识别多业务共享的公共功能
- 为API/数据库/UI设计提供功能基准
- 建立完整的功能追溯链

---

## 二、文档定位

```
PRD（业务目标+范围）
        │ 功能需求ID（FR-001...）
        │
        ▼
业务流程文档（用户操作路径）
        │ 流程步骤ID（S001...）
        │
        ▼
┌─────────────────────────────────────┐
│ 系统功能设计文档                      │
│ - 详细功能清单（F-001...）           │
│ - 公共功能识别                       │
│ - 功能追溯矩阵                       │
└─────────────────┬───────────────────┘
                  │
        ┌─────────┼─────────┐
        ▼         ▼         ▼
    API设计   数据库设计   UI设计
```

**文档依赖关系**：
- **必须输入**：PRD（功能需求ID、验收标准）
- **辅助输入**：业务流程文档（用户操作步骤，非必须但有则更好）

## 三、分阶段设计策略

### 3.1 系统功能分阶段原则

系统功能采用**功能清单整体 + 详情分阶段**策略：

| 原则 | 说明 |
|------|------|
| **功能清单整体** | 必须完整列出所有P0/P1/P2功能（满足追溯矩阵100%覆盖） |
| **详情分阶段** | P0详细设计，P1框架设计，P2占位符 |

### 3.2 各优先级设计深度

| 优先级 | 功能清单 | 功能详情 |
|--------|---------|---------|
| **P0** | ✅ 必须列出 | ✅ 详细设计（完整描述、输入输出、业务规则） |
| **P1** | ✅ 必须列出 | ⚠️ 框架设计（概述 + 关键输入输出） |
| **P2** | ✅ 必须列出 | ❌ 占位符（仅名称 + 功能ID） |

### 3.3 为什么功能清单必须完整？

| 原因 | 说明 |
|------|------|
| **追溯矩阵完整性** | PRD功能需求 → 系统功能，不能有遗漏 |
| **架构一次性规划** | 模块划分、公共功能识别需要整体视角 |
| **避免重复设计** | 后续补设计时发现P0功能接口遗漏 |

### 3.4 单文档 vs 分模块文档

| 项目规模 | 推荐 | 说明 |
|---------|------|------|
| **小型**（FR≤3个，预计<800行）| ✅ 单文档 | 功能清单完整、追溯矩阵完整、维护成本低 |
| **中型**（FR 4-6个，预计800-1500行）| ⚠️ 可选拆分 | 根据复杂度决定 |
| **大型**（FR≥7个，预计>1500行）| ✅ **分模块文档** | 避免AI生成截断、审核遗漏 |

### 3.5 分模块文档策略

**适用场景**：
- FR数量≥5个，且每个FR包含多个系统功能
- 预期单文档超过1500行
- 需要并行开发，模块独立推进

**文档结构**：

```
design-docs/{项目名}/
├── 系统功能设计/
│   ├── 00-功能清单总览.md          # 汇总表 + 模块索引 + PRD覆盖确认
│   ├── 01-{模块1名称}-FR001.md        # 对应FR-001
│   ├── 02-{模块2名称}-FR002.md    # 对应FR-002
│   ├── 03-{核心模块}-FR003.md  # 对应FR-003（最大）
│   ├── ...                         # 其他模块
│   └── 公共功能识别.md             # 跨模块公共功能
```

**单模块文档长度**：300-500行（GLM-5舒适区）

**00-功能清单总览.md 必须包含**：
- 模块索引表
- PRD功能需求覆盖确认表
- 功能追溯矩阵汇总（所有模块合并）

**分模块优势**：
1. **AI生成可控**：每个文档在GLM-5舒适区内
2. **审核更细致**：每个模块独立审核，降低遗漏风险
3. **并行效率高**：可以同时生成多个模块
4. **维护更方便**：修改一个模块不影响其他

### 3.3 功能详情

#### F-001: 创建题目

**功能描述**：用户录入新题目，包含题目内容、图片、关联知识点

**关联业务流程步骤**：S001-S004（题目录入流程）

**业务术语**：
| 术语 | 说明 | 业务规则 |
|------|------|----------|
| 题目内容 | 用户输入的文字描述 | 不能为空，业务最大长度2000字 |
| 图片 | 用户上传的图片文件 | 最多10张，单张不超过5MB |
| 知识点 | 题目关联的知识分类 | 最多关联5个知识点 |

> ⚠️ **分工约定**：系统功能文档定义**业务术语和业务规则**（用户能理解），API设计文档定义**技术字段名和类型**（开发者能执行）。两者通过术语↔字段映射表关联。

**处理逻辑**：
1. 验证题目内容不为空
2. 处理图片上传（调用公共功能F-010）
3. 创建题目记录
4. 创建题目-知识点关联记录
5. 返回题目ID

**输出**：
| 业务术语 | 说明 |
|----------|------|
| 题目ID | 系统生成的唯一标识 |
| 创建时间 | 题目创建的时间戳 |

**业务规则**：
- 题目内容不能为空
- 单个题目最多关联5个知识点
- 图片最多上传10张

## 九、与PRD/业务流程的关系

### 7.1 与PRD分工

| 文档 | 职责 | 内容 |
|------|------|------|
| **PRD** | 业务需求 | 功能需求（业务视角）、用户故事、验收标准 |
| **系统功能设计** | 技术设计 | 详细功能清单、**业务术语+业务规则**、公共功能、技术功能、追溯矩阵 |

**关键区别**：
- PRD的"功能需求"是业务目标，用户能理解
- 系统功能设计的"功能"是技术实现，开发者能执行

### 7.2 与业务流程分工

| 文档 | 职责 | 内容 |
|------|------|------|
| **业务流程文档** | 操作路径 | 用户操作步骤（S001...）、RACI矩阵、异常处理 |
| **系统功能设计** | 功能支撑 | 系统功能（F-001...）支撑哪些操作步骤 |

**关联方式**：
- 业务流程描述"用户怎么操作"
- 系统功能描述"系统怎么支撑"
- 通过6.5追溯表建立关联

### 7.3 与API设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **系统功能设计** | 业务术语 + 业务规则 | "题目内容（不能为空，最长2000字）" |
| **API设计** | 技术字段名 + 类型 + 验证规则 | `content: string, required, maxLength=2000` |

> ⚠️ **禁止重复**：系统功能文档不定义技术字段名（如 `content`），API设计文档不重复业务规则描述。两者通过术语↔字段映射表关联。
