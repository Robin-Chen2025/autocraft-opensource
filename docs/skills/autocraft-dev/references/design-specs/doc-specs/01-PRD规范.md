## 零、需求预处理（PRD生成前置步骤）

### 0.1 生成方式

**推荐方式：主代理交互式生成**

| 方式 | 适用场景 | 流程 |
|------|---------|------|
| **主代理交互式** | 默认方式 | 主代理理解需求来源 → **交互确认框架** → 生成PRD → 子代理审核 |
| **需求清单方式** | 需求来源复杂时 | 先生成需求清单 → Crush生成PRD → 子代理审核 |

> **需求清单定位**：需求清单是 PRD 的框架/骨架，定义功能模块清单和优先级。
>
> - **交互式生成**：先与用户确认框架（口头或文档），再填充详细内容
> - **需求清单方式**：先生成需求清单文档，再基于清单生成 PRD
>
> 两种方式本质相同：先确定框架，再填充内容。

### 0.2 需求来源标注

PRD 功能需求的需求来源直接引用需求来源文档章节：

```markdown
| ID | 功能需求 | 需求来源 |
|----|----------|----------|
| FR-001 | 项目管理 | 需求来源文档-4.1 |
| FR-002 | 计划管理 | 需求来源文档-3.1 |
```

**格式**：`{文档名称}-{章节号}`

### 0.3 核心原则

| 原则 | 说明 |
|------|------|
| **用户视角** | 从用户使用场景描述需求，而非技术实现 |
| **完整内容** | PRD 内容完整，不分 P0/P1/P2 阶段 |
| **来源可追溯** | 每个功能需求标注来源文档章节 |
| **子代理审核** | 生成后由子代理按 Checklist 独立审核 |

### 0.4 需求清单（可选）

需求清单仅在需求来源复杂时使用，详细规范见：

**[00-需求清单规范.md](./00-需求清单规范.md)**

## 一、PRD 概述

### 1.1 什么是PRD

PRD（Product Requirements Document，产品需求文档）是产品开发的核心文档，定义产品要做什么、为谁做、如何衡量成功。

### 1.2 PRD 的核心作用

| 作用 | 说明 |
|------|------|
| **沟通桥梁** | 让用户、开发、测试对需求达成共识 |
| **开发依据** | 开发人员据此进行技术设计和编码 |
| **测试依据** | 测试人员据此编写测试用例 |
| **验收依据** | 产品交付时的验收标准 |

### 1.3 PRD 与其他文档的关系

```
PRD（功能需求ID: FR-001...）
    │
    ▼
系统功能设计文档（系统功能ID: F-001...）
    │
    ├─→ API设计文档
    ├─→ 数据库设计文档
    └─→ UI设计文档
```

## 二、PRD 核心要素

| 要素 | 必须性 | 说明 |
|------|--------|------|
| **执行摘要** | ✅ 必须 | 产品愿景、核心价值 |
| **问题陈述** | ✅ 必须 | 当前痛点、市场机会、业务背景 |
| **目标与成功指标** | ✅ 必须 | 业务目标、可量化指标 |
| **目标用户角色** | ✅ 必须 | 用户群体及其核心需求 |
| **功能需求** | ✅ 必须 | 功能需求清单（带ID）、优先级 |
| **非功能需求** | ✅ 必须 | 性能、安全、可用性要求 |
| **验收标准** | ✅ 必须 | 可测试的验收条件 |
| **风险与依赖** | ⚠️ 建议 | 技术风险、外部依赖 |
| **用户故事** | ⚠️ 可选 | 主要用于团队沟通，AI不需要 |

## 三、PRD内容策略

### 3.1 PRD不分阶段

PRD 采用**完整内容**策略，不分 P0/P1/P2 阶段：

| 策略 | 说明 |
|------|------|
| **完整内容** | 所有功能需求完整描述，不区分优先级 |
| **不分阶段** | PRD 本身不分阶段，下游文档才分阶段 |
| **优先级标记** | 功能需求可标记优先级（P0/P1/P2），但内容都完整 |

### 3.2 下游文档分阶段传递

PRD 功能需求的优先级标记传递到下游文档：

```
PRD功能需求优先级标记（P0/P1/P2，内容都完整）
    │
    ▼
业务流程设计：P0详细、P1/P2框架
    │
    ▼
系统功能设计：功能清单完整、P0详细、P1框架、P2占位符
    │
    ▼
API/UI/数据库设计：清单完整、P0详细、P1/P2占位符
```

### 3.3 设计指导

参见总纲文档《设计阶段文档规范》第10.9节"分阶段设计策略汇总"。

## 5. 功能需求

> **定位**：本章定义业务功能需求的目标和范围，作为系统功能设计的依据和最终验收标准。详细功能设计在《系统功能设计文档》中完成。

### 5.1 功能需求清单

| ID | 功能模块 | 功能需求 | 需求来源ID | 优先级 | 系统功能ID |
|----|----------|----------|-----------|--------|-----------|
| FR-001 | [模块A] | [功能需求1描述] | REQ-001 | P0 | F-001, F-002 |
| FR-002 | [模块A] | [功能需求2描述] | REQ-002 | P1 | F-003 |
| FR-003 | [模块B] | [功能需求3描述] | REQ-003 | P2 | F-004 |

**ID说明**：
- **ID (FR-XXX)**：功能需求ID，用于追溯和验收
- **需求来源ID (REQ-XXX)**：对应《需求清单》中的原始需求ID，每个FR必须有来源
- **系统功能ID**：对应《系统功能设计文档》中的详细功能ID

**优先级说明（MoSCoW）**：
- **P0 (Must Have)**：必须有，否则产品无法使用，≤5个
- **P1 (Should Have)**：应该有，提升用户体验
- **P2 (Could Have)**：可以有，锦上添花
- **P3 (Won't Have)**：本期不做，放入backlog

## 四、验收标准规范

### 4.1 验收标准格式（BDD风格）

```
Given [前置条件]
When [操作]
Then [预期结果]
```

### 4.2 验收标准示例

#### ✅ 好的验收标准

```
AC-001: 题目录入成功

Given 用户已登录
When 用户填写题目内容并选择知识点，点击保存
Then 
  - 题目保存成功
  - 页面显示"保存成功"提示
  - 题目列表中显示新题目
```

**为什么好**：前置条件明确、操作步骤可执行、预期结果可验证

#### ❌ 不好的验收标准

```
题目录入功能正常工作。
```

**为什么不好**：没有前置条件、没有具体操作、无法验证

### 4.3 验收标准覆盖维度

| 维度 | 示例 |
|------|------|
| **正常流程** | 输入正确数据，成功保存 |
| **异常流程** | 输入空数据，提示错误 |
| **边界条件** | 输入最大长度，正常处理 |
| **权限控制** | 未登录用户，跳转登录页 |

## 五、功能优先级规范

### 5.1 MoSCoW 优先级

| 优先级 | 名称 | 说明 | 数量限制 |
|--------|------|------|----------|
| **P0** | Must Have | 必须有，否则产品无法使用 | ≤ 5个 |
| **P1** | Should Have | 应该有，提升用户体验 | 不限 |
| **P2** | Could Have | 可以有，锦上添花 | 不限 |
| **P3** | Won't Have | 本期不做，放入backlog | 不限 |

### 5.2 优先级判断标准

| 问题 | P0 | P1 | P2 |
|------|-----|-----|-----|
| 没有这个功能，产品能用吗？ | 不能 | 能，但体验差 | 能 |
| 这个功能是核心卖点吗？ | 是 | 辅助 | 附加 |
| 用户愿意为此付费吗？ | 愿意 | 可能 | 不一定 |

## 六、PRD 验收 Checklist

### 6.1 完整性检查

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **执行摘要** | 产品愿景、核心价值清晰 | ☐ |
| **问题陈述** | 当前痛点、市场机会明确 | ☐ |
| **成功指标** | 指标可量化、可衡量 | ☐ |

### 6.2 用户角色检查

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **角色定义** | 明确目标用户群体 | ☐ |
| **核心需求** | 每个角色的核心需求清晰 | ☐ |

### 6.3 功能需求检查

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **ID唯一** | 每个功能需求有唯一ID | ☐ |
| **优先级合理** | P0 ≤ 5个，且都是核心功能 | ☐ |
| **范围明确** | 不在范围内的功能已列出 | ☐ |

### 6.4 验收标准检查

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **可测试** | 每个验收标准可测试验证 | ☐ |
| **覆盖完整** | 每个P0功能有对应验收标准 | ☐ |

### 6.5 非功能需求检查

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **性能要求** | 有响应时间、并发量要求 | ☐ |
| **安全要求** | 有认证、授权说明 | ☐ |
| **兼容性** | 有浏览器、设备兼容性要求 | ☐ |

### 6.6 评审确认

| 检查项 | 标准 | 通过 |
|--------|------|------|
| **干系人评审** | 用户已评审并签字 | ☐ |

## 七、PRD 编写最佳实践

### 7.1 写作原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **用用户语言** | 避免技术术语 | ✅ "录入题目" ❌ "POST /api/questions" |
| **具体明确** | 避免模糊表达 | ✅ "响应时间 < 2秒" ❌ "响应要快" |
| **可验证** | 需求可测试 | ✅ "显示保存成功提示" ❌ "用户体验好" |

### 7.2 常见错误

| 错误 | 问题 | 正确做法 |
|------|------|----------|
| 技术细节过多 | 混淆需求和设计 | 只写用户视角，不写实现 |
| 模糊表达 | "大约"、"可能"、"尽量" | 用具体数字和标准 |
| 缺少边界 | 没说做什么、不做什么 | 明确范围边界 |
| 优先级过多 | P0太多，失去焦点 | P0 ≤ 5个核心功能 |

### 7.3 PRD 与技术文档的边界

| 内容 | 属于PRD | 属于技术文档 |
|------|---------|-------------|
| 用户输入什么 | ✅ | |
| 系统输出什么 | ✅ | |
| 业务规则 | ✅ | |
| 数据库表结构 | | ✅ |
| API接口定义 | | ✅ |

## 八、用户故事规范（可选）

> **说明**：用户故事主要用于团队沟通，AI生成设计文档时可直接使用功能需求清单。如需编写用户故事，参考以下规范。

### 8.1 用户故事格式

```
As a [角色]
I want [功能]
So that [价值]
```

### 8.2 用户故事示例

#### ✅ 好的用户故事

```
As a 学生
I want 录入错题并关联知识点
So that 我可以按知识点分类复习错题
```

**为什么好**：角色明确、功能具体、价值清晰

#### ❌ 不好的用户故事

```
用户想要一个好用的错题本。
```

**为什么不好**：角色模糊、功能不具体、价值不明确
