## 一、核心原则（审核底线）

### 🚫 三大禁止行为

| 编号 | 禁止行为 | 验证方法 | 违规处理 |
|------|----------|----------|----------|
| **B01** | 禁止超出PRD范围 | 检查2.2章节是否有"本设计不超出PRD定义的功能范围"确认语 | ❌ 严重问题，必须修复 |
| **B02** | 禁止遗漏PRD功能需求 | 所有FR-XXX在2.1表格中有覆盖状态 | ❌ 严重问题，必须修复 |
| **B03** | 禁止功能ID重复 | 每个F-XXX在文档中只出现一次 | ❌ 严重问题，必须修复 |

### ✅ 三大必须行为

| 编号 | 必须行为 | 验证方法 |
|------|----------|----------|
| **M01** | 必须追溯PRD | 每个系统功能有PRD功能需求ID（FR-XXX） |
| **M02** | 必须定义业务术语 | 功能详情包含业务术语表格 |
| **M03** | 必须完整追溯矩阵 | 四个追溯表格（PRD→F、F→API、F→DB、F→UI）都存在 |

## 二、分模块审核策略

### 2.1 文档类型定义

当系统功能设计采用分模块文档策略时（FR≥4个），分为两类文档：

| 文档类型 | 文件名 | 职责 |
|---------|--------|------|
| **总览文档** | 00-功能清单总览.md | 模块索引 + PRD覆盖确认 + 追溯矩阵汇总 |
| **详细设计文档** | 01-XXX-FR001.md 等 | 功能清单 + 功能详情 + 完整追溯矩阵 |

### 2.2 检查项适用规则

| 检查项类型 | 总览文档 | 详细设计文档 | 说明 |
|-----------|---------|-------------|------|
| **E01-E03** PRD覆盖类 | ✅ 必须检查 | ✅ 必须检查 | 两类文档都需要 |
| **E14** 文档内部一致性 | ✅ 必须检查 | ✅ 必须检查 | 两类文档都需要 |
| **E04** 模块划分 | ✅ 必须检查（模块索引表） | ✅ 必须检查（功能归属模块） | |
| **E05** 详细功能清单 | ❌ 不适用 | ✅ 必须检查 | 总览只有索引，详细清单在各模块文档 |
| **E06** 功能ID格式 | ⚠️ 检查汇总表无重复 | ✅ 必须检查 | |
| **E07-E11** 追溯矩阵类 | ✅ 必须检查（框架存在） | ✅ 必须检查（完整追溯） | 总览有框架，详细设计有完整内容 |
| **E12-E13** 内容一致性 | ✅ 必须检查 | ✅ 必须检查 | 两类文档都需要 |
| **W01-W28** 警告项 | ⚠️ 部分适用 | ✅ 全部适用 | 见下表详细说明 |

### 2.3 警告项适用规则

| 警告项 | 总览文档 | 详细设计文档 | 说明 |
|--------|---------|-------------|------|
| **W01-W02** 文档信息/版本历史 | ✅ 必须 | ✅ 必须 | |
| **W03-W06** 功能详情 | ❌ 不适用 | ✅ 必须 | 在详细设计文档中检查 |
| **W07-W09** 公共功能 | ⚠️ 汇总即可 | ✅ 必须详细 | 总览识别公共功能列表，详细设计定义详情 |
| **W10-W12** 技术功能 | ⚠️ 汇总即可 | ✅ 必须详细 | |
| **W13-W15** 功能清单列 | ✅ 必须（模块索引表） | ✅ 必须（功能清单表） | |
| **W16-W20** 追溯矩阵 | ✅ 必须（框架） | ✅ 必须（完整） | |
| **W21-W22** AI执行适配 | ❌ 不适用 | ✅ 必须 | 在详细设计文档中检查 |
| **W23-W25** 分工约定 | ⚠️ 检查索引表 | ✅ 必须检查 | |
| **W26-W28** 完整性检查 | ✅ 必须 | ✅ 必须 | |

### 2.4 审核流程

```
审核开始
    │
    ├─ 判断文档类型
    │     ├─ 文件名以"00-"开头 → 总览文档
    │     └─ 文件名以"01-"~"99-"开头 → 详细设计文档
    │
    ├─ 总览文档审核
    │     ├─ 检查 E01-E04, E06-E14（排除E05）
    │     ├─ 检查适用的警告项
    │     └─ 输出审核报告
    │
    └─ 详细设计文档审核
          ├─ 检查 E01-E15 全部
          ├─ 检查 W01-W28 全部
          └─ 输出审核报告
```

### 2.5 总览文档特殊要求

| 要求 | 说明 |
|------|------|
| **来源文档信息** | 文档信息表必须包含"来源文档"字段，列出PRD、业务流程等输入文件路径 |
| **模块索引完整** | 模块索引表包含：模块ID、模块名称、对应FR、文档路径、状态 |
| **追溯矩阵框架** | 5个追溯表（5.1~5.5）框架存在，内容可以是"待各模块文档生成后汇总" |

## 三、审核前准备检查

### 2.1 输入材料完整性

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **P01** | PRD文件存在 | 待审核PRD文件路径存在 | 终止审核 |
| **P02** | 业务流程文件存在 | 待审核业务流程文件路径存在 | ⚠️ 续审但标记缺失（非必须） |
| **P03** | 系统功能设计文件存在 | 待审核系统功能设计文件路径存在 | 续审但标记缺失 |
| **P04** | PRD功能需求列表提取成功 | 提取FR-XXX列表成功 | 终止审核，提示PRD格式问题 |

### 2.2 准备合格标准

```
✅ P01, P03, P04 全部通过 → 开始审核
❌ P01或P04不通过 → 终止审核，提示补充PRD
⚠️ P02不通过 → 续审但标记"无业务流程文档"（非必须输入）
⚠️ P03不通过 → 续审但标记"无系统功能设计文档"
```

> 📋 **说明**：业务流程文档是系统功能设计的**辅助输入**（非必须）。如存在，应检查功能与业务流程步骤的关联。

## 三、阻断项检查 [Error] - 必须通过

### 3.1 PRD覆盖类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E01** | PRD覆盖确认表存在 | 2.1章节存在表格，包含FR-XXX和覆盖状态列 | 无PRD覆盖确认表 |
| **E02** | 所有FR已覆盖 | 2.1表格中所有FR状态为"✅ 已覆盖"，无"⚠️ 部分覆盖"或"❌ 未覆盖" | FR-003状态为"❌ 未覆盖" |
| **E03** | 范围约束确认 | 2.2章节包含"本设计不超出PRD定义的功能范围"语句 | 无范围约束确认 |

### 3.2 功能清单类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E04** | 功能清单存在 | 3.1章节存在功能清单表格，包含F-XXX、功能名称、功能描述 | 无功能清单 |
| **E05** | 功能详情存在 | 3.2章节存在各功能详情，包含业务术语表、处理逻辑、业务规则、验收标准 | 无功能详情 |
| **E06** | 功能ID格式正确 | 匹配正则`F-\d{3}`，无重复 | F-1、F-001出现两次 |

### 3.3 追溯矩阵类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E07** | PRD→系统功能追溯存在 | 4.1章节存在追溯表格，包含FR-XXX和F-XXX | 无PRD→系统功能追溯 |
| **E08** | 系统功能→API追溯存在 | 4.2章节存在追溯表格，包含F-XXX和API路径 | 无系统功能→API追溯 |
| **E09** | 系统功能→数据库追溯存在 | 4.3章节存在追溯表格，包含F-XXX和数据库表名 | 无系统功能→数据库追溯 |
| **E10** | 系统功能→UI追溯存在 | 4.4章节存在追溯表格，包含F-XXX和UI页面路径 | 无系统功能→UI追溯 |

### 3.4 ID规范性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E11** | 模块ID格式正确 | 匹配正则`M-\d{2}` | M-1、无模块ID |

### 3.5 内容一致性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E12** | 功能内容与PRD描述一致 | 功能详情的业务规则、处理逻辑与PRD描述保持一致，无理解偏差、过度理解或简化 | PRD定义"题目内容最长2000字"，系统功能设计写"最长1000字"（简化） |
| **E13** | 无虚增功能 | 功能清单中的所有功能都能追溯到PRD的FR-XXX，无凭空添加 | 功能清单有F-010"批量导入"，但PRD无对应FR（虚增） |
| **E14** | 文档内部一致性 | 文档内同一概念在不同位置（功能清单、功能详情、业务术语表、业务规则、验收标准等）的描述必须保持一致，不得出现矛盾或冲突。检查维度：(1)数据流一致：上游功能声明的输出/能力，下游功能必须有对应处理；(2)业务规则一致：同一业务规则在不同功能中的描述不能矛盾；(3)术语一致：业务术语表的定义与功能详情中的使用保持一致；(4)验收标准一致：验收标准与功能描述、业务规则不矛盾 | F-001声明支持.md/.json上传，F-002验证只处理Markdown格式（数据流断裂）；F-001业务规则"文件≤10MB"，F-002验收标准"文件≤5MB"（规则矛盾） |

### 3.6 功能调用链验证类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E15** | 核心模块功能调用链完整性 | 文档中标识为核心模块的，其功能之间的调用/依赖关系必须完整——上游功能声明的下游调用，在下游功能详情中有对应的处理说明；下游功能声明被上游调用，在上游功能详情中有对应的发起说明 | F-001启动功能详情声明"调用F-002调度"，但F-002功能详情中无被F-001调用的说明 |

**E15验证步骤**：
```
步骤1: 识别文档中的核心模块（优先级P0的模块）
步骤2: 提取核心模块内各功能详情中声明的调用关系（"调用F-XXX"）
步骤3: 验证每个声明的调用关系在被调用方功能详情中有对应说明
步骤4: 验证被调用方声明的上游来源与调用方一致
步骤5: 输出调用链完整性报告 → 标记缺失的调用关系
```

### 3.7 阻断项合格标准

```
✅ E01-E15 全部通过 → 继续评分判断
❌ 任一项不通过 → 标记严重问题，必须修复
```

## 五、审核结果判断

### 5.1 阻断项优先判断

```
第一步：检查阻断项 E01-E15
    ├─ 有不通过项 → 无论评分多少，必须修复
    └─ 全部通过 → 进入第二步评分判断
```

### 5.2 评分判断规则

| 评分 | 警告项状态 | 处理 |
|------|-----------|------|
| ≥90分 | 全部通过 | ✅ 定稿 |
| ≥90分 | 有中等项 | ⚠️ 可定稿，建议修复 |
| 80-89分 | 全部通过 | ⚠️ 小修后定稿 |
| 80-89分 | 有中等项 | ⚠️ 修复后定稿 |
| 70-79分 | - | ⚠️ 修复后重新审核 |
| <70分 | - | ❌ 重新生成 |

### 5.3 问题分级与修复优先级

| 问题级别 | 来源 | 修复优先级 |
|---------|------|-----------|
| **严重** | 阻断项不通过（E01-E15） | 必须修复 |
| **中等** | 警告项W01/W03-W07/W10/W13/W15/W18-W19/W21/W23-W24/W26-W28不通过 | 建议修复 |
| **轻微** | 警告项W02/W08-W09/W11-W12/W14/W16-W17/W20/W22/W25不通过 | 可选修复 |

## 八、与PRD/API设计/业务流程的分工约定

### 8.1 与PRD分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **PRD** | 业务需求（业务视角） | "用户可以录入题目" |
| **系统功能设计** | 功能拆分（技术视角） | F-001创建题目、F-002查询题目列表... |

### 8.2 与API设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **系统功能设计** | 业务术语 + 业务规则 | "题目内容（不能为空，最长2000字）" |
| **API设计** | 技术字段名 + 类型 + 验证规则 | `content: string, required, maxLength=2000` |

### 8.3 与业务流程分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **业务流程文档** | 用户操作步骤 | S001: 用户点击"新增题目" → S002: 填写内容... |
| **系统功能设计** | 系统功能支撑 | F-001创建题目（支撑S001-S004） |

**关联方式**：
- 系统功能详情可标注"关联业务流程步骤：S001-S004"
- P0功能必须有对应的业务流程文档支撑（W18）

**审核时的注意事项**：
- 系统功能设计文档定义**业务术语和业务规则**（用户能理解）
- 不应包含技术字段名（如`content`），这由API设计文档定义
- 如发现系统功能设计包含技术字段定义，标记为轻微问题
- 如有业务流程文档，应检查功能与流程步骤的关联（W17）

## 九、来源追溯（联查确认）

### 9.1 业界标准对照

| 我们的定义 | 业界标准 | 核心要求 | 来源 |
|-----------|---------|---------|------|
| 系统功能清单 | **WBS工作分解结构**（工作包层） | 项目→任务→工作→日常活动层级分解 | 百度百科/腾讯云/CSDN |
| 功能追溯矩阵 | **需求追溯矩阵(RTM)** | 需求→设计→编码→测试全流程映射 | 腾讯云/百度百科 |
| 功能模块划分 | WBS第一/二层分解 | 可交付成果导向 | 腾讯云 |
| PRD→功能映射 | RTM"需求→设计"追溯 | 前向追溯 | 腾讯云 |
| 业务流程→功能映射 | RTM扩展追溯 | 补充操作步骤关联 | 我们定义 |

### 9.2 RTM需求追溯矩阵核心要求

**定义**（腾讯云/百度百科）：
> 需求跟踪矩阵(Requirement Traceability Matrix)，把产品需求从其来源连接到能满足需求的可交付成果的一种表格，跟踪从需求到设计、从设计到编码、从编码到测试的全流程映射过程。

**核心功能**：
- 前向追溯：需求 → 设计/实现/测试
- 后向追溯：缺陷/测试 → 需求/业务目标
- 需求完整性保障：确保每个需求都有对应验证
- 变更影响控制：快速定位变更影响范围

**标准字段**：

| 字段 | 说明 | 对应我们的检查项 |
|------|------|-----------------|
| 需求编号 | 唯一标识 | FR-XXX |
| 需求描述 | 需求内容 | PRD功能需求 |
| 设计编号 | 对应设计文档 | F-XXX |
| 代码模块 | 对应代码 | （API设计中） |
| 测试用例 | 对应测试 | AC-XXX |
| 状态 | 实现状态 | 覆盖状态 |

### 9.3 WBS工作分解结构核心要求

**定义**（百度百科/腾讯云）：
> 工作分解结构(Work Breakdown Structure)，以可交付成果为导向对项目要素进行分组，每下降一层代表对项目工作的更详细定义。

**核心原则**：

| 原则 | 说明 | 对应我们的检查项 |
|------|------|-----------------|
| **100%原则** | 子项之和等于父项，无遗漏 | E02（FR全覆盖） |
| **MECE原则** | 相互独立、完全穷尽 | E06（功能ID唯一） |
| **可交付成果导向** | 按产出物分解，非按活动 | E04（模块划分） |
| **层级编码** | 反映层级关系 | E11-E12（ID格式） |

### 9.4 我们的定义与业界标准的差异

| 维度 | 业界标准 | 我们的定义 | 差异说明 |
|------|---------|-----------|---------|
| 编码体系 | WBS层级编码(1.1.1) | 模块ID(M-01) + 功能ID(F-001) | 我们采用双ID系统，更灵活 |
| 追溯范围 | 需求→设计→编码→测试 | PRD→功能→API/DB/UI/业务流程 | 我们扩展了追溯维度 |
| 业务流程关联 | 无此概念 | 新增6.5追溯表 | 我们的创新点 |

### 9.5 人类编程规则 vs AI编程规则

**不适合AI的人类特有规则**：

| 人类规则 | 为什么不适合AI | 我们的Checklist是否检查 |
|---------|---------------|------------------------|
| 8/80规则（工作包8-80小时） | AI没有工时概念，执行速度不同 | ❌ 不检查（正确） |
| 两周原则（工期10-14天） | AI执行速度远超人类 | ❌ 不检查（正确） |
| 人力分配 | AI执行任务，不存在人力概念 | ❌ 不检查（正确） |
| 人力成本估算 | 成本模型不同（token成本） | ❌ 不检查（正确） |
| 沟通成本 | AI不存在沟通成本 | ❌ 不检查（正确） |

**AI适配检查项**（W20/W21）：

| 检查项 | AI适配规则 | 说明 |
|--------|-----------|------|
| W20：功能描述完整 | 不依赖外部上下文 | AI需要完整信息，无法"问同事" |
| W21：功能可独立执行 | 可分配给子代理并行执行 | 支持AI并行处理 |
