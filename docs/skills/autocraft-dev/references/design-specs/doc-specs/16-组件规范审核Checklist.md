# 组件规范文档审核 Checklist

**版本：** v1.4  
**日期：** 2026-04-13  
**用途：** AI审核节点检查指导  
**整理人：** AutoCraft 团队

---

## 一、核心原则（审核底线）

### 🚫 三大禁止行为

| 编号 | 禁止行为 | 验证方法 | 违规处理 |
|------|----------|----------|----------|
| **B01** | 禁止命名规范缺失 | 必须定义文件命名、组件命名、Props命名规则 | ❌ 严重问题，必须修复 |
| **B02** | 禁止Props无类型定义 | 每个Props必须有类型说明 | ❌ 严重问题，必须修复 |
| **B03** | 禁止组件无使用示例 | 每个组件必须有代码示例 | ❌ 严重问题，必须修复 |

### ✅ 三大必须行为

| 编号 | 必须行为 | 验证方法 |
|------|----------|----------|
| **M01** | 必须有目录结构 | 定义组件分类目录（layout/ui/business等） |
| **M02** | 必须有Props表格 | 包含属性、类型、默认值、说明四列 |
| **M03** | 必须有Events定义 | 每个组件有Events表格（如有事件） |

## 二、审核前准备检查

### 2.1 输入材料完整性

| ID | 检查项 | 检测规则 | 不合格处理 |
|----|--------|---------|-----------|
| **P01** | 技术方案文档存在 | 待审核技术方案文件路径存在 | ⚠️ 续审但标记缺失（需前端框架信息） |
| **P02** | UI设计文档存在 | 待审核UI设计文件路径存在 | ⚠️ 续审但标记缺失（可交叉验证组件） |
| **P03** | 组件规范文档存在 | 待审核组件规范文件路径存在 | 续审但标记缺失 |

### 2.2 准备合格标准

```
✅ P03 通过 → 开始审核
⚠️ P01不通过 → 续审但标记"无技术方案"（无法确认前端框架）
⚠️ P02不通过 → 续审但标记"无UI设计"（无法交叉验证组件）
```

> 📋 **说明**：技术方案提供前端框架选择，UI设计提供组件设计参考。

## 三、阻断项检查 [Error] - 必须通过

### 3.1 文档结构类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E01** | 命名规范章节存在 | 第1节存在命名规范内容 | 无命名规范章节 |
| **E02** | 目录结构章节存在 | 第2节存在目录结构内容 | 无目录结构章节 |
| **E03** | 通用组件章节存在 | 第3节存在通用组件内容 | 无通用组件章节 |

### 3.2 命名规范类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E04** | 文件命名规范存在 | 1.1节有文件命名规则，且规则为PascalCase（如`QuestionCard.vue`） | 无文件命名规范或格式错误 |
| **E05** | 组件命名规范存在 | 1.2节有组件命名规则，且规则为PascalCase（如`<QuestionCard />`） | 无组件命名规范或格式错误 |
| **E06** | Props命名规范存在 | 1.3节有Props命名规则，且规则为camelCase（如`userName`） | 无Props命名规范或格式错误 |

### 3.3 组件定义类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E07** | 组件数量≥1 | 至少有1个组件定义 | 无组件定义 |
| **E08** | Props表格存在 | 组件详情有Props表格 | 某组件无Props表格 |
| **E09** | Props表格有三列 | Props表格有类型、默认值、说明三列 | 缺少默认值列 |
| **E10** | 使用示例存在 | 组件详情有代码示例（如`<BaseButton>`） | 某组件无使用示例 |

### 3.4 组件接口类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E11** | Events表格存在（含事件、参数、说明） | 有事件的组件有Events表格，包含事件、参数、说明三列 | 某组件有事件但无Events表格或缺少列 |
| **E12** | 类型定义存在 | 复杂Props有TypeScript interface定义 | Question prop无interface定义 |

### 3.5 内容一致性类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E13** | 组件Props与UI设计一致 | 组件Props定义与UI设计文档的组件定义一致，无理解偏差或简化 | UI设计定义"题目卡片有删除按钮"，组件无onDelete事件（简化） |
| **E14** | 文档内部一致性 | 组件规范文档内同一概念在不同位置（组件列表、Props定义、事件定义、插槽定义等）的描述必须保持一致，不得出现矛盾或冲突。检查维度：(1)Props一致：组件列表与Props详情中的属性定义一致；(2)事件一致：组件列表的事件与事件详情定义一致；(3)命名一致：同一组件在不同位置的命名一致 | 组件列表Props有"size"，Props详情无"size"（Props矛盾）；组件列表命名"QuestionCard"，详情写"QuizCard"（命名矛盾） |
| **E14** | 无虚增组件 | 组件列表中的组件都能追溯到UI设计文档，无凭空添加 | 组件列表有`ExportButton`，但UI设计无此组件（虚增） |

### 3.6 组件使用场景验证类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E31** | Props/Events覆盖实际场景 | 核心组件Props/Events覆盖实际业务场景：TaskNodeProgress组件Props包含current_node等字段、组件Events支持API响应数据绑定 | TaskNodeProgress组件缺少current_node Props，无法显示当前节点进度 |

### 3.7 Slot作用域与防御性编码类

| ID | 检查项 | 检测规则 | 不通过示例 |
|----|--------|---------|-----------|
| **E32** | Slot作用域变量与UI库API一致 | 组件Slots表格中的作用域参数必须与实际使用的UI库（如Element Plus）的API一致，不得自行定义不同的变量名或省略关键作用域变量 | 组件定义slot作用域为`{ node: TreeNode }`，但Element Plus的el-tree实际提供`{ node, data }`（node是内部节点对象，data是业务数据） |
| **E33** | 防御性编码规范存在 | 组件规范必须包含防御性编码要求，至少覆盖：Slot空值保护（row/data可能undefined）、Props空值保护、API响应空值保护 | 组件规范无防御性编码要求，子代理生成的代码直接访问slot作用域变量属性导致运行时崩溃 |

**E32验证步骤**：
```
步骤1: 提取所有组件的Slots定义 → 获取作用域参数列表
步骤2: 对照UI库官方文档 → 确认作用域变量名和类型与UI库一致
步骤3: 检查变量名语义 → 确认没有将UI库内部变量名用于业务数据（如el-tree的node vs data）
步骤4: 输出Slot一致性报告 → 标记不一致的定义
```

**E33验证步骤**：
```
步骤1: 检查最佳实践章节 → 确认有"防御性编码"或"空值保护"相关内容
步骤2: 检查覆盖范围 → 确认至少包含Slot空值保护、Props空值保护
步骤3: 检查代码示例 → 确认有空值保护的正反例代码
步骤4: 输出防御性编码检查报告 → 标记缺失项
```

**E31模拟验证步骤**：
```
步骤1: 检查TaskNodeProgress组件Props → 确认包含current_node、total_nodes、status等字段
步骤2: 检查组件Events定义 → 确认有支持API响应数据绑定的事件（如@update、@complete）
步骤3: 对照UI设计使用场景 → 确认Props覆盖UI设计中的所有使用场景
步骤4: 对照API设计响应数据 → 确认组件能接收API返回的数据结构
步骤5: 输出Props/Events覆盖报告 → 标记缺失的Props或Events
```

### 3.7 阻断项合格标准

```
✅ E01-E33 全部通过 → 继续评分判断
❌ 任一项不通过 → 标记严重问题，必须修复
```

## 五、审核结果判断

### 5.1 阻断项优先判断

```
第一步：检查阻断项 E01-E12
    ├─ 有不通过项 → 无论评分多少，必须修复
    └─ 全部通过 → 进入第二步评分判断
```

### 5.2 评分判断规则

| 评分 | 警告项状态 | 处理 |
|------|-----------|------|
| ≥90分 | 全部通过 | ✅ 定稿 |
| ≥90分 | 有中等项 | ⚠️ 可定稿，建议修复 |
| 80-89分 | 全部通过 | ⚠️ 小修后定稿 |
| 80-89分 | 有中等项 | ⚠️ 修复后定稿 |
| 70-79分 | - | ⚠️ 修复后重新审核 |
| <70分 | - | ❌ 重新生成 |

### 5.3 问题分级与修复优先级

| 问题级别 | 来源 | 修复优先级 |
|---------|------|-----------|
| **严重** | 阻断项不通过（E01-E12） | 必须修复 |
| **中等** | 警告项W03-W05/W07-W08/W10-W17不通过 | 建议修复 |
| **轻微** | 警告项W01/W02/W06/W09不通过 | 可选修复 |

## 八、与其他文档的分工约定

### 8.1 与系统功能设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **系统功能设计** | 业务功能 + 业务规则 | F-001创建题目（题目内容不能为空） |
| **组件规范** | 组件接口 + Props/Events | QuestionCard组件（question prop） |

### 8.2 与UI设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **UI设计** | 页面结构 + 组件使用 | 题目列表页使用QuestionCard组件 |
| **组件规范** | 组件Props + Events + Slots | QuestionCard: props.question, @delete事件 |

### 8.3 与API设计分工

| 文档 | 定义内容 | 示例 |
|------|----------|------|
| **API设计** | 接口路径 + 请求参数 | GET /api/questions, 返回Question[] |
| **组件规范** | 组件Props类型定义 | interface Question { id, content, ... } |

### 8.4 审核时的注意事项

- 组件规范文档定义**组件接口规范**
- 不应包含业务规则细节（由系统功能设计定义）
- 不应包含API接口定义（由API设计定义）
- Props表格必须包含类型、默认值、说明三列
- 每个组件必须有使用示例

## 九、来源追溯（业界标准对照）

### 9.1 业界标准对照

| 我们的检查项 | 业界标准 | 核心要求 | 来源 |
|-------------|---------|---------|------|
| 文件命名规范 | **Vue风格指南** | PascalCase单文件组件 | Vue官方风格指南 |
| Props定义规范 | **组件API设计** | 类型、默认值、验证 | React/Vue组件设计原则 |
| 目录结构分层 | **组件分层架构** | 原子组件/分子组件/有机体 | Atomic Design |
| 组件设计原则 | **组件设计最佳实践** | 单一职责、可组合性 | Clean Components |

### 9.2 Vue官方风格指南核心要求

| 规则 | 说明 | 对应检查项 |
|------|------|-----------|
| **组件名为多个单词** | 避免与HTML元素冲突 | E05 |
| **组件文件名PascalCase** | 或kebab-case | E04 |
| **Props定义尽量详细** | 至少指定类型 | E09 |
| **v-for设置key** | 保持组件状态 | W13（最佳实践） |

### 9.3 Atomic Design核心概念

| 层级 | 说明 | 对应检查项 |
|------|------|-----------|
| **Atoms（原子）** | 基础UI组件 | W05（ui目录） |
| **Molecules（分子）** | 组合组件 | W05（business目录） |
| **Organisms（有机体）** | 复杂组件 | W05（layout目录） |
