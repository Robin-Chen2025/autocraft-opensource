## 一、文档概述

组件规范文档定义前端组件的命名、结构和Props规范，确保代码一致性。

**前置依赖**：
- UI设计文档（必须）：提供页面组件设计、组件命名参考
- 技术方案文档（必须）：提供前端框架选择
- 系统功能设计文档（辅助）：提供功能ID追溯

> ⚠️ **框架约定**：本规范示例基于 **Vue 3 + TypeScript** 编写（`.vue` 文件、`v-model`、`@click` 等）。实际开发时，组件规范应与技术方案文档选定的前端框架保持一致。如使用 React/Svelte 等其他框架，请适配相应的语法和命名规范。

**核心作用**：
- 统一组件命名
- 定义组件接口
- 提高代码可维护性

## 三、分阶段设计策略

### 3.1 组件规范分阶段原则

组件规范采用**一次性框架 + 业务组件按功能分阶段**策略：

| 组件类型 | 分阶段策略 | 说明 |
|---------|-----------|------|
| 命名规范 | ✅ 一次性设计 | 不分阶段，基础框架 |
| 目录结构 | ✅ 一次性设计 | 不分阶段，基础框架 |
| 通用组件 | ⚠️ P0开发时识别 | 在P0开发过程中识别公共组件 |
| 业务组件 | ⚠️ 按功能分阶段 | 与UI设计P0/P1/P2对齐 |

### 3.2 各优先级设计深度

| 优先级 | 组件清单 | 组件详情 |
|--------|---------|---------|
| **P0** | ✅ 必须列出 | ✅ 详细设计（完整Props、事件、样式规范） |
| **P1** | ✅ 必须列出 | ❌ 占位符（仅组件名称 + 用途说明） |
| **P2** | ✅ 必须列出 | ❌ 占位符（仅组件名称 + 用途说明） |

### 3.3 为什么命名规范一次性设计？

| 原因 | 说明 |
|------|------|
| **统一编码风格** | 命名规范是公共层，跨功能复用 |
| **避免混乱** | 如果每个迭代命名规范不同，代码风格不一致 |
| **降低学习成本** | 新加入的开发人员只需学习一套规范 |

### 3.4 通用组件的识别时机

| 时机 | 识别方式 |
|------|---------|
| P0开发前 | 根据UI设计文档，预先识别可能复用的组件 |
| P0开发中 | 发现重复代码时，抽象为通用组件 |
| P0开发后 | 代码审查时，补充通用组件清单 |

### 3.5 一个文档 vs 多个文档

| 方案 | 推荐 | 说明 |
|------|------|------|
| **一个文档** | ✅ 推荐 | 命名规范统一、组件清单完整、维护成本低 |
| **多个文档** | ❌ 不推荐 | 每个迭代单独文档会导致命名规范不一致 |

### 3.6 设计指导

参见总纲文档《设计阶段文档规范（总纲）》第10.9节"分阶段设计策略汇总"。

## 3. 通用组件

### 3.1 BaseButton

**用途**：通用按钮组件

**Props**：

| 属性 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| type | `'default' \| 'primary' \| 'success' \| 'warning' \| 'danger'` | `'default'` | 按钮类型 |
| size | `'small' \| 'medium' \| 'large'` | `'medium'` | 按钮大小 |
| loading | `boolean` | `false` | 加载状态 |
| disabled | `boolean` | `false` | 禁用状态 |
| icon | `string` | `''` | 图标名称 |

**Events**：

| 事件 | 参数 | 说明 |
|------|------|------|
| click | `event: MouseEvent` | 点击事件 |

**Slots**：

| 插槽 | 作用域参数 | 说明 |
|------|-----------|------|
| default | - | 按钮内容 |
| icon | - | 图标插槽 |

> ⚠️ **Slot作用域参数要求**：如果插槽提供作用域参数，必须与实际使用的UI库API一致。例如Element Plus的el-tree默认插槽提供`{ node, data }`（`node`是内部节点对象，`data`是业务数据），不能自行定义不同的作用域变量名。

**使用示例**：

```vue
<BaseButton type="primary" @click="handleSubmit">
  提交
</BaseButton>

<BaseButton type="danger" loading>
  删除中...
</BaseButton>
```

## 5. 最佳实践

### 5.1 组件设计原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 一个组件只做一件事 |
| 可组合性 | 小组件组合成大组件 |
| 可配置性 | 通过Props配置行为 |
| 可扩展性 | 通过Slots扩展内容 |

### 5.2 Props设计规范

| 规范 | 示例 |
|------|------|
| 必填Props尽量少 | 只保留核心属性必填 |
| 提供默认值 | 非必填Props提供合理默认值 |
| 类型明确 | 使用TypeScript定义类型 |
| 命名语义化 | `isLoading` 优于 `loading` |

### 5.3 防御性编码规范

> ⚠️ **必须遵守**：UI库的slot作用域变量在某些渲染阶段可能为undefined，必须添加空值保护。

| 规则 | 说明 | 示例 |
|------|------|------|
| **Slot空值保护** | el-table/el-tree等组件的slot作用域变量（`row`/`data`等）在首次渲染时可能为undefined，访问属性前必须做空值检查 | `v-if="row"` / `v-if="data"` |
| **Props空值保护** | 复杂Props的对象属性访问前应做可选链或空值检查 | `item?.name` / `v-if="item"` |
| **API响应空值保护** | 组件渲染API数据时，应考虑字段缺失的情况 | `data?.list ?? []` |

```vue
<!-- ✅ 正确：空值保护 -->
<template #default="{ row }">
  <div v-if="row" class="name">{{ row.name }}</div>
</template>

<!-- ❌ 错误：直接访问 -->
<template #default="{ row }">
  <div class="name">{{ row.name }}</div>  <!-- row可能undefined，崩溃！ -->
</template>
```

### 5.4 组件通信方式

| 场景 | 方式 |
|------|------|
| 父→子 | Props |
| 子→父 | Events |
| 兄弟组件 | Pinia Store |
| 跨层级 | Provide/Inject |

## 六、常见问题

### Q1: Base组件和业务组件的区别？

**A**: 
- **Base组件（通用）**：与业务无关，可在多个项目复用，如按钮、输入框
- **业务组件**：与特定业务相关，如题目卡片、知识点标签

### Q2: 是否需要为每个Props写注释？

**A**: 是的。Props是组件的API，需要文档化，方便其他开发者使用。
