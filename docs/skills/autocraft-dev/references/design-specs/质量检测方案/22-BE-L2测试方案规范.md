## 三、测试场景清单格式（核心）

### 3.1 三要素强制要求

每个测试用例**必须**包含：

| 要素 | 说明 | 要求 |
|------|------|------|
| **预设数据** | 插入已知状态的测试数据 | 明确写出每条数据的字段值 |
| **操作** | 调用真实API | 使用测试服务器 |
| **预期结果** | 精确的返回值+数据库状态 | 不能只写"返回200" |

### 3.2 测试用例表格格式

```markdown
| 用例ID | 预设数据 | 操作 | 预期结果 |
|--------|---------|------|---------|
| TC-Q-001 | 插入3道题（数学completed 1道，数学pending 1道，语文pending 1道） | GET /api/questions?subject=数学&status=pending | HTTP 200, data长度=1, data[0].id=3, data[0].subject="数学", pagination.total=1 |
```

### 3.3 AI服务测试用例

| 用例ID | 预设数据 | 操作 | 预期结果 |
|--------|---------|------|---------|
| TC-CHAT-001 | 新建会话 | POST /api/chat/messages {message: "1+1=?"} | HTTP 200, reply非空, reply长度>10, 不含"error" |
| TC-CHAT-002 | 新建会话 | POST /api/chat/messages {message: "今天天气怎么样"} | HTTP 200, reply包含引导回学习话题的内容 |
| TC-RECOG-001 | 上传测试图片 | POST /api/questions/recognize {file_id: "xxx"} | HTTP 202, task_id非空; 最终status=completed, questions表有新记录且status="completed" |

**分类规则**：
- **固定格式返回**（API schema、error_code）→ 验证具体值
- **AI生成内容**（对话文本）→ 只验证结构+非空+合理性
- **AI对话异常**（非学习问题）→ 预设预期行为

## 四、测试环境

### 4.1 环境要求

| 组件 | 要求 | 说明 |
|------|------|------|
| 数据库 | 真实SQLite数据库 | 禁止mock数据库 |
| 后端服务 | 真实FastAPI测试服务器 | 使用TestClient或独立进程 |
| AI服务 | 真实调用 | 超时跟生产配置走 |
| 数据 | 预设测试数据 | 禁止自己编造 |

### 4.2 AI服务超时策略

超时阈值从项目 `config.py` 或环境变量读取，**不在测试方案中写死**。

**超时=SKIP，不阻塞其他测试，不计入失败率。**

## 五、执行流程

### 5.1 三步闭环

```
BUILD-TEST: 编写测试代码
    ├─ 严格按测试方案文档的预设数据和预期结果
    ├─ 使用真实数据库
    ├─ 不运行测试
    └─ 产出：测试代码文件 + 断言报告

TEST-RUN: 运行测试
    ├─ 启动测试服务器 + 真实数据库
    ├─ 如实记录结果
    ├─ 不修改任何代码
    └─ 产出：测试运行报告（通过/失败/断言明细+失败分类）

FIX-CODE: 修复bug
    ├─ 根据TEST-RUN发现的code_issue修复生产代码
    ├─ 不修改测试代码
    ├─ 修完后重新跑TEST-RUN
    └─ 产出：代码修改 + 重跑结果
```

### 5.2 失败分类

| 分类 | 判断标准 | 处理 |
|------|---------|------|
| code_issue | 测试逻辑正确，程序返回错误结果 | FIX-CODE |
| test_issue | 测试代码有误 | 新BUILD-TEST修正 |
| env_issue | 环境配置问题 | 解决环境 |

## 六、验收标准

| 指标 | 要求 | 说明 |
|------|------|------|
| 预设数据完整性 | 每个用例必须有 | 不能"插入若干数据" |
| 预期结果精确性 | 必须有数据内容验证 | 不能只有状态码 |
| pytest通过率 | 100%（SKIP不计入失败）| - |
| 数据落库验证 | 写操作必须验证数据库状态 | - |
| 断言报告 | 每个用例的断言内容必须输出 | 暴露测试质量 |
| FIX-CODE验收 | 断言数量不减少，严格度不降低 | 防止降级 |

## 七、测试数据准备

### 7.1 数据隔离策略

| 场景 | 是否清空 | 处理方式 |
|------|---------|---------|
| L2启动时 | ✅ 一次性 | 初始化脚本 |
| 不同测试场景 | ❌ 不清空 | 使用不同数据ID |
| 同一场景多次执行 | pytest自带隔离 | fixture自动隔离 |

### 7.2 数据初始化脚本

```python
# tests/fixtures/init_test_db.py
import sqlite3

def init_test_db(db_path):
    conn = sqlite3.connect(db_path)
    # 清空+插入预设数据
    clear_all_tables(conn)
    insert_preset_data(conn)  # 严格按测试方案文档
    conn.commit()
    conn.close()
```
