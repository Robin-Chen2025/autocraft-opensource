# 项目整体测试方案

**版本：** v2.0  
**日期：** 2026-05-13  
**作者：** AutoCraft 团队  
**核心原则：测试的目的是发现代码bug，不是让测试通过**

---

## 一、概述

本文档定义 AutoCraft 项目 V2 测试体系，基于 V1 实践经验重构。

### V1 问题总结

| 问题 | 根因 | V2 解决 |
|------|------|---------|
| L1发现bug为0 | mock太多脱离真实 | L1仅在3条规则触发时编写 |
| L2前端测了个寂寞 | vitest+jsdom跟真实浏览器差异大 | 取消前端L2，合并到L3 |
| 测试断言太弱 | 只验证状态码不验证数据 | 强制预设数据+精确验证输出 |
| AI写测试又跑测试 | 自己查自己 | 严格三角色分离 |
| bug修复无闭环 | 问题只记录在报告里 | 三步闭环（BUILD-TEST→TEST-RUN→FIX-CODE） |
| L2测试设计缺陷 | 测试内联组件而非真实组件 | 后端L2用真实DB，前端L2取消 |

### V2 核心变化

| 维度 | V1 | V2 |
|------|----|-----|
| 测试层级 | L1+L2+L3 三层 | L1(条件触发)+L2(仅后端)+L3(S/M/L梯度) |
| L1策略 | 所有模块必写 | 仅3条规则触发时编写 |
| 前端L2 | vitest+jsdom | ❌取消，合并到L3 |
| L2后端 | mock集成 | 真实API+真实数据库 |
| L3前端 | E2E | 梯度设计：S单功能→M单模块→L跨模块 |
| AI服务 | mock | 真实调用，超时跟生产配置走 |
| 断言要求 | 通过率100% | 预设数据+精确验证输出 |

## 三、L1测试策略（条件触发）

### 3.1 L1触发规则

L1测试**仅在满足以下任一条件时编写**，否则不写L1：

#### 规则1：外部服务异常处理

| 条件 | 判断标准 | L1必须覆盖的场景 |
|------|---------|----------------|
| 调用外部AI/HTTP服务 | 代码中有agent调用、requests、httpx | ①超时 ②返回格式异常 ③返回错误码 ④返回空值 |

**触发关键词**：`agent`、`requests`、`httpx`、`openai`、`API_KEY`

#### 规则2：多状态流转

| 条件 | 判断标准 | L1必须覆盖的场景 |
|------|---------|-----------------|
| 实体有3个以上状态 | 代码中有status字段且有≥3种值 | ①正常流转 ②非法跳转 ③回退 ④并发冲突 |

**触发关键词**：`status`、`state`、`transition`

#### 规则3：数据格式解析

| 条件 | 判断标准 | L1必须覆盖的场景 |
|------|---------|-----------------|
| 解析外部返回的JSON | 代码中有json.loads+外部数据源 | ①字段缺失 ②类型错误 ③嵌套结构变化 |

**触发关键词**：`json.loads`、`parse`+外部调用

#### 反向规则：不需要L1

- 纯CRUD（增删改查）→ L2
- 纯前端UI渲染 → L3
- 数据库迁移脚本 → L2验证
- 简单查询+筛选 → L2

### 3.2 L1执行流程

L1采用**BUILD+RUN合一**模式——写测试的子代理同时跑测试，因为L1测试简单、不依赖真实后端服务：

```
L1任务执行：
    │
    ├─ 扫描代码，按3条规则判断是否需要L1
    │     ├─ 命中任一规则 → 编写L1测试
    │     └─ 三条都不命中 → 跳过L1
    │
    ├─ 编写L1测试代码（mock外部依赖，覆盖表中场景）
    │
    ├─ 运行L1测试
    │
    └─ 输出结果（通过/失败+失败详情）
```

### 3.3 L1验收标准

| 指标 | 要求 |
|------|------|
| 触发规则覆盖 | 每条命中规则的所有场景都有测试 |
| pytest通过率 | 100% |
| mock范围 | 仅mock外部服务，不mock内部逻辑 |

## 四、L2测试策略（后端接口测试）

### 4.1 核心原则

> **L2测试必须使用真实数据库，预设测试数据，精确验证输出。禁止mock后端服务和数据库。**

每个L2测试用例必须包含三要素：

| 要素 | 说明 | 要求 |
|------|------|------|
| **预设数据** | 插入已知状态的测试数据 | 明确写出每条数据的字段值 |
| **操作** | 调用真实API | 使用测试服务器 |
| **预期结果** | 精确的返回值+数据库状态 | 不能只写"返回200"，必须验证数据内容 |

### 4.2 测试用例格式

每个测试用例必须按以下格式写在测试方案文档中：

```markdown
| 用例ID | 预设数据 | 操作 | 预期结果 |
|--------|---------|------|---------|
| TC-Q-001 | 插入3道题（数学completed 1道，数学pending 1道，语文pending 1道） | GET /api/questions?subject=数学&status=pending | HTTP 200, data长度=1, data[0].id=3, data[0].subject="数学", pagination.total=1 |
```

**AI写测试代码时必须严格照搬文档中的预设数据和预期结果，禁止自己编造。**

### 4.3 AI服务处理

| 场景 | 处理方式 | 验证标准 |
|------|---------|---------|
| AI对话正向 | 真实调用meow agent | reply非空 + 长度>10 + 不含错误标识 |
| AI对话异常（非学习问题） | 真实调用 | 验证拒绝或引导行为（预设预期） |
| 图片识别 | 真实调用dt-ai agent | 验证数据落库完整性（status=completed、字段非空） |
| 固定格式返回 | 真实调用 | 验证schema（字段存在、类型正确、数组非空） |

**超时处理**：超时阈值跟生产配置走（从config.py/环境变量读取），超时标记为SKIP而非FAIL。

**分类**：
- **固定格式内容**（如API返回schema）→ 验证具体值
- **AI生成内容**（如对话文本）→ 只验证结构+非空+合理性

### 4.4 L2执行流程

```
L2（后端）执行：
    │
    ├─ BUILD-TEST: 编写测试代码
    │     ├─ 严格按照测试方案文档的预设数据和预期结果
    │     ├─ 启动测试服务器 + 真实数据库
    │     ├─ 不运行测试（不运行！）
    │     └─ 产出：测试代码文件 + 断言报告
    │
    ├─ TEST-RUN: 运行测试
    │     ├─ 启动测试服务器 + 真实数据库
    │     ├─ 运行所有测试，如实记录结果
    │     ├─ 不修改任何代码
    │     └─ 产出：测试运行报告（通过/失败/断言明细）
    │
    └─ FIX-CODE: 修复bug
          ├─ 根据TEST-RUN发现的code_issue修复生产代码
          ├─ 不修改测试代码
          ├─ 修完后重新跑TEST-RUN验证
          └─ 产出：代码修改 + 重新运行结果
```

### 4.5 L2验收标准

| 指标 | 要求 |
|------|------|
| 预设数据完整性 | 每个用例必须有明确预设数据 |
| 预期结果精确性 | 不能只有状态码，必须有数据内容验证 |
| pytest通过率 | 100%（SKIP不计入失败） |
| 数据落库验证 | 写操作必须验证数据库状态 |
| 断言报告 | 每个用例的断言内容必须输出 |

## 五、L3测试策略（前端E2E测试）

### 5.1 核心原则

> **前端L2取消，所有前端测试在真实浏览器中完成（Playwright）。取消vitest+jsdom前端L2。**

### 5.2 L3梯度设计

| 粒度 | 范围 | 示例 | 发现bug预期 |
|------|------|------|------------|
| **L3-S（单功能）** | 一个页面内一个功能点 | 打开列表页→筛选→验证结果数量变化 | 基础交互bug |
| **L3-M（单模块）** | 一个模块的完整流程 | 上传图片→识别→确认→入库 | 流程衔接bug |
| **L3-L（跨模块）** | 多模块串联的用户场景 | 识别图片→入库→分析→查看结果→AI讲解 | 集成bug |

### 5.3 L3测试用例格式

与L2相同，必须包含预设数据+操作+精确预期结果：

```markdown
| 用例ID | 粒度 | 预设数据 | 操作 | 预期结果 |
|--------|------|---------|------|---------|
| TC-E2E-Q-001 | S | 数据库有5道数学题 | 打开题目列表→筛选"数学" | 列表显示5条，筛选标签显示"数学" |
```

### 5.4 L3执行流程

L3与L2共享三步闭环流程（BUILD-TEST→TEST-RUN→FIX-CODE），区别在于工具和粒度：

```
L3执行：
    │
    ├─ BUILD-TEST: 编写Playwright测试代码
    │     ├─ 严格按照测试方案文档
    │     ├─ 不运行测试
    │     └─ 产出：测试代码文件 + 断言报告
    │
    ├─ TEST-RUN: 运行Playwright测试
    │     ├─ 启动真实浏览器
    │     ├─ 如实记录结果
    │     └─ 产出：测试运行报告
    │
    └─ FIX-CODE: 修复bug
          └─ 同L2
```

### 5.5 L3验收标准

| 指标 | 要求 |
|------|------|
| Playwright通过率 | 100% |
| S级用例覆盖 | 每个页面至少1个S级用例 |
| M级用例覆盖 | 每个模块至少1个M级用例 |
| L级用例覆盖 | 至少1个L级用例（跨模块串联） |
| 断言精确性 | 必须验证数据内容，不能只验证页面能渲染 |
| AI服务 | 真实调用，超时=SKIP |

## 六、测试方案审核机制

### 6.1 审核流程

```
AI生成测试方案 → AI标准化审核 → 用户审核用例和场景 → 定稿
```

| 步骤 | 谁做 | 审核内容 |
|------|------|---------|
| ① 生成测试方案 | AI子代理 | 从设计文档提取测试场景，编写预设数据和预期结果 |
| ② 标准化审核 | AI审核代理 | 检查格式规范、断言完整性、场景覆盖率 |
| ③ 用户审核 | 用户 | 审核用例合理性、场景是否充分、预期结果是否正确 |

### 6.2 AI标准化审核Checklist

| 检查项 | 要求 | 不通过处理 |
|--------|------|-----------|
| 每个用例有预设数据 | 明确的字段值，不是"插入若干数据" | 打回补充 |
| 每个用例有精确预期结果 | 不是"返回成功"，而是具体字段值 | 打回补充 |
| AI服务场景有验证标准 | 正向+异常都覆盖 | 打回补充 |
| 场景覆盖完整 | 每个功能ID至少1个正向+1个异常 | 打回补充 |
| L1触发判断正确 | 按3条规则逐条检查 | 修正 |

### 6.3 测试代码质量保障

测试代码生成后，输出**断言报告**，暴露每个用例的实际断言内容：

```
TC-Q-001: 3 assertions ✅
  - assert response.status_code == 200
  - assert len(data) == 1              ← 精确
  - assert data[0]["subject"] == "数学"  ← 精确

TC-Q-002: 1 assertion ✅
  - assert response.status_code == 200   ← ⚠️ 太弱，只有状态码
```

**用户只需扫描断言报告，1分钟判断测试质量。**

## 七、角色分离与执行规范

### 7.1 三角色分离

| 角色 | 职责 | 红线 | 产出物 |
|------|------|------|--------|
| **BUILD-TEST** | 编写测试代码 | 不得运行测试、不得修改程序代码 | 测试代码文件 + 断言报告 |
| **TEST-RUN** | 运行测试、记录结果 | 不得修改任何代码 | 测试运行报告 |
| **FIX-CODE** | 修复程序代码bug | 不得修改测试代码 | 代码修改 + 重跑结果 |

### 7.2 各层级执行方式

| 层级 | 执行方式 | 理由 |
|------|---------|------|
| L1 | BUILD+RUN合一 | 简单异常场景，不需分离 |
| L2 | BUILD-TEST→TEST-RUN→FIX-CODE | 需要验证断言质量+bug闭环 |
| L3 | BUILD-TEST→TEST-RUN→FIX-CODE | 完整闭环 |

### 7.3 测试失败分类

| 分类 | 说明 | 处理方式 |
|------|------|---------|
| code_issue | 程序代码bug | 创建FIX-CODE任务 |
| test_issue | 测试代码问题 | 创建新的BUILD-TEST任务修正 |
| env_issue | 环境配置问题 | 解决环境配置 |

### 7.4 FIX-CODE验收标准

- 修完后重新跑TEST-RUN，原来失败的用例现在通过
- 断言数量不少于原始
- 断言严格度不低于测试方案文档要求

## 八、测试执行顺序与依赖

### 8.1 执行顺序

```
项目启动
    │
    ▼
BE-DEV（后端开发）
    │ L1：条件触发（3条规则）
```

| L2（BE-DEV后） | BE-DEV | completed | 后端API可用 |
