## 一、方案选择

### 1.1 三个方案对比

| 方案 | 内容 | 成本 | 推荐度 |
|------|------|------|--------|
| **方案A** | 增强验证子代理提示词 | 4h | ⭐⭐⭐⭐⭐ 推荐 |
| **方案B** | 自建Gitee AI审查服务 | 40h | ⭐⭐ 不推荐 |

| 原因 | 说明 |
|------|------|
| 成本最低 | 只需更新提示词，不需要开发新系统 |
| 无缝集成 | 验证子代理已在节点4，直接增强即可 |

| AI审查有价值 | AI能发现静态分析和测试无法发现的问题 |

## 二、职责边界原则

### 2.1 核心原则

> **Guardian负责程序化检查（阈值阻断）**
> **验证子代理负责AI判断检查（业务语义）**

### 2.2 职责划分表

| 检查类型 | Guardian负责 | 验证子代理负责 |
|----------|--------------|----------------|
| 程序化阈值检查 | ✅ | ❌ |
| 业务语义判断 | ❌ | ✅ |
| 逻辑完整性判断 | ❌ | ✅ |
| 验收标准判断 | ❌ | ✅ |

---

## 三、验证子代理核心职责

### 3.1 必须AI独立完成的职责

| 职责 | 说明 | 不能简化/外包 |
|------|------|---------------|
| **业务逻辑验证** | 对照业务流程文档，判断代码是否符合业务规则 | ✅ 必须AI独立判断 |
| **验收标准验证** | 对照验收标准，判断是否满足所有条件 | ✅ 必须AI独立判断 |
| **边界情况验证** | 判断是否有遗漏的边界处理 | ✅ 必须AI独立判断 |
| **结论判定** | 综合1-3，给出"通过/不通过"结论 | ✅ 必须AI独立判断 |
| **报告生成** | 生成完整验证报告 | ✅ 必须AI独立生成 |

### 3.2 不重复Guardian

| 检查项 | 执行位置 | 验证子代理不检查 |
|--------|----------|------------------|
| pytest通过/失败 | ✅ pytest闭环内部（节点2）| ❌ 不重复 |
| 覆盖率≥80% | ✅ pytest闭环内部（节点2）| ❌ 不重复 |
| Ruff lint error | ✅ pytest闭环内部（节点2修复阶段）| ❌ 不重复 |
| bandit高危漏洞 | ✅ **L2测试阶段**（BE-L2最后任务单）| ❌ 不重复 |
| radon圈复杂度>20 | ✅ pytest闭环内部（节点2修复阶段）| ❌ 不重复 |

> **说明**：以上检查在pytest闭环内部或L2测试阶段执行，产出quality_check字段，Guardian节点3只验证结果

## 四、辅助工具/checklist角色

### 4.1 辅助工具定位

**角色**：辅助提供信息，不替代判断

| 辅助类型 | 提供什么 | 验证子代理如何使用 |
|----------|----------|-------------------|
| 工具检测报告 | Ruff/mypy/bandit输出 | AI读取报告，纳入验证报告 |
| Checklist清单 | 必检项提醒 | AI按清单逐项检查 |
| 检测脚本输出 | 覆盖率、重复率数据 | AI读取数据，判断是否达标 |

### 4.2 关键约束

- 工具提供数据，AI判断数据是否达标
- Checklist提供提醒，AI逐项检查并判断
- 工具/checklist不替代AI给出结论

## 六、AI代码审查Checklist

### 6.1 逻辑bug检测（L1-L5）

| ID | 检查项 | 说明 |
|----|--------|------|
| L1 | 是否遗漏条件判断分支？（业务逻辑层面）| Guardian只检查语法，不检查业务逻辑 |
| L2 | 循环终止条件是否符合业务预期？| Guardian检查死循环语法，不检查业务终止条件 |
| L3 | 变量是否在所有业务路径上正确赋值？| Guardian检查未定义变量（语法），不检查业务路径赋值 |
| L4 | 返回值是否符合业务预期？| Guardian检查返回类型（语法），不检查返回值业务含义 |
| L5 | 异步调用顺序是否符合业务流程？| Guardian检查语法，不检查业务顺序 |

### 6.2 边界遗漏检测（B1-B5）

| ID | 检查项 | 说明 |
|----|--------|------|
| B1 | 是否遗漏空值处理（业务关键数据）？| Guardian检查语法，不判断业务关键数据 |
| B2 | 是否遗漏异常处理（业务关键操作）？| Guardian检查语法，不判断哪些是业务关键操作 |
| B3 | 是否遗漏超时处理（业务关键等待）？| Guardian检查语法，不判断业务等待场景 |
| B4 | 是否遗漏并发处理（业务关键并发）？| Guardian检查语法，不判断业务并发场景 |
| B5 | 是否遗漏回滚处理（业务关键失败）？| Guardian检查语法，不判断业务回滚需求 |

## 七、验证子代理提示词模板

```markdown
# 验证子代理提示词（职责边界清晰版）

## 背景说明
执行子代理已在节点2 pytest闭环内部完成以下程序化检查：
- pytest执行和覆盖率阈值检查
- Ruff lint error检查（修复阶段）
- radon圈复杂度检查（修复阶段）

Guardian节点3已验证quality_check字段：
- pytest_passed=true
- coverage_percent≥阈值
- ruff_errors=0
- radon_max_complexity≤20

**以上检查已通过，验证子代理无需重复检查。**

> **注意**：bandit安全扫描在L2测试阶段执行，不在单任务pytest闭环内

## 验证子代理核心职责（AI判断）

### 1. 业务逻辑验证
- 对照业务流程文档，判断代码是否符合业务规则
- 判断业务逻辑是否正确执行
- 判断业务数据流转是否符合预期

### 2. 验收标准验证
- 对照验收标准清单，逐项判断是否满足
- 判断验收标准的业务含义是否正确实现

### 3. 边界情况验证
- 按Checklist（B1-B5）逐项检查
- 判断是否有遗漏的边界处理

### 4. 逻辑bug发现
- 按Checklist（L1-L5）逐项检查
- 发现条件遗漏、逻辑错误等

### 5. 补充参考（非核心职责）
- Guardian warnings可作为补充发现
- 如果发现问题，在报告中提供建议，但不作为阻断依据

## Checklist检查（逐项）

请按以下Checklist逐项检查：

### 逻辑bug检测（L1-L5）
- L1: 是否遗漏条件判断分支？ → [检查结果]
- L2: 循环终止条件是否符合业务预期？ → [检查结果]
- L3: 变量是否在所有业务路径上正确赋值？ → [检查结果]
- L4: 返回值是否符合业务预期？ → [检查结果]
- L5: 异步调用顺序是否符合业务流程？ → [检查结果]

### 边界遗漏检测（B1-B5）
- B1: 是否遗漏空值处理（业务关键数据）？ → [检查结果]
- B2: 是否遗漏异常处理（业务关键操作）？ → [检查结果]
- B3: 是否遗漏超时处理（业务关键等待）？ → [检查结果]
- B4: 是否遗漏并发处理（业务关键并发）？ → [检查结果]
- B5: 是否遗漏回滚处理（业务关键失败）？ → [检查结果]

## 输出要求
- 验证结论：通过/不通过（基于业务逻辑+验收标准+边界情况）
- 不重复Guardian已检查的程序化项目
- 每项Checklist检查结果需包含：检查项编号、检查结果、问题描述（如有）
```

## 九、验证子代理完整流程

```
输入
├── 业务流程文档
├── 验收标准清单
├── 执行产出物
├── Guardian warnings（补充参考）
└── Guardian check summary（已通过项）

        ↓

验证子代理执行（AI独立完成）
├── 1. 业务逻辑验证（AI判断）
├── 2. 验收标准验证（AI判断）
├── 3. 边界情况验证（AI判断，按B1-B5）
├── 4. 逻辑bug发现（AI判断，按L1-L5）
├── 5. 补充发现（读取Guardian warnings）
└── 6. 综合结论判定（AI独立给出"通过/不通过")

        ↓

输出
├── conclusion: "通过"/"不通过"
├── verification_details
├── checklist_results
├── supplemental_findings
├── report
└── suggestions
```
