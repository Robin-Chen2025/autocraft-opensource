## 一、概述

本文档是BE-L2测试方案生成的**执行规范**，定义：

- 测试方案文档结构
- 测试场景划分规则
- 测试数据准备策略
- L2 Guardian检查项
- 验收标准定义

> **核心定位**：L2测试验证**API集成与业务流程正确性**，不检查代码质量（L1已覆盖）。

---

## 二、L1 vs L2职责边界

### 2.1 测试层级对比

| 层级 | 测试焦点 | 范围 | Guardian检查对象 |
|------|---------|------|-----------------|
| **L1** | 代码质量 | 单任务（100-200行）| pytest通过率、覆盖率、lint、类型、复杂度 |
| **L2** | API集成 | API串联、业务流程、数据流转 | pytest通过率、场景覆盖率、环境可用 |

### 2.2 L2不检查的内容

| 检查项 | L1检查 | L2不检查 | 原因 |
|--------|--------|---------|------|
| 代码覆盖率 | ✅ Block 80% | ❌ | L1已覆盖单任务 |
| Ruff lint | ✅ Block | ❌ | L1已覆盖 |
| mypy类型 | ✅ Block | ❌ | L1已覆盖 |
| 圈复杂度 | ✅ Block/Warning | ❌ | L1已覆盖 |

## 四、测试场景划分规则

### 4.1 场景提取来源

| 输入文档 | 提取方法 | 产出场景类型 |
|---------|---------|-------------|
| **业务流程文档** | 流程节点→API串联 | 正向业务流程 |
| **API设计文档** | 错误响应定义 | 异常业务流程 |
| **系统功能设计** | 功能边界条件 | 边界场景 |
| **数据库设计** | 数据约束、外键关系 | 数据边界场景 |

### 4.2 场景命名规范

```
{场景类型}-{模块编号}-{功能编号}-{编号}

场景类型：
  S-NORMAL  正向业务流程
  S-ERROR   异常业务流程
  S-EDGE    边界场景

示例：
  S-NORMAL-{模块}-F002-001  正向场景-{场景描述}-001
  S-ERROR-M02-F004-001   异常场景-工作计划-计划不存在-001
  S-EDGE-M01-F001-001    边界场景-项目管理-项目数量上限-001
```

### 4.3 API串联表示法

| 表示方式 | 说明 | 示例 |
|---------|------|------|
| 简单串联 | API编号箭头连接 | API-001 → API-002 |
| 条件分支 | 条件→不同API | API-015 → [成功:API-016, 失败:API-018] |
| 循环执行 | 多次重复同一API | API-024 × 3（3个节点）|
| 参数标注 | API+参数差异 | API-001(name=null) |

### 4.4 场景最小覆盖规则

| 规则 | 说明 |
|------|------|
| **每个API至少1个场景** | 覆盖所有测试范围的API |
| **每个业务流程至少1个正向场景** | 从业务流程文档提取 |
| **每个API错误响应至少1个异常场景** | 从API设计文档提取 |
| **数据约束边界至少1个边界场景** | 从数据库设计提取 |

## 五、L2 Guardian检查项

### 5.1 pre_guardian（测试执行前）

| 检查项ID | 检查内容 | 检查函数 | 失败处理 |
|---------|---------|---------|---------|
| BE-L2-ENV-001 | 测试数据库存在 | check_file_exists(tests/fixtures/test_L2.db) | create_issue |
| BE-L2-ENV-002 | 后端服务可用 | check_service_running(http://localhost:9001/api/health) | wait |
| BE-L2-FILE-001 | 测试文件存在 | check_file_exists(tests/L2/test_*.py) | create_issue |
| BE-L2-FIXTURE-001 | Fixture文件存在非空 | check_file_not_empty(tests/fixtures/L2_backend_data.py) | create_issue |
| BE-L2-PLAN-001 | 测试方案文档存在 | check_file_exists(docs/test-plans/{plan_id}-L2-test-plan.md) | create_issue |

### 5.2 post_guardian（测试执行后）

| 检查项ID | 检查内容 | 检查函数 | 阈值 | 失败处理 |
|---------|---------|---------|------|---------|
| BE-L2-REPORT-001 | pytest报告存在 | check_file_exists(tests/reports/L2_report.json) | - | create_issue |
| BE-L2-PASS-001 | pytest通过率 | check_pytest_pass_rate(report.json) | **100%** | Block → create_issue |
| BE-L2-SCENE-001 | 场景覆盖率 | check_scene_coverage(report.json, test-plan.md) | **100%** | Block → create_issue |

### 5.3 final_guardian（最终验收）

| 检查项ID | 检查内容 | 检查函数 | 失败处理 |
|---------|---------|---------|---------|
| BE-L2-DOC-001 | 测试报告文档存在 | check_file_exists(docs/test-reports/{plan_id}-backend-L2-report.md) | create_issue |
| BE-L2-CONCLUSION-001 | 验收结论通过 | check_report_conclusion(report.md) | create_issue |

## 六、验收标准定义

### 6.1 核心指标

| 指标 | 计算方式 | 要求 | Guardian处理 |
|------|---------|------|-------------|
| **pytest通过率** | passed_count / total_count × 100% | 100% | Block |
| **场景覆盖率** | 已通过场景数 / 方案定义总数 × 100% | 100% | Block |
| **响应时间** | 平均响应时间 | <500ms | Warning |

### 6.2 pytest报告格式

```json
{
  "summary": {
    "total": 50,
    "passed": 50,
    "failed": 0,
    "error": 0,
    "duration": 12.5
  },
  "scenes": [
    {
      "scene_id": "S-NORMAL-M01-F001-001",
      "status": "passed",
      "duration": 0.15
    },
    ...
  ],
  "conclusion": "通过",
  "coverage": {
    "scene_coverage": "100%",
    "api_coverage": "100%"
  }
}
```

### 6.3 场景覆盖率计算

```python
def check_scene_coverage(report_json, test_plan_md):
    """计算场景覆盖率"""
    # 从测试方案提取场景总数
    total_scenes = extract_scene_count(test_plan_md)
    # 从pytest报告提取通过场景数
    passed_scenes = len([s for s in report_json["scenes"] if s["status"] == "passed"])
    # 计算覆盖率
    coverage = passed_scenes / total_scenes * 100
    return coverage >= 100, coverage
```

## 七、测试数据准备与隔离策略

### 7.1 数据隔离原则

| 场景 | 是否需要清空 | 处理方式 |
|------|-------------|---------|
| **同一测试用例多次执行** | ✅ pytest自带隔离 | 每个test函数独立fixture |
| **不同测试场景之间** | ❌ 不清空 | 使用不同数据ID（fixture_A、fixture_B）|
| **全局清空时机** | 仅L2启动时一次 | 003任务初始化数据库 |

**核心策略**：不同场景使用不同数据ID，数据库一次初始化，多次测试，数据保留可追溯。

### 7.2 Fixture文件结构


### 7.3 数据初始化脚本

```python
# tests/fixtures/init_L2_db.py

import sqlite3
import shutil
from L2_backend_data import ALL_FIXTURES

def init_test_db(db_path="backend-dev/tasks.db"):
    """初始化L2测试数据库（开发数据库）"""
    conn = sqlite3.connect(db_path)
    
    # 清空所有表（仅L2启动时执行一次）
    clear_all_tables(conn)
    
    # 插入所有场景的测试数据
    for scene_name, scene_data in ALL_FIXTURES.items():
        insert_scene_data(conn, scene_name, scene_data)
    
    conn.commit()
    conn.close()

def clear_all_tables(conn):
    """清空所有表数据（仅L2启动时执行）"""
    tables = ["projects", "work_plans", "tasks", "issues"]
    for table in tables:
        conn.execute(f"DELETE FROM {table}")

def insert_scene_data(conn, scene_name, data):
    """插入场景数据（不同ID，无冲突）"""
    if "project" in data:
        conn.execute("INSERT INTO projects VALUES (?, ?, ?, ?)", 
                     (data["project"]["id"], data["project"]["name"], ...))
    if "plan" in data:
        conn.execute("INSERT INTO work_plans VALUES (?, ?, ?, ?, ?)",
                     (data["plan"]["id"], data["plan"]["project_id"], ...))
    if "task" in data:
        conn.execute("INSERT INTO tasks VALUES (?, ?, ?, ?, ?)",
                     (data["task"]["id"], data["task"]["plan_id"], ...))

if __name__ == "__main__":
    init_test_db()
```

### 7.5 数据可追溯性

测试完成后，数据库保留所有场景数据，便于追溯：

```sql
-- 查看场景A产生的数据
SELECT * FROM projects WHERE id = 1;  -- 场景A项目
SELECT * FROM work_plans WHERE project_id = 1;  -- 场景A计划

-- 查看场景B产生的数据
SELECT * FROM projects WHERE id = 2;  -- 场景B项目

-- 查看所有测试场景数据
SELECT * FROM projects WHERE id IN (1, 2, 3);
```

## 八、与现有框架对接

### 8.1 任务单映射

| 任务编号 | 变体编码 | 生成依据 |
|---------|---------|---------|
| 001 | DECODE-BE-L2-001 | FlowTicket启动时自动创建 |
| 002 | DOC-BE-L2-PLAN | 本规范定义的文档结构 |
| 003 | BUILD-BE-L2-ENV | 测试环境章节定义 |
| 004-N | BUILD-BE-L2-TEST | 测试场景清单（动态生成）|
| 005-N+1 | VERIFY-BE-L2 | pytest执行（动态执行）|
| N | DOC-BE-L2-REPORT | 测试报告 |

### 8.2 执行流程

```
BE-L2计划启动
    │
    ├─ 001: 拆解元任务（读取测试方案，生成任务单）
    │
    ├─ 002: 生成测试方案文档
    │     输入：API设计 + 业务流程 + 系统功能
    │     输出：docs/test-plans/{plan_id}-L2-test-plan.md
    │     审核：18-测试方案审核Checklist
    │
    ├─ 003: 测试环境准备
    │     输入：测试环境章节
    │     输出：test_L2.db + fixture文件
    │     Guardian：pre_guardian检查环境可用
    │
    ├─ 004+: 测试场景循环执行
    │     ├─ 生成测试代码（BUILD-BE-L2-TEST）
    │     ├─ 执行pytest（VERIFY-BE-L2）
    │     └─ Guardian：post_guardian检查通过率100%
    │
    └─ N: 生成测试报告
          输出：docs/test-reports/{plan_id}-backend-L2-report.md
          Guardian：final_guardian验收
```
