## 一、概述

本文档是L1后端测试方案的**执行规范**，定义：

- L1-BE测试方案文档结构（测试代码清单）
- pytest闭环机制（执行子代理内部）
- 验收标准与阈值参数
- Guardian检查项（E4-E7）
- execution_log.quality_check结构
- 任务单映射（测试代码生成独立任务单）

> **核心定位**：L1测试解决**代码质量问题**，生成**独立测试方案**（测试代码清单），测试代码生成作为独立任务单排在程序代码生成之前。

## 二、L1-BE测试方案文档结构

| 章节 | 内容 | 必填 | 说明 |
|------|------|------|------|
| **1. 测试范围** | 测试模块清单、不测试内容 | ✅ | 从系统功能设计提取 |

| **4. 验收标准** | pytest通过率、覆盖率阈值 | ✅ | Guardian检查依据 |
| **5. 执行流程** | 任务顺序：测试代码→程序代码→pytest闭环 | ✅ | FlowTicket对接依据 |

### 2.2 测试代码清单表格结构

| 模块编号 | 模块名称 | 测试文件路径 | 场景数量 | 数据来源 |
|---------|---------|-------------|---------|---------|
| {M-01} | {模块名称} | tests/backend/api/test_{module}_api.py | {N} | API设计文档 |
| {M-02} | {模块名称} | tests/backend/api/test_{module}_api.py | {N} | API设计文档 |

- 正向场景：每API方法≥1
- 异常场景：每API错误响应≥1
- 边界场景：每参数约束≥1

### 2.3 测试方案文档命名

```

```

**示例**：`docs/test-plans/{项目名}-L1-BE-test-plan.md`

## 三、L1测试机制定义

### 2.1 测试范围

| 维度 | 定义 |
|------|------|
| **测试焦点** | 代码质量 |
| **测试范围** | 单任务代码（100-200行）|
| **执行时机** | BE-DEV任务内部pytest闭环 |
| **测试工具** | pytest + pytest-cov |
| **生成文档** | ✅ 测试方案（测试代码清单）|

### 2.2 pytest闭环位置

**核心设计**：pytest闭环在**执行子代理内部**（FlowTicket节点2），不是独立任务单。

```
BE-DEV任务单执行（节点2）：
    │
    ├─ Crush生成代码
    │
    ├─ pytest执行 + 覆盖率采集
    │     ├─ pytest通过 → 继续
    │     └─ pytest失败 → 检查lint/类型 → 针对性修复 → pytest → 循环最多3次
    │           └─ 超限 → 返回超限结果 → FlowTicket创建问题单
    │
    ├─ lint检查（修复阶段）
    │     ├─ Ruff无Error → 继续
    │     └─ 有Error → 修复
    │
    ├─ 类型检查（修复阶段，可选）
    │     ├─ mypy无Error → 继续
    │     └─ 有Error → 修复
    │
    └─ 产出 execution_log.quality_check
          ├─ pytest_passed: true/false
          ├─ coverage_percent: 85
          ├─ ruff_errors: 0
          ├─ mypy_errors: 0
          └─ radon_max_complexity: 8
```

### 2.3 Guardian验证位置

**Guardian节点3验证**（结果验证，不执行检查工具）：

| 检查项ID | 验证内容 | 通过条件 | 不通过处理 |
|---------|----------|---------|-----------|
| E1 | 产出物文件存在 | 所有output_files存在 | 返工节点2（上限2次）|
| E2 | 产出物文件非空 | 所有output_files非空 | 返工节点2（上限2次）|
| E3 | execution_log格式正确 | JSON格式合法 | 返工节点2（上限2次）|
| **E4** | pytest执行通过 | quality_check.pytest_passed=true | 返工节点2（上限2次）|
| **E5** | 覆盖率达标 | quality_check.coverage_percent≥80% | 返工节点2（上限2次）|
| **E6** | lint检查通过 | quality_check.ruff_errors=0 | 返工节点2（上限2次）|
| **E7** | 类型检查通过（可选）| quality_check.mypy_errors=0 | Warning/返工 |

> **核心设计**：Guardian验证execution_log.quality_check字段，不直接执行pytest/ruff/mypy

## 三、验收标准与阈值参数

### 3.1 验收标准表格

| 指标 | 要求 | 检测位置 | Guardian验证 | 适用任务 |
|------|------|----------|-------------|---------|
| **pytest通过率** | 100% | pytest闭环内部 | E4: pytest_passed=true | A类任务 |
| **代码覆盖率** | ≥80% | pytest闭环内部 | E5: coverage_percent≥80% | A类任务 |
| **Ruff lint** | 无Error | pytest闭环内部（修复阶段）| E6: ruff_errors=0 | A类/B类任务 |
| **mypy类型检查** | 无Error（渐进）| pytest闭环内部（修复阶段）| E7: mypy_errors=0（可选）| A类任务 |
| **圈复杂度** | ≤10 Warning, >20 Block | pytest闭环内部（修复阶段）| 验证 radon_max_complexity≤20 | A类任务 |

### 3.2 阈值参数说明

| 参数 | 后端阈值 | 说明 |
|------|---------|------|
| coverage_percent | ≥80% | pytest-cov采集，行覆盖率 |
| ruff_errors | =0 | Ruff lint检查，Error级别 |
| mypy_errors | =0（渐进）| mypy类型检查，渐进启用 |
| radon_max_complexity | ≤20（Block）| 圈复杂度，>20阻断 |

### 3.3 B类任务处理

| 任务类型 | pytest闭环 | quality_check字段 | Guardian检查 |
|---------|-----------|-----------------|-------------|
| **A类**（业务逻辑代码）| ✅ 需要 | 有值 | E4-E7全部检查 |
| **B类**（基础设施代码）| ❌ 不需要 | null | 跳过E4-E7 |
| **DOC类**（文档生成）| ❌ 不需要 | null | 跳过E4-E7 |

## 四、execution_log.quality_check结构

### 4.1 结构定义

```json
{
  "pytest_passed": true,          // pytest执行结果（true/false）
  "coverage_percent": 85,         // 代码覆盖率百分比（后端≥80%）
  "ruff_errors": 0,               // Ruff lint Error数量
  "mypy_errors": 0,               // mypy类型检查Error数量（可选）
  "radon_max_complexity": 8       // 最大圈复杂度
}
```

### 4.2 A类任务示例（通过）

```json
{
  "pytest_passed": true,
  "coverage_percent": 85,
  "ruff_errors": 0,
  "mypy_errors": 0,
  "radon_max_complexity": 8
}
```

**Guardian验证**：E4-E7全部通过 → 任务完成

### 4.3 A类任务示例（覆盖率不足）

```json
{
  "pytest_passed": true,
  "coverage_percent": 65,         // <80% 不达标
  "ruff_errors": 0,
  "mypy_errors": 0,
  "radon_max_complexity": 12      // ≤20 通过
}
```

**Guardian验证**：E5不通过 → 返工节点2

### 4.4 B类任务示例

```json
{
  "pytest_passed": null,
  "coverage_percent": null,
  "ruff_errors": 0,
  "mypy_errors": null,
  "radon_max_complexity": null
}
```

**Guardian验证**：pytest_passed=null → 跳过E4-E7，只检查E1-E3

## 五、测试代码生成规范

### 5.1 测试文件命名

| 规则 | 说明 | 示例 |
|------|------|------|
| 文件位置 | tests/backend/{module}/ | tests/backend/api/test_user_api.py |
| 文件命名 | test_{功能名}.py | test_task_api.py |
| 函数命名 | test_{场景描述} | test_create_task_success |

### 5.2 测试代码结构

```python
# tests/backend/api/test_task_api.py

import pytest
from fastapi.testclient import TestClient
from main import app

client = TestClient(app)

class TestTaskAPI:
    """任务API测试类"""
    
    def test_create_task_success(self):
        """正向场景：创建任务成功"""
        # 准备测试数据
        task_data = {
            "plan_id": 1,
            "task_name": "测试任务",
            "task_type": "BUILD"
        }
        
        # 执行API调用
        response = client.post("/api/tasks", json=task_data)
        
        # 验证响应
        assert response.status_code == 200
        assert response.json()["task_id"] > 0
        
    def test_create_task_plan_not_found(self):
        """异常场景：计划不存在"""
        task_data = {
            "plan_id": 999,  # 不存在的计划ID
            "task_name": "测试任务",
            "task_type": "BUILD"
        }
        
        response = client.post("/api/tasks", json=task_data)
        
        assert response.status_code == 404
        assert response.json()["error_code"] == "PLAN_NOT_FOUND"
```

### 5.3 覆盖率采集配置

```bash
# pytest命令（执行子代理内部）
pytest tests/backend/api/test_task_api.py \
    --cov=backend/api \
    --cov-report=json \
    --cov-fail-under=80
```

**覆盖率报告输出**：`.coverage` + `coverage.json`

## 六、L1场景清单模板（执行子代理参考）

> **说明**：L1场景清单不在总纲展开，由执行子代理在pytest闭环内部动态生成

### 6.1 场景提取规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 每个API方法≥1正向场景 | 成功路径测试 | test_create_task_success |
| 每个参数边界≥1边界场景 | 参数验证测试 | test_create_task_name_empty |
| 每个错误响应≥1异常场景 | 错误处理测试 | test_create_task_plan_not_found |

### 6.2 场景命名规范

```
test_{API方法}_{场景类型}_{描述}

场景类型：
  success   正向场景
  error     异常场景
  edge      边界场景

示例：
  test_create_task_success
  test_create_task_error_plan_not_found
  test_create_task_edge_name_empty
```

### 6.3 最小场景清单示例（单个API）

| 场景ID | 场景名称 | 测试函数 | 验收标准 |
|--------|---------|---------|---------|
| L1-API-001-S001 | 创建任务成功 | test_create_task_success | 200响应 |
| L1-API-001-E001 | 计划不存在 | test_create_task_error_plan_not_found | 404响应 |
| L1-API-001-E002 | 任务名称为空 | test_create_task_edge_name_empty | 400响应 |

## 八、与FlowTicket对接

### 8.1 任务单映射（BE-DEV计划）

| 任务编号 | 变体编码 | 任务内容 | 说明 |
|---------|---------|---------|------|
| 001 | DECODE-BE-DEV-001 | 拆解任务单 | 读取设计文档，生成任务单清单 |
| 002 | DOC-L1-BE-PLAN | 生成L1-BE测试方案 | 本规范定义的测试代码清单 |
| **003** | **BUILD-L1-BE-TEST** | **生成测试代码（独立任务单）** | 根据测试方案生成测试代码文件 |
| 004-N | BUILD-BE-CODE | 生成程序代码 + pytest闭环 | pytest闭环验证测试代码 |
| N+1 | DOC-BE-REPORT | 生成测试报告 | pytest执行结果汇总 |

> **核心设计**：测试代码生成（003）是独立任务单，排在程序代码生成（004）之前

### 8.2 执行流程（BE-DEV计划）

```
BE-DEV计划启动
    │
    ├─ 001: 拆解任务单
    │     输入：API设计 + 系统功能设计
    │     输出：任务单清单
    │
    ├─ 002: 生成L1-BE测试方案
    │     输入：API设计 + 系统功能设计
    │     输出：docs/test-plans/{plan_id}-L1-BE-test-plan.md
    │     审核：32-L1-BE审核Checklist
    │
    ├─ 003: 生成测试代码（独立任务单）
    │     输入：L1-BE测试方案
    │     输出：tests/backend/api/test_*.py
    │     Guardian：测试文件存在 + 非空
    │     **关键**：测试代码在程序代码之前
    │
    ├─ 004-N: 生成程序代码 + pytest闭环
    │     ├─ Crush生成程序代码
    │     ├─ pytest执行（运行测试代码）
    │     ├─ 失败 → 修复 → pytest → 循环最多3次
    │     ├─ lint检查 + 类型检查
    │     └─ 产出 execution_log.quality_check
    │
    └─ N+1: 生成测试报告
          输出：docs/test-reports/{plan_id}-L1-BE-report.md
          Guardian：final_guardian验收
```

### 8.3 测试先行价值

| 价值维度 | 说明 |
|---------|------|
| **TDD思想** | 测试代码先行，程序代码满足测试要求 |
| **任务独立** | 测试代码生成可追踪、可审阅、可验证 |
| **验收前置** | 测试代码通过后才生成程序代码 |
| **错误隔离** | 测试代码问题不影响程序代码生成 |
