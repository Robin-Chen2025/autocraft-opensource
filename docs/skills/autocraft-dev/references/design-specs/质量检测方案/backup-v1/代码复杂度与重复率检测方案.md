## 一、圈复杂度检测方案

### 1.1 什么是圈复杂度

**通俗理解**：圈复杂度衡量的是"代码有多少条岔路"。

**计算公式**：圈复杂度 = 分支数量 + 1

**类比**：开车时的路口选择
- 一条直路（无岔路）→ 圈复杂度 = 1
- 一个岔路口（if判断）→ 圈复杂度 = 2
- 两个岔路口（if + else if）→ 圈复杂度 = 3

### 1.2 阈值分级

| 圈复杂度 | 驾车类比 | 代码含义 | 建议 | Guardian处理 |
|----------|----------|----------|------|--------------|
| **1-5** | 直路或少量岔路 | 简单明了，一目了然 | ✅ 无需关注 | Passed |
| **6-10** | 中等复杂路口 | 可接受，但建议简化 | ⚠️ 建议简化 | Warning |
| **11-20** | 复杂立交桥 | 难以理解，容易出错 | ❌ 需要重构 | Warning |
| **> 20** | 迷宫 | 极难理解，几乎必然出错 | 🚫 **必须重构** | **Block** |

### 1.3 单任务检测方案

**检测时机**：pytest闭环内部（修复阶段）

> **Guardian节点3职责**：验证 `quality_check.radon_max_complexity≤20`（>20则Block）

**检测工具**：radon（Python）

```bash
# 安装
pip install radon

# 检查圈复杂度
radon cc app/ -a -nc  # 显示复杂度>=C的函数

# 输出示例
app/flowticket/executor.py
    F 10:0 execute_node_2 - A (10)  # 圈复杂度=10，评级A（可接受）
    F 20:0 execute_node_3 - C (15)  # 圈复杂度=15，评级C（需要重构）
```

**评级对应**：

| 评级 | 圈复杂度范围 | 建议 |
|------|-------------|------|
| A | 1-5 | ✅ 简单 |
| B | 6-10 | ⚠️ 可接受 |
| C | 11-20 | ❌ 需要重构 |
| D | 21-50 | 🚫 必须重构 |
| F | >50 | 🚫 极其复杂 |

### 1.4 pytest闭环内部检测流程

```
执行子代理内部 pytest 闭环（修复阶段）：

1. radon圈复杂度检测
   ├── radon cc app/ -nc -s
   ├── 解析高复杂度函数
   └── 产出quality_check.radon_max_complexity字段

2. 针对性修复（如有高复杂度函数）
   ├── 拆分函数（见1.5降低圈复杂度的方法）
   └── 重新检测 → 循环最多3次

3. 产出quality_check字段
   ├── radon_max_complexity: 8
   ├── complex_functions: ["execute_node_2(A)", ...]
   └── block_functions: []  # >20的函数列表
```

> **Guardian节点3验证**：检查 `quality_check.radon_max_complexity≤20`
> - ≤10 → Passed（无需关注）
> - 11-20 → Warning（建议重构）
> - >20 → Block（必须重构，返工节点2）

### 1.5 降低圈复杂度的方法

**方法：拆分函数**

```python
# 原代码（圈复杂度=7）
def calculate_final_grade(score, attendance, has_project):
    if attendance < 80:  # 岔路1
        score -= 10
    if has_project:  # 岔路2
        if project_score > 90:  # 岔路3（嵌套）
            score += 5
        else:
            score += 2
    if score >= 90:  # 岔路4
        return "优秀"
    elif score >= 60:  # 岔路5
        return "通过"
    else:
        return "不通过"

# 优化后（每个函数圈复杂度=2-4）
def calculate_final_grade(score, attendance, has_project):
    score = adjust_by_attendance(score, attendance)
    score = adjust_by_project(score, has_project)
    return get_grade_label(score)

def adjust_by_attendance(score, attendance):  # 圈复杂度=2
    if attendance < 80:
        return score - 10
    return score

def adjust_by_project(score, has_project):  # 圈复杂度=2
    if has_project:
        return score + 5
    return score

def get_grade_label(score):  # 圈复杂度=4
    if score >= 90:
        return "优秀"
    elif score >= 60:
        return "通过"
    else:
        return "不通过"
```

## 二、代码重复率检测方案

### 2.1 什么是代码重复率

**通俗理解**：代码重复率衡量的是"有多少代码是抄来的"。

**高重复率的含义**：存在可以抽取的公共逻辑，可生成标准类/公共函数来简化。

### 2.2 单任务检测的局限性

**当前任务粒度**：100-200行代码

| 局限 | 说明 |
|------|------|
| **检测范围小** | 只检测当前100-200行代码，无法跨任务检测 |
| **内部重复有限** | 单个小任务内，重复概率本身就低 |
| **跨任务重复看不到** | Task A和Task B的相似代码，Guardian只检测A，看不到B |

**结论**：单任务Guardian检测代码重复率价值低，不采用。

### 2.3 检测时机分级

| 检测时机 | 检测范围 | 能发现的问题 | 是否采用 |
|----------|----------|--------------|----------|
| **单任务Guardian** | 当前100-200行 | 单任务内重复 | ⚪ **不检测** |
| **L2测试阶段** | 后端整体/前端整体 | 跨任务重复 | ✅ **检测** |
| **系统级审计** | 前后端+配置+文档 | 跨前后端重复 | ✅ **检测** |

## 三、L2测试阶段检测方案

### 3.1 L2测试流程调整

```
BE-L2/FE-L2 测试计划
├── 001: 测试方案生成
├── 002: 测试计划拆解
├── 003: 测试环境准备
├── 004-N: 测试场景执行（每个场景）
│   ├── 生成测试代码
│   ├── 执行测试（pytest/Vitest）
│   └── 测试报告
├── N-1: 测试总结报告
└── **新增：N 代码质量综合检测**
    ├── 覆盖率汇总（各场景覆盖率合并）
    ├── ✅ 代码重复率检测（全局）
    ├── 圈复杂度汇总
    ├── 安全扫描汇总
    └── 生成质量综合报告
```

### 3.2 检测内容

| 检测项 | 检测范围 | 工具 | 输出 |
|--------|----------|------|------|
| **覆盖率汇总** | 后端/前端整体 | pytest-cov/Vitest | 覆盖率综合报告 |
| **代码重复率** | 全项目（后端或前端） | pylint/Ruff | 重复模式分析 + 抽取建议 |
| **圈复杂度汇总** | 全项目 | radon | 高复杂度函数清单 |
| **安全扫描汇总** | 全项目 | bandit/pip audit | 安全问题汇总 |

### 3.3 检测结果处理

| 检测结果 | 处理方式 |
|----------|----------|
| 覆盖率不达标 | Block L2测试通过，需要补充测试 |
| 代码重复率 >10% | Warning + 生成优化任务单（不阻断） |
| 圈复杂度超标函数 | Warning + 列入重构清单（不阻断） |
| 安全问题 | Block（高危）/ Warning（中危） |

### 3.4 L2质量检测任务单设计

```yaml
# BE-L2 最后一个任务单示例
task_id: BE-L2-M03-N
task_type: VERIFY
name: 后端L2质量综合检测
description: |
  1. 覆盖率汇总（各场景覆盖率合并）
  2. 代码重复率检测（后端整体）
  3. 圈复杂度汇总
  4. 安全扫描汇总
  5. 生成质量综合报告
  
executor: 质量检测脚本
verifier: 验证子代理（审查报告质量）
input:
  - 各场景覆盖率报告
  - 后端源代码
output:
  - 覆盖率综合报告
  - 重复模式分析报告
  - 抽取建议任务单（如有）
acceptance_criteria:
  - 覆盖率 ≥ 目标阈值
  - 安全问题无高危
  - 质量报告完整
```

## 四、系统级质量审计方案

### 4.1 触发条件标准化

| 触发条件 | 定义 | 自动检测方式 |
|----------|------|--------------|
| **唯一触发** | **里程碑完成** | 程序检测：所有计划状态=completed + 所有L2/L3测试=passed |

**里程碑定义**：

| 里程碑类型 | 定义 | 触发条件检测 |
|------------|------|--------------|
| **M1: V1.0完成** | 所有计划完成 | 状态检测 |
| **M2: V1.1完成** | 新增功能计划完成 | 状态检测 |
| **M3: V2.0完成** | 大版本升级计划完成 | 状态检测 |

### 4.2 触发检测时机

| 检测时机 | 触发方式 | 说明 |
|----------|----------|------|
| **计划完成时** | 自动检测 | 每个计划状态变为completed时触发检测 |
| **L2/L3测试通过时** | 自动检测 | 测试状态变为passed时触发检测 |
| **心跳检测** | 定时检测 | 每30分钟检测一次（兜底机制） |

### 4.3 自动触发流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│              系统级质量审计自动触发流程                                     │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  里程碑检测（自动化）                                                     │
│  ├── 每次计划完成时触发检测                                               │
│  ├── 检测：所有计划状态=completed                                         │
│  ├── 检测：所有L2/L3测试=passed                                           │
│  └── 条件满足 → 自动创建系统级质量审计任务单                               │
│                                                                         │
│           ↓                                                             │
│                                                                         │
│  系统级质量审计任务单                                                     │
│  ├── 执行质量检测脚本                                                     │
│  │   ├── 前后端覆盖率汇总                                                │
│  │   ├── 系统级代码重复率检测                                            │
│  │   ├── 跨前后端常量/验证规则对比                                       │
│  │   ├── 安全扫描汇总                                                    │
│  │   └── 生成系统质量综合报告                                            │
│  ├── 验证子代理审查报告质量                                              │
│  └── 状态=completed                                                      │
│                                                                         │
│           ↓                                                             │
│                                                                         │
│  里程碑状态更新                                                          │
│  ├── milestone_status = "quality_audit_completed"                       │
│  └── 发送通知给用户（报告摘要）                                           │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 4.4 检测内容

| 检测项 | 检测范围 | 能发现的问题 |
|--------|----------|--------------|
| **前后端覆盖率汇总** | 后端+前端 | 整体覆盖情况 |
| **系统级代码重复率** | 后端+前端+配置 | 跨前后端重复 |
| **跨前后端常量对比** | 前端常量 vs 后端常量 | 定义不一致 |
| **验证规则对比** | 前端验证 vs 后端验证 | 规则不一致 |
| **安全扫描汇总** | 后端+前端+依赖 | 整体安全状况 |

## 五、数据库设计支持

### 5.1 milestones表

```sql
-- milestones 表
CREATE TABLE milestones (
    id INTEGER PRIMARY KEY,
    project_id INTEGER NOT NULL,
    name VARCHAR(100) NOT NULL,           -- 如 "V1.0完成"
    version VARCHAR(50),                  -- 如 "1.0.0"
    status VARCHAR(20) DEFAULT 'pending', -- pending/in_progress/quality_audit_completed/completed
    quality_audit_task_id INTEGER,        -- 关联的系统级质量审计任务单
    created_at TIMESTAMP,
    completed_at TIMESTAMP,
    
    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (quality_audit_task_id) REFERENCES tasks(id)
);
```

## 六、检测判断标准化

### 6.1 计划完成判断

```python
def is_plan_completed(plan_id: str) -> bool:
    """判断计划是否完成（标准化）"""
    
    plan = get_plan(plan_id)
    
    # 条件1：计划状态为completed
    if plan.status != "completed":
        return False
    
    # 条件2：计划的L2/L3测试状态为passed（如果有测试计划）
    test_plans = get_test_plans(plan_id)
    if test_plans:
        all_tests_passed = all(
            tp.test_status == "passed"
            for tp in test_plans
        )
        if not all_tests_passed:
            return False
    
    return True
```

### 6.2 里程碑完成判断

```python
def check_milestone_completed(project_id: str) -> bool:
    """检测里程碑是否完成（标准化）"""
    
    # 获取项目所有计划
    plans = get_project_plans(project_id)
    
    # 检测条件：所有计划状态为completed
    all_completed = all(
        plan.status == "completed" 
        for plan in plans
    )
    
    # 检测条件：所有计划的L2/L3测试通过
    all_tests_passed = all(
        plan.test_status == "passed"
        for plan in plans
    )
    
    return all_completed and all_tests_passed
```

## 七、最终方案汇总

### 7.1 圈复杂度检测

| 检测时机 | 检测范围 | 阈值 | Guardian处理 |
|----------|----------|------|--------------|
| 单任务Guardian | 当前任务代码 | ≤10 Warning, >20 Block | Warning/Block |

### 7.2 代码重复率检测

| 检测时机 | 检测范围 | 阈值 | 处理方式 |
|----------|----------|------|----------|
| 单任务Guardian | 无 | - | ⚪ 不检测 |
| L2测试阶段 | 后端/前端整体 | >10% Warning | Warning + 优化任务单 |
| 系统级审计 | 前后端+配置 | >10% Warning | 报告 + 用户通知 |

### 7.3 系统级审计触发

| 项目 | 标准 |
|------|------|
| **触发条件** | 所有计划完成 + 所有L2/L3测试通过 |
| **触发时机** | 计划完成时自动检测 + 心跳兜底检测 |
| **执行内容** | 系统级质量检测脚本 |
| **输出** | 系统质量综合报告 + 抽取建议 |
