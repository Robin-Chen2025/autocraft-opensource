## 一、概述

本文档定义 AutoCraft 项目测试体系的顶层架构，包括：

- 测试层级定义（L1/L2/L3职责边界）
- 各层级测试策略与验收标准
- 测试执行顺序与依赖关系
- Guardian检查体系

> **核心定位**：用户审阅此文档可看到完整测试体系，开发计划引用此文档标注验收标准。

## 二、测试层级定义

### 2.1 三层测试架构

| 层级 | 测试焦点 | 范围 | 执行时机 | 生成文档 |
|------|---------|------|---------|---------|
| **L1** | 代码质量 | 单任务（100-200行）| BE-DEV/FE-DEV计划 | ✅ 测试方案（测试代码清单）|
| **L2** | API/组件集成 | API串联、页面流程、数据流转 | BE-L2/FE-L2计划 | ✅ 测试方案（场景清单）|
| **L3** | 业务流程E2E | 端到端用户操作流程 | L3-E2E计划 | ✅ 测试方案（流程清单）|

> **核心设计变更（2026-04-15）**：L1也生成独立测试方案（测试代码清单），测试代码生成作为独立任务单（003）排在程序代码生成之前。

### 2.2 职责边界

| 检查项 | L1 | L2 | L3 |
|--------|----|----|-----|
| pytest通过率 | ✅ | ✅ | - |
| 代码覆盖率 | ✅ | ❌ | ❌ |
| lint检查 | ✅ | ❌ | ❌ |
| 类型检查 | ✅ | ❌ | ❌ |
| 圈复杂度 | ✅ | ❌ | ❌ |
| API串联正确性 | ❌ | ✅ | ✅ |
| 数据流转正确性 | ❌ | ✅ | ✅ |
| 用户操作流程 | ❌ | ❌ | ✅ |

**核心原则**：
- L1解决代码质量问题，后续层级不再重复检查
- L2验证API/组件集成，确保模块间协作正确
- L3验证完整业务流程，确保用户体验正确

## 三、L1测试策略（代码质量）

### 3.1 验收标准

| 指标 | 要求 | 检测位置 | Guardian验证 | 适用任务 |
|------|------|----------|-------------|---------|
| **pytest通过率** | 100% | pytest闭环内部（节点2）| E4: pytest_passed=true | A类任务 |
| **代码覆盖率** | ≥80%（后端）/≥70%（前端）| pytest闭环内部（节点2）| E5: coverage_percent≥阈值 | A类任务 |
| **Ruff lint** | 无Error | pytest闭环内部（修复阶段）| E6: ruff_errors=0 | A类/B类任务 |
| **mypy类型检查** | 无Error | pytest闭环内部（修复阶段）| E7: mypy_errors=0（渐进）| A类任务 |
| **圈复杂度** | ≤10 Warning, >20 Block | pytest闭环内部（修复阶段）| 验证 radon_max_complexity≤20 | A类任务 |

> **L1-BE/FE-DEV任务顺序（测试先行）**：

> **设计澄清（2026-04-16）**：L1测试方案已在制定阶段预生成，开发计划执行时直接读取测试方案生成测试代码，不需要单独的"生成测试方案"任务。

```
BE-DEV/FE-DEV计划执行：
    │
    ├─ 001: 拆解任务单
    │
    ├─ 002~N-1: 功能循环（每功能2个任务）
    │     │
    │     ├─ 第1个任务：生成测试代码
    │     │     ├─ 输入：L1测试方案（已预生成）
    │     │     ├─ 输出：tests/backend/api/test_*.py 或 tests/frontend/components/*.test.ts
    │     │     └─ Guardian：测试文件存在 + 非空
    │     │
    │     └─ 第2个任务：生成程序代码 + pytest/vitest闭环
    │           ├─ Crush生成程序代码
    │           ├─ pytest/vitest执行（运行测试代码）
    │           ├─ 失败 → 修复 → pytest/vitest → 循环最多3次
    │           ├─ lint检查 + 类型检查
    │           └─ 产出 execution_log.quality_check
    │
    └─ N: 生成测试报告
```

> **关键设计**：
> 1. L1测试方案（测试代码清单）在制定阶段预生成，开发计划直接引用
> 2. 每个功能循环包含2个任务：测试代码 → 程序代码+pytest闭环
> 3. 测试代码生成在程序代码之前，实现测试先行（TDD思想）

**Guardian节点3验证（结果验证）**：

| 检查项ID | 验证内容 | 通过条件 | 不通过处理 |
|---------|----------|---------|-----------|
| E1 | 产出物文件存在 | 所有output_files存在 | 返工节点2（上限2次）|
| E2 | 产出物文件非空 | 所有output_files非空 | 返工节点2（上限2次）|
| E3 | execution_log格式正确 | JSON格式合法 | 返工节点2（上限2次）|
| **E4** | pytest执行通过 | quality_check.pytest_passed=true | 返工节点2（上限2次）|
| **E5** | 覆盖率达标 | quality_check.coverage_percent≥阈值 | 返工节点2（上限2次）|
| **E6** | lint检查通过 | quality_check.ruff_errors=0 | 返工节点2（上限2次）|
| **E7** | 类型检查通过（可选）| quality_check.mypy_errors=0 | Warning/返工 |

> **B类任务处理**：quality_check字段为null，跳过E4-E7

### 3.3 L1任务类型分类

| 任务类型 | 验证分类 | pytest闭环 | Guardian检查 |
|---------|---------|-----------|-------------|
| **A类**（业务逻辑代码）| pytest验证 | ✅ 需要 | pytest + 覆盖率 + lint + 类型 |
| **B类**（基础设施代码）| Guardian直接检查 | ❌ 不需要 | 文件存在 + 格式检查 |
| **DOC类**（文档生成）| Guardian格式检查 | ❌ 不需要 | 文档存在 + 章节完整 |

### 3.4 B类任务验证

| 产出物类型 | Guardian检查项 |
|-----------|---------------|
| 数据库表结构 | alembic迁移检查 |
| 公共后端模块 | 文件存在 + 导入检查 |
| 公共前端组件 | 文件存在 + 构建检查 |
| 项目骨架 | 目录结构检查 |
| 环境配置 | 服务启动检查 |

## 四、L2测试策略（集成测试）

### 4.1 验收标准

| 指标 | 要求 | Guardian处理 | 适用计划 |
|------|------|-------------|---------|
| **pytest通过率** | 100% | Block | BE-L2 |
| **vitest通过率** | 100% | Block | FE-L2 |
| **场景覆盖率** | 100% | Block | BE-L2/FE-L2 |

### 4.2 L2执行流程

```
BE-L2/FE-L2计划执行：
    │
    ├─ 002: 生成测试方案文档
    │     ├─ 从API设计/业务流程提取场景
    │     ├─ 定义测试场景清单
    │     └─ 审核：18-L2测试方案审核Checklist
    │
    ├─ 003: 测试环境准备
    │     ├─ 初始化测试数据库（开发数据库）
    │     ├─ 插入Fixture数据（不同场景不同ID）
    │     └─ 启动后端服务
    │
    ├─ 004-N: 测试场景循环执行
    │     ├─ 生成测试代码（每个场景）
    │     ├─ pytest/vitest执行
    │     └─ Guardian检查：通过率100% + 场景覆盖率100%
    │
    └─ N: 生成测试报告
```

### 4.3 数据隔离策略

| 场景 | 是否清空数据库 | 处理方式 |
|------|---------------|---------|
| L2启动时 | ✅ 一次性清空 | 003任务初始化 |
| 不同测试场景之间 | ❌ 不清空 | 使用不同数据ID |
| 同一场景多次执行 | pytest自带隔离 | Fixture自动隔离 |

**核心原则**：数据库一次初始化，多次测试，数据保留可追溯。

### 4.4 测试场景划分规则

| 场景类型 | 提取来源 | 命名规范 |
|---------|---------|---------|
| 正向业务流程 | 业务流程文档 | S-NORMAL-MXX-FXXX-XXX |
| 异常业务流程 | API错误响应定义 | S-ERROR-MXX-FXXX-XXX |
| 边界场景 | 数据约束、功能边界 | S-EDGE-MXX-FXXX-XXX |

## 五、L3测试策略（E2E测试）

### 5.1 验收标准

| 指标 | 要求 | Guardian处理 |
|------|------|-------------|
| **Playwright通过率（两层合计）** | 100% | Block |
| **流程覆盖率** | 100% | Block |

### 5.2 L3执行流程

```
L3-E2E计划执行：
    │
    ├─ 002: 生成L3测试方案文档
    │     ├─ 从业务流程文档提取用户操作流程
    │     └─ 定义E2E测试场景清单
    │
    ├─ 003: 测试环境准备
    │     ├─ 验证前后端服务可用
    │     └─ 验证数据库状态
    │
    ├─ 004-N: 业务流程循环测试（两层强制）
    │     ├─ 第一层：无头快速验证（必须通过）
    │     ├─ 第二层：有头完整验证（必须通过）
    │     └─ Guardian检查：两层通过率都100%
    │
    └─ N: 生成L3测试报告
```

**强制要求**：两层测试全部通过才算L3验收通过。

## 六、测试执行顺序与依赖

### 6.1 计划类型执行顺序

```
项目启动
    │
    ▼
INFRA（基础建设）
    │ L1：项目骨架检查
    ▼
BE-DEV（后端开发）
    │ 001: 拆解任务单
    │ 002~N-1: 功能循环（每功能：测试代码 → 程序代码+pytest闭环）
    │ 测试方案引用：L1-BE测试方案（已预生成）
    ▼
BE-L2（后端集成测试）
    │ L2：API串联验证
    │ 前置：BE-DEV status="completed"
    │ 测试方案引用：BE-L2测试方案（已预生成）
    ▼
FE-DEV（前端开发）
    │ 001: 拆解任务单
    │ 002~N-1: 功能循环（每功能：测试代码 → 程序代码+vitest闭环）
    │ 前置：BE-L2 status="completed"（后端API可用）
    │ 测试方案引用：L1-FE测试方案（已预生成）
    ▼
FE-L2（前端集成测试）
    │ L2：页面流程验证
    │ 前置：FE-DEV status="completed"
    │ 测试方案引用：FE-L2测试方案（已预生成）
    ▼
L3-E2E（端到端测试）
    │ L3：业务流程验证
    │ 前置：FE-L2 status="completed"
    │ 测试方案引用：L3-E2E测试方案（已预生成）
    ▼
项目验收
```

> **核心设计澄清（2026-04-16）**：
> 1. 所有测试方案（L1/L2/L3）在制定阶段预生成，开发计划直接引用
> 2. BE-DEV/FE-DEV任务结构：001拆解 → 002~N-1功能循环（测试代码→程序代码）
> 3. BE-L2/FE-L2/L3-E2E任务结构：001拆解 → 002环境 → 003~N-1场景/流程循环

### 6.2 L1/L2/L3执行依赖矩阵

| 当前层级 | 前置层级 | 前置状态 | 说明 |
|---------|---------|---------|------|
| L1（BE-DEV任务）| INFRA | completed | 项目骨架已建立 |
| L2（BE-L2计划）| BE-DEV | completed | 后端API已开发并通过L1 |
| L1（FE-DEV任务）| BE-L2 | completed | 后端API已通过L2验证 |
| L2（FE-L2计划）| FE-DEV | completed | 前端页面已开发并通过L1 |
| L3（L3-E2E计划）| FE-L2 | completed | 前端已通过L2验证 |

## 八、Guardian检查体系

> **核心设计**：Guardian节点3验证执行子代理产出的quality_check字段，不直接执行检查工具

### 8.1 Guardian检查项总览

| 层级 | pre_guardian（节点1）| post_guardian（节点3）| final_guardian（节点5）|
|------|---------------------|----------------------|----------------------|
| **L1** | 输入文档存在、测试文件存在 | **验证quality_check字段**（E4-E7）| 验证子代理验收 |
| **L2** | 测试环境可用、Fixture存在 | pytest通过率、场景覆盖率 | 测试报告存在 |
| **L3** | 前后端服务可用 | Playwright通过率、流程覆盖率 | E2E报告存在 |

**L1 post_guardian 检查项详情（验证quality_check）**：

> 详见第三章3.2节"Guardian节点3验证（结果验证）"表格，包含E1-E7检查项完整定义

**检查项概览**：
| 检查项ID | 验证内容 | 适用任务 |
|---------|----------|---------|
| E1-E3 | 产出物文件存在+非空+格式正确 | 所有任务 |
| E4-E7 | pytest+覆盖率+lint+类型检查结果验证 | A类任务 |

> **说明**：E4-E7验证的是执行子代理内部pytest闭环产出的结果，Guardian不执行检查工具

### 8.2 Guardian验证函数库

| 验证函数 | 适用层级 | 说明 |
|---------|---------|------|
| `check_file_exists` | 所有层级 | 产出物文件存在检查 |
| `check_file_not_empty` | 所有层级 | 产出物文件非空检查 |
| `check_json_format` | 所有层级 | JSON格式合法性检查 |
| `validate_quality_check` | L1 | **验证quality_check字段**（E4-E7）|
| `check_scene_coverage` | L2 | 场景覆盖率检查 |
| `check_playwright_result` | L3 | Playwright检查 |
