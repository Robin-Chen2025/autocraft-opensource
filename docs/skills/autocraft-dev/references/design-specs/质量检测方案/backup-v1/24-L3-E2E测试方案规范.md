## 一、概述

本文档是L3-E2E测试方案生成的**执行规范**，定义：

- 测试方案文档结构
- 业务流程场景划分规则
- Playwright两层测试策略
- L3 Guardian检查项
- 验收标准定义

> **核心定位**：L3验证**完整业务流程**（用户视角），前置条件是前后端已通过L2验证。

## 二、L2 vs L3核心差异

### 2.1 测试焦点对比

| 维度 | L2 | L3 |
|------|----|----|
| **测试焦点** | API/组件集成 | **完整业务流程** |
| **测试视角** | 模块视角 | **用户视角** |
| **测试范围** | API串联、页面流程 | 登录→操作→完成全流程 |
| **数据来源** | Fixture数据（开发数据库）| 真实系统数据（前后端已部署）|
| **依赖关系** | 后端API可用 | **前后端+数据库全部可用** |

### 2.2 执行时机对比

| 层级 | 前置条件 | 执行时机 |
|------|---------|---------|
| L2（BE-L2）| BE-DEV completed | 后端API开发完成后 |
| L2（FE-L2）| FE-DEV completed + BE-L2 completed | 前端开发完成后 |
| **L3（L3-E2E）** | **FE-L2 completed** | 前后端全部验证通过后 |

## 6. 两层测试策略

### 6.1 第一层：无头快速验证

| 维度 | 配置 |
|------|------|
| 执行模式 | headless: true |
| 浏览器 | Chromium |
| 速度 | 快（自动化CI）|
| 目的 | 发现基础问题、验证流程连通 |

### 6.2 第二层：有头完整验证（必须执行）

| 维度 | 配置 |
|------|------|
| 执行模式 | headless: false |
| 浏览器 | Chromium |
| 速度 | 慢（人工观察）|
| 目的 | **验证用户交互、视觉细节、真实体验** |
| 要求 | **必须执行，不可跳过** |

### 6.3 两层执行策略

```
第一层（无头）执行 → 全部通过 → 第二层（有头）执行（必须）
                ↓              ↓
              失败 → 分析问题 → 失败 → 分析问题 → 修复 → 重试两层
                ↓              ↓
              修复 → 重试第一层    通过 → L3完成
```

**强制要求**：两层全部通过才算L3验收通过。

## 四、业务流程场景划分规则

### 4.1 场景提取来源

| 输入文档 | 提取方法 | 产出场景类型 |
|---------|---------|-------------|
| **业务流程文档** | 每个流程→用户操作步骤 | 核心业务流程场景 |
| **PRD** | 用户故事→端到端流程 | 核心业务流程场景 |
| **UI设计文档** | 页面跳转关系 | 页面导航流程场景 |
| **系统功能设计** | 功能边界条件 | 边界业务流程场景 |

### 4.2 场景命名规范

```
{场景类型}-{流程编号}-{编号}

场景类型：
  E2E-FXXX   核心业务流程（F001项目管理流程）
  E2E-ERR    异常业务流程
  E2E-EDGE   边界业务流程

示例：
  E2E-F001-001  核心流程-项目管理-001
  E2E-F002-001  核心流程-工作计划执行-001
  E2E-ERR-001   异常流程-API错误处理-001
```

### 4.3 用户操作步骤表示法

| 表示方式 | 说明 | 示例 |
|---------|------|------|
| 顺序操作 | 箭头连接步骤 | 打开首页 → 点击创建 → 输入 → 保存 |
| 条件分支 | 条件→不同路径 | 点击执行 → [成功：查看进度， 失败：查看错误] |
| 验证点 | 预期结果标注 | 保存 → **验证：项目创建成功** |

### 4.4 场景最小覆盖规则

| 规则 | 说明 |
|------|------|
| **每个核心业务流程至少1个场景** | 从业务流程文档提取 |
| **每个P0功能至少1个端到端场景** | 从PRD P0需求提取 |
| **关键用户路径至少1个场景** | 用户最常用的操作路径 |

## 五、测试用例生成时机与分层策略

### 5.1 分层生成原则

| 层级 | 生成时机 | 产出内容 | 用户审阅 |
|------|---------|---------|---------|
| **测试场景清单 + 设计思路** | 测试方案生成时（002任务）| 流程ID、流程名称、用户操作步骤、**测试用例设计思路**、验收标准 | ✅ 审阅方案文档 |
| **测试代码** | 测试执行时（004-N任务）| Playwright代码（动态生成）| ❌ 程序自动生成 |

### 5.2 测试用例设计思路格式

**表格新增列**：测试用例设计思路

```markdown
| 流程ID | 流程名称 | 用户操作步骤 | 测试用例设计思路 | 验收标准 |
```

**设计思路内容规范**：

| 内容 | 说明 | 示例 |
|------|------|------|
| **用户操作路径** | 页面导航和交互步骤 | 打开首页 → 点击创建项目 → 输入信息 → 保存 |
| **预期页面变化** | 每步操作后的页面状态 | 点击保存后：显示成功提示 → 跳转到详情页 |
| **数据验证点** | 如何验证数据正确 | 详情页显示项目名称与输入一致 |

**完整示例**：

| 流程ID | 流程名称 | 用户操作步骤 | 测试用例设计思路 | 验收标准 |
|--------|---------|-------------|----------------|---------|
| E2E-F001-001 | 项目创建完整流程 | 打开首页 → 点击创建项目 → 输入信息 → 保存 → 查看详情 | 操作路径：首页 → 创建按钮 → 表单输入 → 保存按钮 → 详情页；预期变化：保存后显示成功提示、跳转详情页；数据验证：详情页项目名称="E2E测试项目"，描述="测试描述" | 全流程无报错，数据正确 |
| E2E-ERR-001 | API错误处理流程 | 打开首页 → 触发API 500错误 | 操作路径：首页 → Mock API返回500；预期变化：显示错误提示，页面不崩溃；数据验证：错误提示内容="加载失败" | 显示错误提示，不崩溃 |

### 5.3 测试代码动态生成

**004-N任务执行流程**：

```
读取测试方案文档
    │
    ├─ 解析业务流程场景清单
    │     ├─ 提取流程ID
    │     ├─ 提取用户操作步骤
    │     ├─ 提取测试用例设计思路
    │     └─ 提取验收标准
    │
    ├─ AI生成Playwright代码
    │     ├─ 根据设计思路生成page.goto()和page.click()
    │     ├─ 根据用户操作步骤生成测试流程
    │     └─ 根据验收标准生成expect断言
    │
    ├─ 第一层：无头执行
    │     └─ 快速验证流程连通性
    │
    └─ 第二层：有头执行
          └─ 验证用户交互和视觉细节
```

**测试代码文件命名规则**：

```
tests/L3/e2e_{流程类型}-{流程编号}-{编号}.spec.ts

示例：
tests/L3/e2e_f001_project_flow.spec.ts
tests/L3/e2e_err_error_handling.spec.ts
```

### 5.4 核心价值

| 维度 | 说明 |
|------|------|
| **用户审阅便利** | 测试方案包含设计思路，用户可审阅业务流程覆盖是否完整 |
| **代码灵活调整** | 测试代码根据实际页面实现动态生成，UI变化不影响方案 |
| **文档精简** | 测试方案聚焦流程设计，不包含大量代码实现细节 |
| **可追溯** | 流程ID → 测试代码文件命名对应，便于追溯 |

## 六、验收标准定义

### 6.1 核心指标

| 指标 | 计算方式 | 要求 | Guardian处理 |
|------|---------|------|-------------|
| **Playwright通过率（两层合计）** | (第一层通过 + 第二层通过) / total × 100% | 100% | Block |
| **流程覆盖率** | 已通过流程数 / 方案定义总数 × 100% | 100% | Block |
| **页面响应时间** | 平均首次加载时间 | <2s | Warning |

**强制要求**：两层测试全部通过才算验收通过。

### 6.2 Playwright报告格式（两层）

**第一层报告（L3_layer1.json）**：

```json
{
  "layer": 1,
  "summary": {
    "total": 10,
    "passed": 10,
    "failed": 0,
    "duration": 45.5
  },
  "flows": [
    {
      "flow_id": "E2E-F001-001",
      "status": "passed",
      "steps": 5,
      "duration": 8.2
    },
    ...
  ]
}
```

**第二层报告（L3_layer2.json）**：

```json
{
  "layer": 2,
  "summary": {
    "total": 10,
    "passed": 10,
    "failed": 0,
    "duration": 65.3
  },
  "flows": [
    {
      "flow_id": "E2E-F001-001",
      "status": "passed",
      "steps": 5,
      "duration": 12.5,
      "visual_check": "通过"
    },
    ...
  ]
}
```

**验收判断**：
- 第一层通过率 = 100% → 执行第二层
- 第二层通过率 = 100% → 验收通过
- 任一层失败 → 验收失败

import { test, expect } from '@playwright/test';

test.describe('E2E-F001: 项目管理流程', () => {

  });
