## 一、概述

本文档是FE-L2测试方案生成的**执行规范**，定义：

- 测试方案文档结构
- 测试场景划分规则（页面流程、组件交互）
- 测试数据准备策略（Mock API）
- L2 Guardian检查项
- 验收标准定义

> **核心定位**：FE-L2验证**前端组件集成与页面流程正确性**，依赖后端API可用。

## 二、BE-L2 vs FE-L2对比

### 2.1 测试差异对比

| 维度 | BE-L2 | FE-L2 |
|------|-------|-------|
| **测试框架** | pytest | vitest |
| **测试对象** | API集成、数据流转 | 页面流程、组件交互 |
| **依赖关系** | 独立（后端服务）| 依赖后端API可用 |
| **Mock策略** | Mock外部服务 | Mock后端API响应 |
| **覆盖率计算** | pytest覆盖率 | vitest覆盖率 |
| **响应时间** | API响应 <500ms | 页面渲染 <1s |

### 2.2 执行顺序依赖

```
BE-L2 completed → FE-DEV completed → FE-L2启动
```

**前置条件**：后端API已通过BE-L2验证，前端页面已开发完成。

## 四、测试场景划分规则

### 4.1 场景提取来源

| 输入文档 | 提取方法 | 产出场景类型 |
|---------|---------|-------------|
| **UI设计文档** | 页面清单→页面流程 | 页面流程场景 |
| **组件规范文档** | 组件交互定义 | 组件交互场景 |
| **API设计文档** | 错误响应定义 | 异常流程场景 |
| **业务流程文档** | 用户操作流程 | 页面流程场景 |

### 4.2 场景命名规范

```
{场景类型}-{对象类型}-{编号}-{子编号}

场景类型：
  S-FLOW     页面流程场景
  S-INTERACT 组件交互场景
  S-ERROR    异常流程场景
  S-EDGE     边界场景

对象类型：
  P-XXX      页面（P-001项目管理）
  C-XXX      组件（C-001项目列表）

示例：
  S-FLOW-P001-001      页面流程-项目管理-001
  S-INTERACT-C002-001  组件交互-计划树-001
  S-ERROR-P002-001     异常流程-项目详情-001
```

### 4.3 页面流程表示法

| 表示方式 | 说明 | 示例 |
|---------|------|------|
| 单页面渲染 | 页面加载完成 | P-001渲染 |
| 页面跳转 | 页面间导航 | P-001 → P-002 |
| 组件交互 | 组件触发动作 | C-001筛选 → 列表更新 |
| API依赖 | 页面依赖的API | API-002(列表) |

### 4.4 场景最小覆盖规则

| 规则 | 说明 |
|------|------|
| **每个页面至少1个流程场景** | 页面加载渲染验证 |
| **每个页面跳转至少1个场景** | 页面间导航验证 |
| **每个组件交互至少1个场景** | 组件事件响应验证 |
| **每个API错误响应至少1个异常场景** | 错误处理验证 |

## 五、测试用例生成时机与分层策略

### 5.1 分层生成原则

| 层级 | 生成时机 | 产出内容 | 用户审阅 |
|------|---------|---------|---------|
| **测试场景清单 + 设计思路** | 测试方案生成时（002任务）| 场景ID、名称、页面流程、**测试用例设计思路**、验收标准 | ✅ 审阅方案文档 |
| **测试代码** | 测试执行时（004-N任务）| vitest代码（动态生成）| ❌ 程序自动生成 |

### 5.2 测试用例设计思路格式

**表格新增列**：测试用例设计思路

```markdown
| 场景ID | 场景名称 | 页面流程 | 测试用例设计思路 | 验收标准 |
```

**设计思路内容规范**：

| 内容 | 说明 | 示例 |
|------|------|------|
| **用户操作** | 页面交互步骤 | 点击"创建项目"按钮 → 输入名称 → 点击保存 |
| **预期页面状态** | 页面显示变化 | 显示成功提示 → 跳转到详情页 |
| **验证步骤** | 如何验证结果正确 | 检查详情页项目名称正确 |

**完整示例**：

| 场景ID | 场景名称 | 页面流程 | 测试用例设计思路 | 验收标准 |
|--------|---------|---------|----------------|---------|
| S-FLOW-P001-001 | 项目列表加载 | P-001渲染完成 | 操作：打开项目管理页面；预期：列表显示测试项目数据，加载时间<1s；验证步骤：检查列表项数量≥3，检查项目名称显示正确 | 列表正确显示，数据完整 |
| S-INTERACT-C001-001 | 项目列表筛选 | C-001筛选条件变化 → 列表更新 | 操作：输入筛选条件"name=测试"，点击搜索；预期：列表只显示匹配项目；验证步骤：检查列表项数量减少，检查所有项目名称包含"测试" | 筛选生效，列表正确 |
| S-ERROR-P001-001 | API失败显示 | P-001加载 → API返回500 | 操作：打开项目管理页面（Mock API返回500）；预期：显示错误提示"加载失败"，页面不崩溃；验证步骤：检查错误提示可见，检查页面状态正常 | 显示错误提示 |

### 5.3 测试代码动态生成

**004-N任务执行流程**：

```
读取测试方案文档
    │
    ├─ 解析场景清单表格
    │     ├─ 提取场景ID
    │     ├─ 提取页面流程
    │     ├─ 提取测试用例设计思路
    │     └─ 提取验收标准
    │
    ├─ AI生成vitest代码
    │     ├─ 根据设计思路生成测试步骤
    │     ├─ 根据页面流程生成组件操作
    │     └─ 根据验收标准生成expect断言
    │
    └─ vitest执行验证
```

**测试代码文件命名规则**：

```
tests/L2/test_S-{场景类型}-{对象}-{编号}.spec.ts

示例：
tests/L2/test_S-FLOW-P001-001.spec.ts
tests/L2/test_S-INTERACT-C001-001.spec.ts
```

### 5.4 核心价值

| 维度 | 说明 |
|------|------|
| **用户审阅便利** | 测试方案包含设计思路，用户可审阅交互逻辑是否合理 |
| **代码灵活调整** | 测试代码根据实际页面实现动态生成，UI变化不影响方案 |
| **文档精简** | 测试方案聚焦场景设计，不包含大量代码实现细节 |
| **可追溯** | 场景ID → 测试代码文件命名对应，便于追溯 |

### 5.1 pre_guardian（测试执行前）

| 检查项ID | 检查内容 | 检查函数 | 失败处理 |
|---------|---------|---------|---------|
| FE-L2-ENV-001 | 前端服务可用 | check_service_running(http://localhost:8081) | wait |
| FE-L2-ENV-002 | 后端API可用 | check_service_running(http://localhost:9001/api/health) | wait |
| FE-L2-FILE-001 | 测试文件存在 | check_file_exists(tests/L2/*.spec.ts) | create_issue |
| FE-L2-MOCK-001 | Mock数据文件存在 | check_file_exists(tests/mocks/apiMocks.ts) | create_issue |
| FE-L2-PLAN-001 | 测试方案文档存在 | check_file_exists(docs/test-plans/{plan_id}-FE-L2-test-plan.md) | create_issue |

### 5.2 post_guardian（测试执行后）

| 检查项ID | 检查内容 | 检查函数 | 阈值 | 失败处理 |
|---------|---------|---------|------|---------|
| FE-L2-REPORT-001 | vitest报告存在 | check_file_exists(tests/reports/FE_L2_report.json) | - | create_issue |
| FE-L2-PASS-001 | vitest通过率 | check_vitest_pass_rate(report.json) | **100%** | Block → create_issue |
| FE-L2-SCENE-001 | 场景覆盖率 | check_scene_coverage(report.json, test-plan.md) | **100%** | Block → create_issue |

### 5.3 final_guardian（最终验收）

| 检查项ID | 检查内容 | 检查函数 | 失败处理 |
|---------|---------|---------|---------|
| FE-L2-DOC-001 | 测试报告文档存在 | check_file_exists(docs/test-reports/{plan_id}-frontend-L2-report.md) | create_issue |
| FE-L2-CONCLUSION-001 | 验收结论通过 | check_report_conclusion(report.md) | create_issue |

## 六、验收标准定义

### 6.1 核心指标

| 指标 | 计算方式 | 要求 | Guardian处理 |
|------|---------|------|-------------|
| **vitest通过率** | passed_count / total_count × 100% | 100% | Block |
| **场景覆盖率** | 已通过场景数 / 方案定义总数 × 100% | 100% | Block |
| **页面渲染时间** | 平均渲染时间 | <1s | Warning |

    ...
  ],

    "scene_coverage": "100%",

  }

## 七、测试数据准备与隔离策略

### 7.1 数据隔离原则

| 场景 | 是否需要清空 | 处理方式 |
|------|-------------|---------|
| **同一测试用例多次执行** | ✅ vitest自带隔离 | 每个test函数独立上下文 |
| **不同测试场景之间** | ❌ 不清空（依赖后端隔离）| 后端使用不同数据ID |
| **全局清空时机** | 仅FE-L2启动时一次 | 003任务配合后端初始化 |

**核心策略**：FE-L2依赖后端API，后端已按场景分配不同数据ID，前端无需额外清空。

### 7.2 Mock vs 真实API策略

| 策略 | 适用场景 | 优先级 |
|------|---------|--------|
| **真实API调用** | 正向流程、BE-L2已验证的API | **优先**（数据隔离在后端实现）|
| **Mock错误响应** | 异常流程、API返回错误 | 必要 |
| **Mock边界数据** | 边界场景、空数据/大数据 | 必要 |

### 7.3 场景级数据配置

```typescript
// tests/fixtures/L2_frontend_data.ts

// 场景A前端测试数据（对应后端场景A）
export const SCENE_A_CONFIG = {
  projectId: 1,        // 后端场景A项目ID
  projectName: "场景A-测试项目",
}

// 场景B前端测试数据（对应后端场景B）
export const SCENE_B_CONFIG = {
  projectId: 2,        // 后端场景B项目ID
  projectName: "场景B-测试项目",
}

// 场景C前端测试数据（对应后端场景C）
export const SCENE_C_CONFIG = {
  projectId: 3,
  planId: 103,
  taskId: 201,
}
```

### 7.4 vitest场景级Fixture（自动隔离）

```typescript
// tests/L2/test_scene_a.spec.ts

import { SCENE_A_CONFIG } from '../fixtures/L2_frontend_data'

describe('场景A：项目管理流程', () => {
  // 使用场景A专用配置（vitest自动隔离）
  const projectId = SCENE_A_CONFIG.projectId
  
  it('项目列表加载', async () => {
    // 调用真实API，后端返回场景A数据（ID=1）
    const res = await fetch(`/api/projects/${projectId}`)
    expect(res.status).toBe(200)
  })
  
  it('项目详情查看', async () => {
    // 同一场景内多次测试，vitest自动隔离
    const res = await fetch(`/api/projects/${projectId}/detail`)
    expect(res.data.name).toBe(SCENE_A_CONFIG.projectName)
  })
})
```

### 7.5 Mock文件结构（仅异常/边界场景）

```typescript
// tests/mocks/apiMocks.ts

import { vi } from 'vitest'

// 真实API调用配置（优先使用）
export const realApiConfig = {
  baseUrl: 'http://localhost:9001'
}

// 错误响应Mock（仅异常场景）
export const mockApiError = (statusCode: number) => {
  return vi.fn().mockResolvedValue({
    status: statusCode,
    data: { error: `API错误 ${statusCode}` }
  })
}

// 空数据Mock（仅边界场景）
export const mockEmptyData = () => {
  return vi.fn().mockResolvedValue({
    status: 200,
    data: []
  })
}

// 大数据量Mock（仅边界场景）
export const mockLargeData = (count: number) => {
  const data = Array.from({ length: count }, (_, i) => ({
    id: i + 1,
    name: `边界测试项目${i + 1}`
  }))
  return vi.fn().mockResolvedValue({
    status: 200,
    data
  })
}
```

## 八、与现有框架对接

### 8.1 任务单映射

| 任务编号 | 变体编码 | 生成依据 |
|---------|---------|---------|
| 001 | DECODE-FE-L2-001 | FlowTicket启动时自动创建 |
| 002 | DOC-FE-L2-PLAN | 本规范定义的文档结构 |
| 003 | BUILD-FE-L2-ENV | 测试环境章节定义 |
| 004-N | BUILD-FE-L2-TEST | 测试场景清单（动态生成）|
| 005-N+1 | VERIFY-FE-L2 | vitest执行（动态执行）|
| N | DOC-FE-L2-REPORT | 测试报告 |

### 8.2 执行流程

```
FE-L2计划启动（前置：BE-L2 completed）
    │
    ├─ 001: 拆解元任务（读取测试方案，生成任务单）
    │
    ├─ 002: 生成测试方案文档
    │     输入：UI设计 + 组件规范 + API设计
    │     输出：docs/test-plans/{plan_id}-FE-L2-test-plan.md
    │     审核：18-测试方案审核Checklist
    │
    ├─ 003: 测试环境准备
    │     输入：测试环境章节
    │     输出：Mock数据文件 + vitest配置
    │     Guardian：pre_guardian检查环境可用
    │
    ├─ 004+: 测试场景循环执行
    │     ├─ 生成测试代码（BUILD-FE-L2-TEST）
    │     ├─ 执行vitest（VERIFY-FE-L2）
    │     └─ Guardian：post_guardian检查通过率100%
    │
    └─ N: 生成测试报告
          输出：docs/test-reports/{plan_id}-frontend-L2-report.md
          Guardian：final_guardian验收
```
