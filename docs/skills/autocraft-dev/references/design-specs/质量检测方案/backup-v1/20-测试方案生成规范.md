## 一、规范层级与定位

### 1.1 三层文档架构

AutoCraft测试方案采用三层文档架构：

| 层级 | 文档类型 | 说明 | 示例 |
|------|---------|------|------|
| **设计层** | 架构设计文档 | 定义测试架构、工具栈、检测规则 | 项目整体测试方案.md |
| **指导层** | 方法论指导文件 | 定义测试方案应该怎么生成 | 本规范（20）|
| **执行层** | 操作执行规范 | 定义具体执行标准和流程 | 21/22/23/24/25规范 |

**文档关系**：
```
设计层（架构设计）
    ↓ 指导
指导层（方法论）
    ↓ 引用
执行层（操作手册）
    ↓ 执行
测试方案文档（产出物）
```

### 1.2 本规范定位

本规范是**指导性文件**，回答"测试方案应该怎么生成"的问题：
- 定义测试方案层级（L1/L2/L3）
- 定义总纲先行流程（方法论）
- 定义各层方案生成指导原则
- 定义执行规范索引关系

**具体执行标准和流程**见21/22/23/24/25执行规范。

## 二、测试方案层级定义

### 2.1 L1/L2/L3三层测试架构

| 层级 | 测试焦点 | 测试范围 | 执行时机 | 生成方案文档 |
|------|---------|---------|---------|-------------|
| **L1** | 代码质量 | 单任务（100-200行）| BE-DEV/FE-DEV计划 | ✅ 测试方案（测试代码清单）|
| **L2** | API/组件集成 | API串联、页面流程、数据流转 | BE-L2/FE-L2计划 | ✅ 测试方案（场景清单）|
| **L3** | 业务流程E2E | 端到端用户操作流程 | L3-E2E计划 | ✅ 测试方案（流程清单）|

**核心原则**：
- **L1生成独立测试方案**：测试代码清单（模块→测试文件→场景数量）
- **L2/L3生成独立方案**：详细场景清单（API串联、页面流程、E2E流程）
- **测试代码生成作为独立任务单（003）**：排在程序代码生成之前

> **核心设计变更（2026-04-15）**：L1测试代码生成不再在pytest闭环内部动态生成，而是作为独立任务单（003），实现测试先行（TDD思想）。

### 2.2 L1测试方案执行规范位置

L1测试方案执行规范在**30/31规范**，包含：
- L1测试方案文档结构（测试代码清单）
- pytest/vitest闭环机制（执行位置、修复次数上限）
- 验收标准（通过率、覆盖率阈值）
- Guardian检查项（E4-E7）
- 任务单映射（002方案 → 003测试代码 → 004程序代码）
- execution_log.quality_check结构

**L1任务顺序（测试先行）**：

```
BE-DEV/FE-DEV计划：
    ├─ 002: 生成L1测试方案（测试代码清单）← 用户审阅
    ├─ 003: 生成测试代码（独立任务单）← 测试先行
    └─ 004: 生成程序代码 + pytest/vitest闭环 ← pytest验证测试代码
```

> **关键设计**：测试代码生成（003）是独立任务单，排在程序代码生成（004）之前。

## 三、总纲先行流程（核心方法论）

### 3.1 总纲先行原则

**核心理念**：先生成总纲供用户审阅确认覆盖完整，再分层生成L2/L3具体方案。

```
设计文档 → 测试方案总纲（AI生成）→ 用户审阅确认覆盖完整
              ↓
        分层生成 BE-L2 → FE-L2 → L3（按总纲指引）
              ↓
        开发计划（引用总纲+L1规范+L2/L3方案）
```

### 3.2 总纲价值

| 价值维度 | 说明 |
|---------|------|
| **用户审阅入口** | 总纲精简版（~200行），用户快速确认覆盖完整 |
| **L1规范明确** | L1测试执行规范在总纲中明确，开发计划制定有依据 |
| **验收标准前置** | 用户审阅总纲时确认三层验收标准 |
| **任务拆解依据** | 明确哪些任务需要拆解测试任务单、哪些内部闭环 |
| **开发计划输入** | 开发计划引用总纲+L1规范+L2/L3方案 |

### 3.3 总纲与具体方案的关系

| 文档 | 长度 | 内容 | 用户审阅 |
|------|------|------|---------|
| **总纲** | ~200行 | L1/L2/L3三层规范摘要 + 场景覆盖矩阵 + 验收标准 + L1测试执行规范 | ✅ 审阅确认覆盖完整 |
| **BE-L2方案** | ~500行 | 具体测试场景清单 + 测试数据准备 | ✅ 审阅具体场景 |
| **FE-L2方案** | ~400行 | 具体测试场景清单 + 测试数据准备 | ✅ 审阅具体场景 |
| **L3方案** | ~300行 | 具体业务流程场景清单 + 测试数据状态 | ✅ 审阅具体场景 |

### 3.4 开发计划制定引用

开发计划制定时引用：

| 引用内容 | 规范位置 | 用途 |
|---------|---------|------|
| 总纲 | 21规范第2-4章 | 验收标准前置、任务拆解依据 |
| L1测试规范 | 21规范第5章 | pytest闭环机制、Guardian检查项 |
| L2/L3测试方案 | 22/23/24规范 | 具体场景清单 |

## 四、L2/L3方案生成指导原则

### 4.1 文档结构指导

L2/L3方案统一采用**5章节结构**：

| 章节 | 内容 | 数据来源 |
|------|------|---------|
| **1. 测试范围** | 测试对象清单、不测试内容 | API设计/UI设计/业务流程文档 |
| **2. 测试环境** | 数据库、服务、Mock策略 | 技术方案文档 |
| **3. 测试场景清单** | 场景ID、名称、流程、设计思路、验收标准 | 业务流程 + 设计文档 |
| **4. 测试数据准备** | Fixture文件、数据初始化脚本、数据隔离策略 | 数据库设计文档 |
| **5. 验收标准** | 通过率、覆盖率、Guardian处理方式 | 项目整体测试方案 |

**详细执行标准**见22/23/24规范。

### 4.2 测试场景划分指导

| 场景类型 | 场景代码 | 提取来源 |
|---------|---------|---------|
| 正向业务流程 | S-NORMAL | 业务流程文档 |
| 异常业务流程 | S-ERROR | API错误响应定义 |
| 边界场景 | S-EDGE | 数据约束、功能边界 |

**最小覆盖规则**：
- L2：每API≥1场景、每业务流程≥1正向、每API错误≥1异常
- L3：每核心流程≥1场景、每P0功能≥1端到端

**详细执行标准**见22/23/24规范。

### 4.3 文档长度阈值指导

| 层级 | GLM-5舒适区 | 警告阈值 | 处理策略 |
|------|------------|---------|---------|
| L2方案 | <1000行 | 1000-1500行 | 场景清单分模块拆分 |
| L3方案 | <800行 | 800-1200行 | 流程清单分模块拆分 |

## 五、测试方案审核指导原则

### 5.1 三层审核流程

测试方案审核采用**三层审核流程**，确保格式完整 + 内容质量 + 跨文档一致性：

```
第一层：格式审核（26/27/28 Checklist E系列）
    ├─ 文档结构完整性（E01-E05）
    ├─ 测试范围完整性（E06-E10）
    ├─ 场景清单完整性（E11-E20）
    ├─ 验收标准完整性（E21-E25）
    └─ 数据准备完整性（E26-E30）
    └─ 通过 → 进入第二层

第二层：内容审核（26/27/28 Checklist C系列）
    ├─ 业务流程模拟验证（对照设计文档）
    ├─ API/页面/流程覆盖验证
    ├─ 数据边界验证
    ├─ PRD功能覆盖验证
    └─ 异常分支深度验证
    └─ 通过 → 进入第三层

第三层：整体性审核（29 Checklist）
    ├─ 上下文一致性（总纲与各方案数据匹配）
    ├─ 逻辑冲突检查（验收标准/流程无冲突）
    ├─ 重复冗余检查（各方案无重复描述）
    ├─ 引用完整性检查（总纲引用正确）
    └─ 数据依赖一致性（L1→L2→L3依赖一致）
    └─ 通过 → 整体定稿
```

### 5.2 审核评分规则

| 评分等级 | 分数范围 | 结论 |
|---------|---------|------|
| **通过** | ≥90分 | 定稿，可进入下一层审核 |
| **需修改** | 70-89分 | 修复阻断项+内容审核缺失项后重新审核 |
| **不通过** | <70分 | 重大缺失，重新生成 |

**评分计算**：
```
总分 = 100 - (阻断项缺失数 × 10) - (内容审核缺失数 × 5) - (警告项缺失数 × 2)
```

### 5.3 检查项分类

| 检查项类型 | 说明 | 扣分 |
|---------|------|------|
| **阻断项（E系列）** | 格式完整性，缺失则无法进入测试执行 | 每项10分 |
| **内容审核（C系列）** | 业务流程模拟验证，缺失影响测试质量 | 每项5分 |
| **警告项（W系列）** | 细节完整性，缺失可能影响测试效率 | 每项2分 |
| **整体性（X/L/R/Y/D系列）** | 跨文档一致性，缺失影响体系完整 | 每项5-10分 |

**详细执行标准**见25/26/27/28/29审核Checklist。

### 5.4 内容审核对照文档

内容审核（C系列）需要对照设计文档进行业务流程模拟验证：

| 审核对象 | 对照文档 | 提取内容 |
|---------|---------|---------|
| **BE-L2** | API设计文档 | P0 API清单、错误响应、参数约束 |
| **BE-L2** | 业务流程设计文档 | 流程节点、分支、异常中断 |
| **BE-L2** | 数据库设计文档 | 字段约束、外键、唯一键 |
| **BE-L2** | PRD | P0功能、验收标准、业务规则 |
| **FE-L2** | UI设计文档 | P0页面清单、页面跳转、表单字段 |
| **FE-L2** | 组件规范文档 | 公共组件、表单验证、按钮状态 |
| **FE-L2** | PRD | P0功能、验收标准、用户交互 |
| **L3** | 业务流程设计文档 | P0流程清单、分支、异常节点 |
| **L3** | PRD | P0功能、验收标准、用户故事 |
