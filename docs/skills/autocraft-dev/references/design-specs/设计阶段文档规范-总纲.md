## 二、文档边界定义

| 文档 | 包含内容 | 不包含内容 |
|------|---------|-----------|
| **PRD** | 业务功能需求（目标+范围）、用户故事、验收标准 | 详细功能设计、数据库结构、API定义 |
| **系统功能设计** | 详细功能清单、公共功能、技术功能、功能追溯矩阵 | 业务需求描述、用户故事 |
| **技术方案文档** | 技术选型、架构设计、目录结构、部署方案 | 业务逻辑、接口细节 |
| **API设计文档** | 接口路径、请求参数、响应格式、错误码 | 业务规则细节、数据库表结构 |
| **数据库设计文档** | 表结构、字段定义、索引、外键、数据字典 | 业务逻辑、接口定义 |
| **业务流程文档** | 用户操作流程、角色职责、异常处理、业务规则 | 技术实现、数据存储细节 |
| **UI设计文档** | 页面结构、组件定义、交互流程、状态说明 | 业务逻辑、数据来源 |
| **组件规范文档** | 命名规范、样式规范、Props定义 | 具体页面内容 |

## 六、整体性验证阶段

> ⚠️ **新增（v3.4）**：全部文档定稿后，必须执行整体性验证，验证跨文档一致性。

### 6.1 验证时机与必要性

| 验证时机 | 说明 |
|---------|------|
| 全部文档定稿后 | 8个文档状态均为"已定稿"或"已批准" |
| 开发实施前 | 整体性验证通过后才能进入开发阶段 |

**必要性说明**：

单文档审核只检查"文档有没有"，无法发现"能不能用"。整体性验证弥补以下缺失：

| 发现问题类型 | 单文档审核 | 整体性验证 |
|-------------|-----------|-----------|
| API字段缺失 | ❌ 无法发现 | ✅ 对照组件Props可发现 |
| 流程断裂 | ❌ 无法发现 | ✅ 模拟流程步骤可发现 |
| 字段定义不一致 | ❌ 无法发现 | ✅ 跨文档搜索可发现 |
| 外键缺失 | ❌ 无法发现 | ✅ ER图分析可发现 |

### 6.2 验证维度

| 验证维度 | 检查项数 | 核心问题 |
|---------|---------|---------|
| 功能追溯链（G01-G06） | 6项 | FR→功能→API→数据库是否完整追溯？ |
| API→数据库映射（G07-G26） | 20项 | API字段与数据库字段是否一致？ |
| UI→API绑定（G27-G30） | 4项 | 组件Props能否接收API数据？ |
| 业务流程可执行性（G31-G32） | 2项 | 流程能否实际执行？ |
| 数据模型完整性（G33-G36） | 4项 | 外键链是否完整？ |
| 新增字段一致性（G37-G41） | 5项 | 同一字段各文档定义是否一致？ |

**合计42项检查**。

### 6.3 验证流程

```
第一步：检查准备条件
    ├─ 8个文档全部定稿？
    ├─ 单文档审核报告存在？
    └─ 文档版本一致？
    
第二步：执行6个维度验证
    ├─ 功能追溯链验证（对照PRD→功能→API→数据库）
    ├─ API→数据库映射验证（逐字段对照）
    ├─ UI→API绑定验证（对照组件Props与API响应）
    ├─ 业务流程可执行性验证（模拟流程步骤）
    ├─ 数据模型完整性验证（检查ER图外键链）
    └─ 新增字段一致性验证（跨文档搜索）
    
第三步：发现问题处理
    ├─ 有阻断项不通过 → 必须修复后重新验证
    └─ 警告项不通过 → 建议修复，不影响通过
    
第四步：生成验证报告
    ├─ 输出42项检查结果
    ├─ 发现问题详情（类型、影响、修复建议）
    └─ 验证结论（通过/不通过）
```

### 6.4 验证方法

| 方法 | 工具/步骤 | 发现问题类型 |
|------|---------|-------------|
| 文档对照阅读 | 逐章节读取、建立映射表 | 结构缺失、追溯断裂 |
| 关键字段搜索 | grep跨文档搜索字段名 | 定义不一致、字段缺失 |
| 流程模拟推演 | 手动推演业务流程步骤 | 流程断裂、接口缺失 |
| ER图分析 | 检查外键链和级联规则 | 外键缺失、级联错误 |
| 组件Props对照 | 对照组件Props与API响应 | Props无法绑定API数据 |

### 6.5 验证报告输出

验证报告必须包含：

| 字段 | 说明 |
|------|------|
| **验证维度汇总表** | 6个维度检查结果（通过数/不通过数） |
| **发现问题详情** | 每个问题的类型、影响范围、修复建议 |
| **修复记录** | 修复过程和重新验证结果 |
| **验证结论** | 通过/不通过 + 处理建议 |

### 6.6 整体性验证通过标准

```
✅ 42项全部通过 → 整体性验证通过，进入开发实施
❌ 任一阻断项不通过 → 必须修复后重新验证
⚠️ 警告项不通过 → 可通过，建议修复
```

## 七、文档生成流程

### 7.1 工具选择

| 场景 | 推荐工具 | 原因 |
|------|---------|------|
| 文档生成 | **Crush + GLM-5** | 稳定、快速（30秒）、无长度限制 |
| 文档审核 | **GLM-5** | 发现问题更全面 |
| 文档修复 | **Crush** | 直接修改文件，支持大文件 |
| 最终把关 | DeepSeek（可选） | 仅关键文档使用，控制成本 |

**为什么不用子代理？**

| 对比项 | Crush | 子代理 |
|--------|-------|--------|
| 稳定性 | ✅ 高（100%成功） | ❌ 低（超时/截断） |
| 速度 | ✅ 快（30秒） | ❌ 慢（>3分钟） |
| 长文档支持 | ✅ 无限制 | ❌ 有限制 |

### 7.2 标准流程

```
第一步：准备输入材料
    ↓
第二步：Crush + GLM-5 生成第一版
    ↓
第三步：GLM-5 审核
    ↓
第四步：根据问题清单修复
    ↓
第五步：GLM-5 再审核确认
    ↓
第六步：定稿（可选 DeepSeek 最终把关）
```

### 6.5 审核Checklist使用指引

#### 6.5.1 Checklist定位

审核Checklist是**AI审核节点**使用的检查指导，用于：

| 用途 | 说明 |
|------|------|
| **审核指导** | 明确阻断项和警告项，指导AI进行文档审核 |
| **质量门禁** | 阻断项不通过则禁止定稿 |
| **问题分级** | 区分严重/中等/轻微问题，明确修复优先级 |

#### 6.5.2 PRD审核Checklist核心内容

**核心原则（三大禁止+三大必须）**：

| 类型 | 编号 | 内容 |
|------|------|------|
| 🚫 禁止 | B01 | 禁止凭空添加功能（必须有REQ来源） |
| 🚫 禁止 | B02 | 禁止在PRD中添加技术实现细节 |
| 🚫 禁止 | B03 | 禁止根据"常识"发挥 |
| ✅ 必须 | M01 | 功能必须有出处（REQ→FR） |

#### 6.5.3 审核流程

```
第一步：审核前准备
    ├─ 预处理清单审核（R01-R06）

    ├─ 80-89分 → 修复后定稿

## 十、迭代设计原则

### 10.1 为什么需要迭代设计？

**传统开发 vs AI编程**：

| 维度 | 传统开发 | AI编程 |
|------|---------|--------|
| P0限制原因 | 资源约束、时间有限 | 复杂度控制、降低出错概率 |
| 需求变更代价 | 人力重做 | AI重新理解、重新生成 |

**AI编程怕改需求**：每次变更都要重新理解全文，已有代码可能作废。

**迭代开发 ≠ 改需求**：
- 迭代开发：有计划的增量，每个迭代内需求稳定
- 改需求：需求定义后还变，推倒重来

### 10.2 核心原则

| 原则 | 说明 |
|------|------|
| **架构一次性规划** | 避免重构，前瞻性设计 |
| **功能分阶段设计** | 降低设计成本，灵活迭代 |
| **迭代内需求冻结** | 每个迭代周期内需求不变 |

### 10.3 设计文档分层

```
设计文档分层结构：
├─ L1：整体架构设计（一次性规划）
│   ├─ 系统架构
│   ├─ 技术选型
│   ├─ 核心数据模型
│   └─ 扩展预留点
│
├─ L2：核心功能详细设计（第一迭代）
│   ├─ 核心API
│   ├─ 核心表结构
│   └─ 核心流程
│
└─ L3：扩展功能详细设计（后续迭代）
    ├─ 扩展API
    ├─ 扩展表结构
    └─ 扩展流程
```

### 10.4 各文档类型分层规则

#### PRD 分层

| 章节 | 规划方式 |
|------|---------|
| 产品愿景 | 一次性写完 |
| 核心功能（P0） | 详细描述 |
| 扩展功能（P1/P2） | 简要描述 + 预留 |
| 迭代计划 | 明确阶段划分 |

#### 系统功能设计分层

| 功能类型 | 规划方式 |
|---------|---------|
| 系统架构 | ✅ 一次性规划 |
| 核心模块 | ✅ 详细设计 |
| 扩展模块 | ⚠️ 接口预留 |

#### API设计分层

| API类型 | 规划方式 |
|---------|---------|
| 核心API | ✅ 完整设计（第一迭代） |
| 扩展API | ⚠️ 接口定义 + TODO标记 |
| 预留API | ⚠️ 仅定义接口签名 |

#### 数据库设计分层

| 表类型 | 规划方式 |
|--------|---------|
| 核心表 | ✅ 完整设计 |
| 扩展字段 | ⚠️ 预留字段（如 `extra_data` JSON） |
| 扩展表 | ⚠️ 仅定义表名 + 核心字段 |

### 10.5 标记规则

| 标记 | 含义 | 示例 |
|------|------|------|
| `[核心]` | 第一迭代必须完成 | `[核心] 用户管理模块` |
| `[扩展]` | 后续迭代完成 | `[扩展] VIP功能` |
| `[预留]` | 接口已定义，实现待定 | `[预留] 第三方登录接口` |
| `[TODO]` | 设计待补充 | `[TODO] 详细表结构` |

### 10.6 版本与迭代对应

```
设计文档版本号与迭代对应：

PRD-v4.2.0 → 第一迭代设计（P0功能完整）
PRD-v4.2.1 → 第二迭代补充（P1功能补充）
PRD-v4.2.2 → 第三迭代补充（P2功能补充）
```

### 10.7 对项目管理流程的影响

| 项目管理环节 | 影响 | 调整建议 |
|-------------|------|---------|
| **工作计划创建** | 按迭代分阶段创建，而非一次性创建全部 | 先创建第一迭代计划，后续迭代按需创建 |
| **任务单拆解** | 只拆解当前迭代的功能，不拆解全部功能 | 避免一次性拆解过多任务单 |
| **里程碑定义** | 每个迭代对应一个里程碑 | 明确里程碑验收标准 |
| **需求变更管理** | 迭代内冻结，迭代间可调整 | 建立需求变更评审机制 |
| **进度跟踪** | 按迭代跟踪进度 | 重点关注当前迭代完成度 |

### 10.8 迭代周期建议

| P0数量 | 建议迭代周期 | 理由 |
|--------|-------------|------|
| ≤3个 | 1周 | 快速验证核心流程 |
| 4-5个 | 1-2周 | 适度复杂度 |
| >5个 | ⚠️ 需拆分 | 复杂度过高 |

### 10.9 分阶段设计策略汇总

#### 核心原则

| 原则 | 说明 |

| **骨架整体规划** | 业务流程、系统功能清单必须完整，定义系统边界 |
| **清单完整覆盖** | 所有文档的清单（功能、接口、表、页面）必须完整，满足追溯矩阵100%覆盖 |
| **详情分阶段细化** | P0详细设计，P1/P2框架设计或占位符 |

#### 分阶段策略汇总表

| 设计阶段 | 清单完整性 | P0详情 | P1详情 | P2详情 | 说明 |
|---------|-----------|--------|--------|--------|------|

### 10.10 文档长度安全阈值与拆分策略

#### AI生成能力约束

| 约束项 | 阈值 | 说明 |
|--------|------|------|
| **GLM-5舒适区** | < 1000行 | 生成和审核质量稳定 |
| **警告阈值** | 1000-1500行 | 质量可能下降，建议拆分 |
| **截断风险** | > 2000行 | 系统截断限制（2000行或50KB） |
| **审核风险** | > 1500行 | 中间遗忘，检查不细致 |

#### 文档拆分决策规则

| 阶段 | 单文档条件 | 拆分条件 | 单文档阈值 | 拆分依据 |
|------|-----------|---------|-----------|---------|
| **PRD** | ✅ 始终单文档 | - | 无限制 | 功能需求集中，不建议拆分 |
| **业务流程设计** | 流程≤5个 | 流程>5个 | < 800行 | 按业务模块拆分 |
| **系统功能设计** | FR≤3个 | FR≥4个 | < 1000行 | 按FR模块拆分 |
| **技术方案** | ✅ 始终单文档 | - | 无限制 | 架构一次性规划原则 |
| **API设计** | 接口≤20个 | 接口>20个 | < 1000行 | 按业务模块拆分 |
| **数据库设计** | 表≤15个 | 表>15个 | < 800行 | 按业务模块拆分 |
| **UI设计** | 页面≤10个 | 页面>10个 | < 1000行 | 按业务模块拆分 |
| **组件规范** | ✅ 始终单文档 | - | 无限制 | 命名规范统一，不建议拆分 |

> ⚠️ **阈值说明**：
> - **单文档阈值**：建议不超过此行数，确保AI生成和审核质量
> - **硬性截断**：2000行或50KB（系统限制），超过必须拆分
> - **GLM-5舒适区**：< 1000行（生成和审核质量最稳定）

#### 拆分决策流程

```
文档预期行数评估
    │
    ├─ < 1000行 → ✅ 单文档
    │
    ├─ 1000-1500行 → ⚠️ 评估拆分必要性
    │     ├─ 已定稿 → 不拆分
    │     ├─ 模块独立性强 → 拆分
    │     └─ 模块关联紧密 → 不拆分
    │
    └─ > 1500行 → ✅ 必须拆分
          └─ 按模块/FR/功能拆分
```

#### 分模块文档结构模板

**适用文档**：系统功能设计、业务流程设计、API设计、UI设计

```
<设计文档目录>/
├── 00-总览.md              # 汇总表 + 索引 + 追溯矩阵汇总
├── 01-<模块1名称>-<ID>.md  # 模块1详细设计
├── 02-<模块2名称>-<ID>.md  # 模块2详细设计
└── ...                     # 其他模块
```

**单模块文档长度要求**：300-500行（GLM-5舒适区）

#### 各文档拆分细则

**业务流程设计**：
- 单文档：流程≤5个，或流程间调用紧密
- 拆分：流程>5个，按业务模块拆分

**系统功能设计**：
- 单文档：FR≤3个
- 拆分：FR≥4个，按FR模块拆分

**API设计**：
- 单文档：接口≤20个
- 拆分：接口>20个，按业务模块拆分

**数据库设计**：
- 单文档：表≤15个
- 拆分：表>15个，按业务模块拆分

**UI设计**：
- 单文档：页面≤10个
- 拆分：页面>10个，按业务模块拆分

#### 总览文档必须包含

| 章节 | 内容 |
|------|------|
| 模块索引表 | 模块ID、名称、文档路径、状态 |
| 清单汇总 | 完整的FR/流程/接口/页面清单 |
| 追溯矩阵汇总 | 所有模块的追溯表合并 |
| 范围约束确认 | 不超出PRD定义范围 |

## 十一、Vue骨架原型设计策略

### 11.1 骨架原型的定位

**为什么需要骨架原型**：

| 问题 | 解决方案 |
|------|---------|
| 用户想快速看到原型 | 骨架原型1-2天即可运行 |
| 原型与正式开发一致性 | Vue骨架代码直接复用 |
| 需要可视化确认页面结构 | 真实页面布局，用户可操作 |

**骨架原型 ≠ 可视化设计工具**：

| 对比项 | OpenPencil/Figma | Vue骨架原型 |
|--------|-----------------|------------|
| 输出 | 设计稿/图片 | 可运行代码 |
| 与开发一致性 | ❌ 需要重建 | ✅ 直接复用 |
| 用户体验 | 看静态图 | 操作真实页面 |
| 适用环境 | 需GUI环境 | 服务器可运行 |

### 11.2 骨架原型包含内容

| 内容 | 说明 |
|------|------|
| 项目骨架 | Vue项目初始化（Vite + Vue3 + Element Plus） |
| 页面布局 | Header + Sidebar + Content布局结构 |
| 路由导航 | 所有页面路由入口（空页面占位） |
| 组件引入 | Element Plus组件库引入 |
| 空页面占位 | 仅页面标题，无业务逻辑 |

**不包含**：业务逻辑、数据交互、完整组件实现

### 11.3 骨架原型执行流程

```
技术方案文档（确定Vue框架）
    ↓
Crush生成Vue骨架原型
    ↓
部署运行（端口：8081）
    ↓
用户确认原型（页面结构、导航布局）
    ↓
UI设计文档 + 组件规范（基于确认的骨架）
    ↓
正式开发（骨架代码复用）
```

### 11.4 骨架原型的优势

| 优势 | 说明 |
|------|------|
| **快速可见** | 1-2天即可让用户看到真实页面 |
| **代码复用** | 骨架代码直接成为正式开发基础 |
| **一致性高** | 原型确认的内容不会推倒重来 |
| **降低返工** | 用户确认后，UI设计基于实际骨架 |

### 11.5 骨架原型与UI设计文档的关系

| 阶段 | 内容 | 关系 |
|------|------|------|
| 骨架原型 | 页面布局 + 路由 + 空页面 | **确定页面结构** |
| 用户确认 | 结构是否满足需求 | **确认结构** |
| UI设计文档 | 详细交互 + 状态说明 | **基于骨架补充细节** |

**骨架原型解决"页面结构"，UI设计文档解决"交互细节"**

### 11.6 骨架原型验收标准

| 检查项 | 标准 |
|--------|------|
| 页面布局 | Header/Sidebar/Content结构正确 |
| 路由导航 | 所有页面路由入口可点击 |
| 空页面 | 每个页面有标题占位 |
| Element Plus | 组件库正确引入 |
| 可运行 | 页面可正常访问 |

## 十二、附录

### A. 验证结论（2026-04-08测试）

**生成方式对比**：

| 方式 | 稳定性 | 速度 | 长文档支持 |
|------|--------|------|-----------|
| Crush + GLM-5 | ✅ 100%成功 | ✅ 30秒 | ✅ 无限制 |
| 子代理 + GLM-5 | ❌ 超时失败 | ❌ >3分钟 | ❌ 截断 |

**审核模型对比**：

| 版本 | GLM-5发现问题 | DeepSeek发现问题 |
|------|-------------|-----------------|
| V1 | 10个 | 8个 |
| V2 | **5个** | 1个 |

**结论**：GLM-5发现问题更全面，DeepSeek仅用于关键文档最终把关。

### B. 成本控制

| 阶段 | 模型 | 调用次数 | 成本 |
|------|------|---------|------|
| 生成 | GLM-5 | 1次 | 低 |
| 初次审核 | GLM-5 | 1次 | 低 |
| 修复 | GLM-5 | 1~3次 | 低 |
| 再审核 | GLM-5 | 1次 | 低 |
| 最终把关 | DeepSeek | 0~1次 | 较高 |
| **合计** | GLM-5: 4~6次 | DeepSeek: 0~1次 | **可控** |
