## 一、规范层级与定位

### 1.1 开发计划文档架构

AutoCraft开发计划采用两层文档架构：

| 层级 | 文档类型 | 说明 | 示例 |
|------|---------|------|------|
| **总览级** | 项目总览文档 | 批次顺序、模块依赖、功能概览 | 开发计划总览.md |
| **计划级** | 工作计划清单 | 单个工作计划的任务单结构概要 | WP-M03-BE-DEV-001.md |

**文档关系**：
```
设计层（功能设计文档）
    ↓ 输入
开发计划总览（项目级）
    ↓ 指引
工作计划清单（计划级）
    ↓ 直接执行
程序代码 + 测试代码（产出物）
```

### 1.2 本规范定位

本规范是**指导性文件**，回答"开发计划应该怎么生成"的问题：
- 定义开发计划层级（总览级 + 计划级）
- 定义预生成与动态生成边界
- 定义一致性保障机制
- 定义执行规范索引关系

**具体执行标准和流程**见41/42/43执行规范。

## 二、核心策略定义

### 2.1 预生成 vs 动态生成边界

| 内容 | 生成时机 | 生成者 | 用户审阅 | 理由 |
|------|---------|--------|---------|------|
| **开发计划总览** | 制定阶段预生成 | AI生成 | ✅ 审阅 | 整体规划、批次协调 |
| **工作计划清单** | 制定阶段预生成 | AI生成 | ✅ 审阅概要表格 | 整体性+一致性保障 |

### 2.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **清单预生成保障整体性** | 编号统一规划、命名规范一致、任务数量明确 |
| **AI作为智能开发者** | 输入设计文档原文而非预定义步骤 |
| **pytest闭环应对不确定性** | 失败→修复→重试，迭代修复是不可预测的 |
| **无拆解任务：清单已明确定义所有任务，直接执行即可** | 清单已明确定义所有任务，无需001拆解步骤 |

### 2.3 AI软件开发特性适配

| AI特性 | 适配方式 |
|--------|---------|
| **不确定性** | pytest闭环迭代修复（最多3次） |
| **灵活性** | 任务单详情动态生成，按最新设计文档 |
| **渐进式** | 工作计划清单提供概要，详情运行时补全 |
| **实验性** | 设计文档可迭代更新版本 |

## 三、开发计划层级定义

### 3.1 两层结构

| 层级 | 名称 | 内容 | 数量 | 用户审阅 |
|------|------|------|------|---------|
| **L1: 总览级** | 开发计划总览 | 批次顺序 + 模块依赖 + L3-E2E节点 + 功能概览 | 1份 | ✅ 审阅 |
| **L2: 计划级** | 工作计划清单 | 基本信息 + 任务单结构表格 + 验收标准 + 编号约束 | 按项目实际 | ✅ 审阅清单（不审阅JSON） |

### 3.2 工作计划数量统计（示例）

> 以下为AutoCraft项目的实际统计示例，各项目按实际模块和计划类型填写。

| 模块 | BE-DEV | BE-L2 | FE-DEV | FE-L2 | 合计 |
|------|--------|-------|--------|-------|------|
| M-01 项目管理 | 1 | 1 | 1 | 1 | 4 |
| M-06 配置管理 | 1 | 1 | 1 | 1 | 4 |
| M-02 工作计划管理 | 1 | 1 | 1 | 1 | 4 |
| M-04 问题管理 | 1 | 1 | 1 | 1 | 4 |
| M-03 FlowTicket执行引擎 | 1 | 1 | 1 | 1 | 4 |
| INFRA | 1 | - | - | - | 1 |
| L3-E2E | - | - | - | - | 4（跨模块） |
| **合计** | 6 | 5 | 5 | 5 | **25** |

> 各项目的模块数量和计划类型组合不同，工作计划总数由实际设计决定。

### 3.3 工作计划ID命名规范

```
WP-{模块编号}-{计划类型}-{序号}

示例：
- WP-M01-BE-DEV-001：M-01模块后端开发计划
- WP-M03-BE-L2-001：M-03模块后端L2测试计划
- WP-L3-E2E-001：L3-E2E增量测试计划
```

## 四、与测试方案体系对齐

### 4.1 引用关系

| 维度 | 测试方案总纲定义 | 开发计划清单引用 |
|------|-----------------|-----------------|
| **L1任务结构** | 001拆解 → 002~N-1功能循环（每功能：测试代码→程序代码）| BE-DEV/FE-DEV任务单按此顺序编号 |
| **L1验收标准** | pytest100% + 覆盖率≥80% + lint无Error | 工作计划清单引用测试方案总纲章节 |
| **L2任务结构** | 001拆解 → 002环境 → 003~N-1场景循环 | BE-L2/FE-L2任务单按此顺序编号 |
| **L2验收标准** | pytest100% + 场景覆盖率100% | 工作计划清单引用测试方案总纲章节 |
| **L3任务结构** | 001拆解 → 002环境 → 003~N-1流程循环（两层）| L3-E2E任务单按此顺序编号 |
| **L3验收标准** | Playwright两层100% + 流程覆盖率100% | 工作计划清单引用测试方案总纲章节 |

> **设计澄清（2026-04-16）**：所有测试方案（L1/L2/L3）在制定阶段预生成，开发计划直接引用，不需要"生成测试方案"任务。

### 4.2 不重复定义原则

| 原则 | 说明 |
|------|------|
| **验收标准不重复定义** | 工作计划清单引用测试方案总纲章节（如"详见测试方案总纲3.1节"） |
| **任务结构不重复定义** | 测试方案总纲已定义L1/L2/L3任务结构，清单标注编号范围 |
| **Guardian/质量检查项不重复定义** | 测试方案总纲已定义检查项，清单引用 |

## 五、一致性保障机制

### 5.1 编号约束机制

工作计划清单提供编号约束，FlowTicket动态生成时必须遵守：

```markdown
## 任务单编号约束（FlowTicket必须遵守）

**BE-DEV/FE-DEV计划**：
| 任务编号范围 | 任务类型 | 验证分类 | 说明 |
|-------------|---------|---------|------|
| 001~N-2 | BUILD | A类 | 功能循环（每功能2个任务：测试代码→程序代码）|
| N-1 | DOC | DOC类 | 开发报告 |

**BE-L2/FE-L2计划**：
| 任务编号范围 | 任务类型 | 验证分类 | 说明 |
|-------------|---------|---------|------|
| 001 | BUILD | B类 | 测试环境准备 |
| 002~N-2 | BUILD | A类 | 场景循环（每场景1个任务）|
| N-1 | DOC | DOC类 | 测试报告 |

**L3-E2E计划**：
| 任务编号范围 | 任务类型 | 验证分类 | 说明 |
|-------------|---------|---------|------|
| 001 | BUILD | B类 | 测试环境准备 |
| 002~N-2 | BUILD | A类 | 流程循环（每流程2个任务：两层测试）|
| N-1 | DOC | DOC类 | E2E报告 |

**约束规则**：
- 动态生成任务单详情时，编号必须在此范围内
- 任务类型、验证分类必须匹配
- 测试方案引用已预生成的文档路径
```

### 5.2 版本一致性检查机制

工作计划清单标注设计文档版本，FlowTicket启动时检查：

```markdown
## 设计文档版本（FlowTicket启动时检查）

| 文档类型 | 当前版本 | 文档路径 |
|---------|---------|---------|
| API设计 | v1.2 | docs/design/API设计-{项目名}-v1.2.md |
| 数据库设计 | v1.0 | docs/design/数据库设计-{项目名}-v1.0.md |
| 测试方案总纲 | v1.2 | docs/test-plans/{项目名}-项目整体测试方案.md |

**版本检查规则**：
- FlowTicket启动时读取设计文档版本号
- 与清单标注版本对比，不一致则创建问题单
```

### 5.3 任务单详情生成约束

```markdown
## 任务单详情生成约束

**必须输入**：
- 设计文档原文（API设计对应章节）
- 测试方案总纲（已预生成，验收标准章节）
- L1/L2/L3测试方案（已预生成，场景/流程清单）

**输出必须包含**：
- task_no（符合编号约束）
- test_plan_reference（引用预生成的测试方案路径）
- verification_checklist（引用测试方案总纲）
```

## 六、生成流程定义

### 6.1 制定阶段生成流程

```
开发计划制定阶段（AI分批生成）：
    │
    ├─ 第1批：INFRA + 第一批模块
    │     ├─ 生成开发计划总览
    │     ├─ 生成工作计划清单（Markdown）
    │     └─ 用户审阅清单 → Git提交
    │
    ├─ 第2批：第二批模块
    │     ├─ 生成工作计划清单
    │     └─ 用户审阅清单 → Git提交
    │
    ├─ 第3批：第三批模块（核心模块）
    │     ├─ 生成工作计划清单
    │     └─ 用户审阅清单 → Git提交
    │
    ├─ 第4批：L3-E2E（增量测试）
    │     ├─ 生成工作计划清单
    │     └─ 用户审阅清单 → Git提交
    │
完成：全部清单（不生成任务单详情JSON）
```

### 6.2 FlowTicket执行流程

```
FlowTicket启动：
    │
    ├─ 读取工作计划清单（Markdown）
    ├─ 版本一致性检查
    │     ├─ 读取设计文档版本号
    │     ├─ 与清单标注版本对比
    │     └─ 不一致 → 创建问题单 → 暂停
    │
    ├─ 循环执行任务单（001~N）
    │     ├─ 执行子代理读取设计文档原文
    │     ├─ AI自主决策实现方式
    │     ├─ pytest/vitest闭环迭代
    │     ├─ Guardian合规检查（验证quality_check）
    │     └─ 验证子代理业务验证
    │
    └─ 完成：更新计划状态
```

## 七、审核标准索引

### 7.1 审核Checklist体系

| Checklist | 适用文档 | 审核维度 | 审核时机 |
|----------|---------|---------|---------|
| 43-开发计划总览审核Checklist | 开发计划总览.md | 批次顺序、模块依赖、功能概览、内容审核 | 总览生成后 |
| 44-工作计划清单审核Checklist | 工作计划清单.md（按项目实际份数）| 任务单结构、编号约束、验收标准引用、内容审核 | 清单批量生成后 |
| 45-开发计划整体性审核Checklist | 总览 + 全部清单 | 跨文档一致性、逻辑冲突、重复冗余、引用完整性 | 总览+清单单独审核通过后 |

### 7.2 三层审核流程

```
第1层：总览审核（43 Checklist）
    ├─ 格式完整性（E系列）
    ├─ 内容审核（C系列）
    └─ 通过标准：总分≥80

第2层：清单批量审核（44 Checklist）
    ├─ 格式完整性（E系列）
    ├─ 内容审核（C系列）
    ├─ 特殊检查项（按计划类型）
    └─ 通过标准：25份全部≥80

第3层：整体性审核（45 Checklist）
    ├─ 上下文一致性（X系列）
    ├─ 逻辑冲突检查（L系列）
    ├─ 重复冗余检查（R系列）
    ├─ 引用完整性检查（Y系列）
    ├─ 批次依赖一致性（D系列）
    └─ 通过标准：总分≥90
```

### 7.3 审核评分计算

**43/44 Checklist**：
```
总分 = 100 - (阻断项 × 10) - (内容审核 × 5) - (警告项 × 2)
```

**45 Checklist**：
```
总分 = 100 - (一致性 × 10) - (逻辑冲突 × 10) - (重复冗余 × 5) - (引用缺失 × 5)
```
