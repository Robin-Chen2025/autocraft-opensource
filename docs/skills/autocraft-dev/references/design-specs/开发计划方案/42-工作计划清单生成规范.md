## 一、规范定位

### 1.1 清单定位

工作计划清单是**计划级概要文档**，包含：
- 基本信息（计划ID、模块、计划类型、依赖）
- 输入文档（设计文档路径）
- 任务单结构表格（编号、内容、类型、验证分类、功能归属）
- 验收标准（引用测试方案总纲）
- **编号约束**（FlowTicket动态生成必须遵守）
- **版本一致性检查**（设计文档版本标注）

**用户审阅入口**：清单让用户确认任务单概要结构，不审阅任务单详情JSON。

### 1.2 与FlowTicket关系

| 文档 | 生成时机 | 内容 | AI决策空间 |
|------|---------|------|-----------|
| **工作计划清单** | 制定阶段预生成 | 任务单概要表格 + 编号约束 + 版本标注 | ❌ 不涉及 |
| **任务单详情JSON** | FlowTicket启动时动态生成 | execution_steps + input_docs + output_files | ✅ AI自主规划 |

## 二、清单章节结构规范

### 2.1 清单章节结构（7章节）

| 章节 | 内容 | 长度预估 | 对FlowTicket的价值 |
|------|------|---------|-------------------|
| **1. 基本信息** | 计划ID、模块、类型、批次、依赖 | ~10行 | FlowTicket入口识别 |
| **2. 输入文档** | 设计文档路径表格（含版本号）| ~20行 | **版本一致性检查** |
| **3. 功能模块概览** | 功能ID、API数、场景数概要 | ~15行 | 任务单循环依据 |
| **4. 任务单结构表格** | 编号、内容、类型、验证分类、功能归属、产出物 | ~30行 | **编号约束** |
| **5. 验收标准** | 引用测试方案总纲验收标准 | ~15行 | Guardian检查依据 |
| **6. 编号约束规则** | FlowTicket动态生成必须遵守的编号规则 | ~20行 | **一致性保障** |
| **7. 预计工时** | 任务类型工时估算 | ~10行 | 进度跟踪 |

**总长度**：~120行（紧凑版，用户快速审阅）

### 3.2 版本不一致处理

| 场景 | 处理方式 |
|------|---------|
| 设计文档版本升级（v1.2→v1.3）| 创建问题单，提示"设计文档已更新，需重新生成清单" |
| 设计文档版本回退（v1.3→v1.2）| 创建问题单，提示"设计文档版本回退，需确认清单版本" |
| 设计文档不存在 | 创建问题单，提示"设计文档缺失，无法启动计划" |

## 五、第六章：编号约束规则规范

### 5.1 编号约束定义

```markdown
## 编号约束规则（FlowTicket动态生成必须遵守）

### INFRA编号约束（基础建设计划）
| 编号范围 | 允许的任务类型 | 允许的验证分类 | 说明 |
|---------|--------------|--------------|------|
| 001 | BUILD | B类 | 项目骨架搭建（目录结构）|
| 002 | BUILD | B类 | 开发环境配置（依赖安装）|
| 003-A | **BUILD** | B类 | **数据库代码生成：基础表（无外键）** |
| 003-B | **BUILD** | B类 | **数据库代码生成：依赖表（依赖003-A）** |
| 003-C | **BUILD** | B类 | **数据库代码生成：配置表（独立）** |
| 004 | **EXECUTE** | B类 | **数据库初始化（执行models创建实际表）← 依赖003-A/B/C** |
| 005 | BUILD | B类 | 启动脚本创建（start.sh）|
| 006 | DOC | DOC类 | 验收报告 |

> **003拆分规则**：
> - **拆分必要性**：数据库表代码按依赖层级拆分，确保代码生成顺序正确（先生成无依赖表，后生成依赖表）
> - **拆分条件**：当数据库包含多个层级（Level ≥ 1）时必须拆分；单层级（Level 0）不拆分
> - **分析方法**：读取数据库设计文档，提取表外键关系，计算依赖层级（Level 0~N）
> - **依赖层级定义**：
>   - Level 0：无外键依赖的根表/独立配置表
>   - Level N：外键引用Level 0~N-1的表
> - **拆分粒度**：每个Level对应一个子任务单
>   - 命名规则：003-A=Level 0, 003-B=Level 1, 003-C=Level 2, ...（字母递增）
>   - 产出物命名：models/group{level+1}.py（Level 0→group1.py）
> - **执行顺序**：按Level递增顺序执行（Level 0 → Level 1 → ... → Level N）
> - **示例**：某项目数据库分析结果为Level 0~5（6个层级），拆分为A~F（6个子任务单）

> **关键区分**：
> - 001~002~003-A/B/C~005是BUILD类型：创建代码文件
> - 004是EXECUTE类型：执行models创建实际数据库表，依赖003全部产出

### BE-DEV/FE-DEV编号约束
| 编号范围 | 允许的任务类型 | 允许的验证分类 | 说明 |
|---------|--------------|--------------|------|
| 001~N-2 | BUILD | A类 | 功能循环（每功能：测试代码→程序代码+pytest闭环）|
| N-1 | DOC | DOC类 | 开发报告 |

### BE-L2/FE-L2编号约束
| 编号范围 | 允许的任务类型 | 允许的验证分类 | 说明 |
|---------|--------------|--------------|------|
| 001 | BUILD | B类 | 测试环境准备 |
| 002~N-2 | BUILD | A类 | 场景循环（每场景1个任务）|
| N-1 | DOC | DOC类 | 测试报告 |

### L3-E2E编号约束
| 编号范围 | 允许的任务类型 | 允许的验证分类 | 说明 |
|---------|--------------|--------------|------|
| 001 | BUILD | B类 | 测试环境准备 |
| 002~N-2 | BUILD | A类 | 流程循环（每流程2个任务：两层测试）|
| N-1 | DOC | DOC类 | E2E报告 |

### 约束检查规则
FlowTicket动态生成任务单详情时：
1. task_no必须在清单定义的编号范围内
2. task_type必须与清单定义一致
3. verify_category必须与清单定义一致
4. test_plan_reference必须指向已预生成的测试方案文档

违反约束 → 创建问题单 → 暂停计划
```

### 5.2 约束检查逻辑

```python
def validate_task_detail(task_detail, plan_manifest):
    # 编号范围检查
    if not task_detail['task_no'] in plan_manifest['valid_task_ranges']:
        raise ConstraintViolationError(f"task_no {task_detail['task_no']} 超出允许范围")
    
    # 任务类型检查
    expected_type = plan_manifest['task_types'][task_detail['task_no']]
    if task_detail['task_type'] != expected_type:
        raise ConstraintViolationError(f"task_type 不匹配，期望 {expected_type}")
    
    # 验证分类检查
    expected_category = plan_manifest['verify_categories'][task_detail['task_no']]
    if task_detail['verify_category'] != expected_category:
        raise ConstraintViolationError(f"verify_category 不匹配，期望 {expected_category}")
    
    # 功能归属检查
    expected_function = plan_manifest['function_ids'][task_detail['task_no']]
    if task_detail['function_id'] != expected_function:
        raise ConstraintViolationError(f"function_id 不匹配，期望 {expected_function}")
```

**错误理解**：
```
❌ 每功能3个任务：方案设计 → 测试代码 → 程序代码

```
✅ 每功能2个任务：测试代码（BUILD/A类）→ 程序代码+pytest闭环（BUILD/A类）
✅ 方案设计在执行子代理内部动态完成，不是独立任务单

## 八、生成流程

### 8.1 AI生成流程

```
第1步：读取设计文档
    ├─ API设计文档（对应模块章节）
    ├─ 数据库设计文档（对应表结构）
    ├─ 系统功能设计文档（对应功能ID）

    ├─ 批次顺序（确定依赖关系）
    ├─ 模块依赖矩阵（确定前置模块）
    └─ 功能概览（确定功能数量）

第3步：生成清单Markdown
    ├─ 按章节结构填充内容
    ├─ 计算任务单数量（固定任务 + 功能循环）
    ├─ 定义编号约束表格
    └─ 标注设计文档版本号

第4步：用户审阅
    ├─ 确认任务单数量
    ├─ 确认编号约束合理
    └─ 确认验收标准引用正确

第5步：Git提交
    └─ docs/development-plans/{模块}/WP-{模块}-{类型}-001.md
```
